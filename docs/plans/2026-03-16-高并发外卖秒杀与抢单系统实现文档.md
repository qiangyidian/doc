# 高并发外卖秒杀与抢单系统实现文档

**目标：** 从零创建一个 Spring Boot 实战项目，围绕“高并发外卖秒杀与抢单”这个业务，完整演示 Redis Cluster、主从复制、Sentinel、RabbitMQ、Canal、Redisson、Lua、布隆过滤器、缓存预热、缓存击穿保护和秒杀下单。

**架构说明：** 这份文档按你给出的原始方案来写，整体采用 `MySQL + Redis Cluster + Sentinel + RabbitMQ + Canal + Redisson`。其中真正承载业务读写的是 Redis Cluster；Sentinel 这部分作为“原文保留版”的高可用教学配置一并演示；缓存一致性通过 `MySQL -> Canal -> RabbitMQ -> 删除 Redis 缓存` 这条链路说明，并提供“手动模拟 Canal 发消息”的代码，方便你本地一步一步调试。

**技术栈：** Java 17、Spring Boot 3.2.5、Spring Web、Spring Data Redis、Redisson、MyBatis、MySQL 8.0、Redis 7.2、RabbitMQ 3.13-management、Canal 1.1.7、Docker Compose

---

## 一、这份文档解决什么问题

这不是一个“只讲单点技术”的文档，而是一份完整业务场景文档。

你最终会学到下面 5 个高频场景：

1. 商家修改菜单和库存时，如何避免 MySQL 与 Redis 双写脏数据
2. 每天凌晨如何预热热门菜单，并通过随机过期时间防止缓存雪崩
3. 恶意请求大量查询不存在的菜品时，如何通过布隆过滤器防止缓存穿透
4. 热门秒杀菜品缓存刚好过期时，如何通过 Redisson 分布式锁防止缓存击穿
5. 用户点击“立即抢单”时，如何通过可重入锁、Lua 和 Hash Tag 保证库存不超卖

## 二、先讲清楚这份文档的“教学化”处理

这份文档按你的原文来写，但有 3 个地方我会明确告诉你它们是“教学化写法”：

1. `Redis Cluster + Sentinel`
   这份文档会同时保留这两部分配置，方便你理解分片、复制、高可用和故障探测这些概念。

2. `Canal`
   真实生产里通常会有独立的 Canal 客户端程序去消费 Binlog 变更并投递到 MQ。
   这份文档为了方便你本地练习，除了写标准链路，也额外提供一个“模拟 Canal 发 MQ 消息”的接口。

3. `秒杀下单原子性`
   这里的“原子性”重点是指 Redis 侧的“库存扣减 + 重复下单标记”通过 Lua 一次完成。
   MySQL 最终订单落库仍然在 Java 代码里完成，这是新手阶段更容易理解也更容易调试的方式。

## 六、创建 Redis 配置文件

### 6.1 先创建 `init/redis/redis-7001.conf`

```conf
# 当前 Redis 实例监听的端口。
port 7001

# 关闭保护模式，方便本地容器互联。
protected-mode no

# 允许容器内部网络访问。
bind 0.0.0.0

# 开启集群模式。
cluster-enabled yes

# 当前节点的集群元数据文件。
cluster-config-file nodes-7001.conf

# 集群节点超时时间，单位毫秒。
cluster-node-timeout 5000

# 集群总线端口，通常就是普通端口 + 10000。
cluster-announce-bus-port 17001

# 这里开启 AOF，方便你观察数据持久化。
appendonly yes

# 数据目录。
dir /data

# 你的原文里提到了防脑裂配置，这里按原文保留。
min-replicas-to-write 1
min-replicas-max-lag 10
```

这段代码的作用：

- 定义 7001 节点的端口、集群模式、AOF 持久化和脑裂防护参数
- `min-replicas-to-write 1` 的意思是主节点至少看到 1 个副本在线才继续写入
- 这个文件是 6 个 Redis 节点的模板

### 6.2 再创建 `redis-7002.conf` 到 `redis-7006.conf`

这 5 个文件的内容和 `redis-7001.conf` 完全一样，只需要改下面 3 个地方：

| 文件名 | `port` | `cluster-config-file` | `cluster-announce-bus-port` |
|---|---:|---|---:|
| `redis-7002.conf` | `7002` | `nodes-7002.conf` | `17002` |
| `redis-7003.conf` | `7003` | `nodes-7003.conf` | `17003` |
| `redis-7004.conf` | `7004` | `nodes-7004.conf` | `17004` |
| `redis-7005.conf` | `7005` | `nodes-7005.conf` | `17005` |
| `redis-7006.conf` | `7006` | `nodes-7006.conf` | `17006` |

这段代码的作用：

- 6 个 Redis 节点本质上就是同一个模板，只是端口不同
- 这样你不用抄 6 次几乎完全一样的配置
- 后面创建 Cluster 时，会自动把这 6 个节点组织成 `3 主 3 从`

## 七、创建 Sentinel 配置文件

### 7.1 创建 `init/redis/sentinel-27001.conf`

```conf
# Sentinel 自己的端口。
port 27001

# 关闭保护模式，方便容器之间通信。
protected-mode no

# 允许容器内部访问。
bind 0.0.0.0

# Sentinel 要监控的主节点名字叫 mymaster。
# 这里按你的原文保留 Sentinel 监控逻辑。
sentinel monitor mymaster redis-7001 7001 2

# 主观下线判断时间，单位毫秒。
sentinel down-after-milliseconds mymaster 5000

# 故障转移超时时间。
sentinel failover-timeout mymaster 10000

# 一次允许多少个副本同时重新同步。
sentinel parallel-syncs mymaster 1
```

这段代码的作用：

- 启动一个 Sentinel 实例去监控 `redis-7001`
- `2` 表示至少要有 2 个 Sentinel 同意，才认为主节点客观下线
- 这里主要用于教学演示“主从 + 哨兵”的概念

### 7.2 再创建 `sentinel-27002.conf` 和 `sentinel-27003.conf`

它们和 `sentinel-27001.conf` 内容一样，只需要把 `port` 改成：

- `27002`
- `27003`

这段代码的作用：

- 3 个 Sentinel 组成一个哨兵集群
- 多个 Sentinel 才能完成多数派判断和故障探测

## 八、启动中间件并创建 Redis Cluster

先启动全部容器：

```bash
docker compose up -d
```

这段代码的作用：

- 后台启动全部中间件容器
- 这是你后面所有实验的基础

然后执行 Redis Cluster 创建命令：

```bash
docker exec -it redis-7001 redis-cli --cluster create \
redis-7001:7001 redis-7002:7002 redis-7003:7003 \
redis-7004:7004 redis-7005:7005 redis-7006:7006 \
--cluster-replicas 1 --cluster-yes
```

这段代码的作用：

- 把 6 个 Redis 节点初始化成一个 Cluster
- `--cluster-replicas 1` 表示每个主节点带 1 个从节点
- 最终会形成 `3 主 3 从` 的分片结构

检查 Cluster 节点状态：

```bash
docker exec -it redis-7001 redis-cli -p 7001 cluster nodes
```

检查 Sentinel 监控状态：

```bash
docker exec -it sentinel-27001 redis-cli -p 27001 SENTINEL masters
```

检查 Hash Tag 是否在同一槽位：

```bash
docker exec -it redis-7001 redis-cli -p 7001 CLUSTER KEYSLOT "{meal:1001}:stock"
docker exec -it redis-7001 redis-cli -p 7001 CLUSTER KEYSLOT "{meal:1001}:order:10001"
```

这段代码的作用：

- 验证两个带同样 Hash Tag 的 key 会落到同一个槽位
- 这是后面 Lua 脚本能原子操作多个 key 的前提

```java
import org.springframework.amqp.support.converter.Jackson2JsonMessageConverter;
import org.springframework.amqp.support.converter.MessageConverter;
import org.springframework.amqp.core.Binding;
import org.springframework.amqp.core.BindingBuilder;
import org.springframework.amqp.core.DirectExchange;
import org.springframework.amqp.core.Queue;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class RabbitMqConfig {

    // 交换机名称。
    public static final String CACHE_INVALID_EXCHANGE = "takeaway.cache.invalid.exchange";

    // 队列名称。
    public static final String CACHE_INVALID_QUEUE = "takeaway.cache.invalid.queue";

    // 路由键。
    public static final String CACHE_INVALID_ROUTING_KEY = "takeaway.cache.invalid.meal";

    @Bean
    public DirectExchange cacheInvalidExchange() {
        // DirectExchange 表示精确路由。
        return new DirectExchange(CACHE_INVALID_EXCHANGE, true, false);
    }

    @Bean
    public Queue cacheInvalidQueue() {
        // durable=true 表示队列持久化。
        return new Queue(CACHE_INVALID_QUEUE, true);
    }

    @Bean
    public MessageConverter messageConverter() {
        // 把 Java 对象自动转换成 JSON。
        // 如果不配这个转换器，RabbitTemplate 直接发送普通 Java 对象时会更容易报错。
        return new Jackson2JsonMessageConverter();
    }

    @Bean
    public Binding cacheInvalidBinding(Queue cacheInvalidQueue,
                                       DirectExchange cacheInvalidExchange) {
        // 把队列绑定到交换机上，并指定路由键。
        return BindingBuilder.bind(cacheInvalidQueue)
                .to(cacheInvalidExchange)
                .with(CACHE_INVALID_ROUTING_KEY);
    }
}
```

```java
package com.example.takeaway.lua;

public final class StockOrderLuaScripts {

    private StockOrderLuaScripts() {
    }

    public static final String STOCK_ORDER_LUA = """
            -- KEYS[1] 是库存 key，例如 {meal:1001}:stock
            local stockKey = KEYS[1]
            -- KEYS[2] 是用户下单标记 key，例如 {meal:1001}:order:10001
            local orderKey = KEYS[2]

            -- 先判断这个用户是不是已经抢过了。
            if redis.call('EXISTS', orderKey) == 1 then
                -- 返回 2 表示重复下单。
                return 2
            end

            -- 读取当前库存。
            local stock = tonumber(redis.call('GET', stockKey))

            -- 如果库存不存在，说明预热没做好，或者 key 被删了。
            if stock == nil then
                -- 返回 3 表示库存 key 不存在。
                return 3
            end

            -- 如果库存已经小于等于 0，直接返回抢空。
            if stock <= 0 then
                -- 返回 0 表示库存不足。
                return 0
            end

            -- 扣减 1 份库存。
            redis.call('DECR', stockKey)

            -- 写入“这个用户已经抢到过”的标记。
            -- 这里用 SET 而不是 Hash，只是为了让新手更容易理解。
            redis.call('SET', orderKey, '1')

            -- 给下单标记一个过期时间，避免长期堆积无用 key。
            redis.call('EXPIRE', orderKey, 3600)

            -- 返回 1 表示抢单成功。
            return 1
            """;
}
```

这段代码的作用：

- 在 Redis 里一次完成“检查重复下单 + 检查库存 + 扣减库存 + 写下单标记”
- 只要两个 key 在同一个槽位里，Lua 脚本就可以原子执行

```java
package com.example.takeaway.mq;

import com.example.takeaway.common.RedisKeyConstants;
import com.example.takeaway.config.RabbitMqConfig;
import com.rabbitmq.client.Channel;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.amqp.support.AmqpHeaders;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.messaging.handler.annotation.Header;
import org.springframework.stereotype.Component;

@Slf4j
@Component
@RequiredArgsConstructor
public class CacheInvalidConsumer {

    private final StringRedisTemplate stringRedisTemplate;

    @RabbitListener(queues = RabbitMqConfig.CACHE_INVALID_QUEUE)
    public void consume(CacheInvalidMessage message,
                        Channel channel,
                        @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) throws Exception {
        try {
            // 删除菜品详情缓存。
            stringRedisTemplate.delete(RedisKeyConstants.mealKey(message.getMealId()));

            // 同时删除 Redis 里的秒杀库存 key，避免旧库存继续被使用。
            stringRedisTemplate.delete(RedisKeyConstants.stockKey(message.getMealId()));

            // 如果商家 ID 不为空，再删除热门菜单缓存。
            if (message.getMerchantId() != null) {
                stringRedisTemplate.delete(RedisKeyConstants.hotMealsKey(message.getMerchantId()));
            }

            log.info("消费删缓存消息成功，mealId={}, reason={}", message.getMealId(), message.getReason());

            // 手动确认消息处理成功。
            channel.basicAck(deliveryTag, false);
        } catch (Exception e) {
            log.error("消费删缓存消息失败，mealId={}", message.getMealId(), e);
            // 这里简单拒绝并重新入队，便于你本地练习看到重试效果。
            channel.basicNack(deliveryTag, false, true);
        }
    }
}
```

这段代码的作用：

- 消费 MQ 删除缓存消息
- 真正执行“删除菜品缓存、删除库存缓存、删除热门菜单缓存”
- 用了手动 ack，让你清楚看到消息什么时候算真正处理成功

### 21.1 创建 `service/impl/CacheServiceImpl.java`

```java
package com.example.takeaway.service.impl;

import com.example.takeaway.common.RedisKeyConstants;
import com.example.takeaway.entity.Meal;
import com.example.takeaway.mapper.MealMapper;
import com.example.takeaway.service.CacheService;
import com.fasterxml.jackson.core.JsonProcessingException;
import com.fasterxml.jackson.databind.ObjectMapper;
import jakarta.annotation.PostConstruct;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.stereotype.Service;

import java.time.Duration;
import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.ThreadLocalRandom;

@Slf4j
@Service
@RequiredArgsConstructor
public class CacheServiceImpl implements CacheService {

    private final StringRedisTemplate stringRedisTemplate;
    private final ObjectMapper objectMapper;
    private final MealMapper mealMapper;

    @Value("${takeaway.cache.meal-ttl-minutes}")
    private long mealTtlMinutes;

    @Value("${takeaway.cache.hot-meal-ttl-hours}")
    private long hotMealTtlHours;

    @Value("${takeaway.cache.random-ttl-minutes-min}")
    private long randomTtlMinMinutes;

    @Value("${takeaway.cache.random-ttl-minutes-max}")
    private long randomTtlMaxMinutes;

    @Value("${takeaway.bloom.bitmap-size}")
    private long bloomBitmapSize;

    // 这里直接把多个 hash 种子定义死，方便新手理解。
    private static final int[] BLOOM_SEEDS = {5, 7, 11, 13, 31, 37, 61};

    @PostConstruct
    public void initBloomFilter() {
        // 项目启动时，把所有合法 mealId 先放进布隆过滤器。
        rebuildMealBloomFilter();
    }

    @Override
    public Meal getMealCache(Long mealId) {
        String json = stringRedisTemplate.opsForValue().get(RedisKeyConstants.mealKey(mealId));
        if (json == null || json.isBlank()) {
            return null;
        }
        try {
            return objectMapper.readValue(json, Meal.class);
        } catch (JsonProcessingException e) {
            throw new RuntimeException("解析菜品缓存失败", e);
        }
    }

    @Override
    public void cacheMeal(Meal meal) {
        try {
            String json = objectMapper.writeValueAsString(meal);
            stringRedisTemplate.opsForValue().set(
                    RedisKeyConstants.mealKey(meal.getId()),
                    json,
                    buildMealTtl()
            );
            // 缓存菜品详情时，顺手把库存 key 也一起准备好。
            prepareStockKey(meal);
        } catch (JsonProcessingException e) {
            throw new RuntimeException("写入菜品缓存失败", e);
        }
    }

    @Override
    public void warmHotMeals(Long merchantId) {
        List<Meal> hotMeals = mealMapper.selectHotMeals(merchantId);
        for (Meal meal : hotMeals) {
            cacheMeal(meal);
            putMealIdIntoBloom(meal.getId());
        }

        if (merchantId != null) {
            try {
                String json = objectMapper.writeValueAsString(hotMeals);
                stringRedisTemplate.opsForValue().set(
                        RedisKeyConstants.hotMealsKey(merchantId),
                        json,
                        buildHotMealTtl()
                );
            } catch (JsonProcessingException e) {
                throw new RuntimeException("写入热门菜单缓存失败", e);
            }
        }
    }

    @Override
    public void rebuildMealBloomFilter() {
        stringRedisTemplate.delete(RedisKeyConstants.mealBloomKey());
        List<Long> mealIds = mealMapper.selectAllValidMealIds();
        for (Long mealId : mealIds) {
            putMealIdIntoBloom(mealId);
        }
        log.info("布隆过滤器重建完成，mealId 数量={}", mealIds.size());
    }

    @Override
    public boolean mightContainMealId(Long mealId) {
        String bloomKey = RedisKeyConstants.mealBloomKey();
        for (int seed : BLOOM_SEEDS) {
            long offset = hash(String.valueOf(mealId), seed);
            Boolean exists = stringRedisTemplate.opsForValue().getBit(bloomKey, offset);
            if (Boolean.FALSE.equals(exists)) {
                return false;
            }
        }
        return true;
    }

    @Override
    public void prepareStockKey(Meal meal) {
        stringRedisTemplate.opsForValue().set(
                RedisKeyConstants.stockKey(meal.getId()),
                String.valueOf(meal.getStock()),
                buildMealTtl()
        );
    }

    @Override
    public List<Meal> getHotMealsCache(Long merchantId) {
        String json = stringRedisTemplate.opsForValue().get(RedisKeyConstants.hotMealsKey(merchantId));
        if (json == null || json.isBlank()) {
            return new ArrayList<>();
        }
        try {
            return objectMapper.readerForListOf(Meal.class).readValue(json);
        } catch (JsonProcessingException e) {
            throw new RuntimeException("解析热门菜单缓存失败", e);
        }
    }

    private void putMealIdIntoBloom(Long mealId) {
        String bloomKey = RedisKeyConstants.mealBloomKey();
        for (int seed : BLOOM_SEEDS) {
            long offset = hash(String.valueOf(mealId), seed);
            stringRedisTemplate.opsForValue().setBit(bloomKey, offset, true);
        }
    }

    private long hash(String value, int seed) {
        long result = 0;
        for (int i = 0; i < value.length(); i++) {
            result = result * seed + value.charAt(i);
        }
        return Math.abs(result) % bloomBitmapSize;
    }

    private Duration buildMealTtl() {
        long randomMinutes = ThreadLocalRandom.current()
                .nextLong(randomTtlMinMinutes, randomTtlMaxMinutes + 1);
        return Duration.ofMinutes(mealTtlMinutes + randomMinutes);
    }

    private Duration buildHotMealTtl() {
        long randomMinutes = ThreadLocalRandom.current()
                .nextLong(randomTtlMinMinutes, randomTtlMaxMinutes + 1);
        return Duration.ofHours(hotMealTtlHours).plusMinutes(randomMinutes);
    }
}
```

这段代码的作用：

- 实现缓存查询、缓存写入、布隆过滤器、热门菜单预热
- 随机 TTL 的逻辑也在这里
- 这是“防雪崩、防穿透”的核心代码

```java
package com.example.takeaway.service.impl;

import com.example.takeaway.common.RedisKeyConstants;
import com.example.takeaway.entity.Meal;
import com.example.takeaway.mapper.MealMapper;
import com.example.takeaway.service.CacheService;
import com.example.takeaway.service.MealService;
import lombok.RequiredArgsConstructor;
import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.springframework.stereotype.Service;

import java.util.concurrent.TimeUnit;

@Service
@RequiredArgsConstructor
public class MealServiceImpl implements MealService {

    private final CacheService cacheService;
    private final MealMapper mealMapper;
    private final RedissonClient redissonClient;

    @Override
    public Meal queryMealDetail(Long mealId) {
        // 第一步：先查布隆过滤器。
        if (!cacheService.mightContainMealId(mealId)) {
            throw new IllegalArgumentException("菜品不存在，已被布隆过滤器拦截");
        }

        // 第二步：查 Redis 缓存。
        Meal cacheMeal = cacheService.getMealCache(mealId);
        if (cacheMeal != null) {
            return cacheMeal;
        }

        // 第三步：缓存没命中，开始做缓存击穿保护。
        RLock lock = redissonClient.getLock(RedisKeyConstants.rebuildCacheLockKey(mealId));
        boolean locked = false;
        try {
            // 这里没有主动指定 leaseTime，所以 Redisson Watch Dog 可以自动续期。
            locked = lock.tryLock(2, TimeUnit.SECONDS);
            if (locked) {
                Meal dbMeal = mealMapper.selectMealDetailById(mealId);
                if (dbMeal == null) {
                    throw new IllegalArgumentException("菜品不存在");
                }
                cacheService.cacheMeal(dbMeal);
                return dbMeal;
            }

            // 没拿到锁的线程，短暂休眠后重试 Redis。
            Thread.sleep(100);
            Meal retryCacheMeal = cacheService.getMealCache(mealId);
            if (retryCacheMeal != null) {
                return retryCacheMeal;
            }

            // 最后再兜底查一次数据库。
            Meal dbMeal = mealMapper.selectMealDetailById(mealId);
            if (dbMeal != null) {
                cacheService.cacheMeal(dbMeal);
            }
            return dbMeal;
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("查询菜品详情时线程被中断", e);
        } finally {
            if (locked && lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

这段代码的作用：

- 完整实现“布隆过滤器 -> Redis -> Redisson 锁 -> MySQL -> 重建缓存”这一条链路
- 这是“防缓存穿透 + 防缓存击穿”的核心业务代码

```java
package com.example.takeaway.service.impl;

import com.example.takeaway.common.RedisKeyConstants;
import com.example.takeaway.dto.SeckillOrderRequest;
import com.example.takeaway.entity.Meal;
import com.example.takeaway.entity.OrderInfo;
import com.example.takeaway.mapper.MealMapper;
import com.example.takeaway.service.OrderService;
import com.example.takeaway.service.SeckillService;
import lombok.RequiredArgsConstructor;
import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.data.redis.core.script.DefaultRedisScript;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.UUID;

@Service
@RequiredArgsConstructor
public class SeckillServiceImpl implements SeckillService {

    private final RedissonClient redissonClient;
    private final StringRedisTemplate stringRedisTemplate;
    private final DefaultRedisScript<Long> stockOrderRedisScript;
    private final MealMapper mealMapper;
    private final OrderService orderService;

    @Override
    public String createOrder(SeckillOrderRequest request) {
        // 第一层加锁：模拟 createOrder() 方法先拿到商品锁。
        RLock lock = redissonClient.getLock(RedisKeyConstants.mealLockKey(request.getMealId()));
        // 注意：这里调用的是 lock()，没有指定租约时间。
        // 这样 Redisson 的 Watch Dog 就会自动续期。
        lock.lock();
        try {
            return deductStockAndCreateOrder(request);
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    private String deductStockAndCreateOrder(SeckillOrderRequest request) {
        // 第二层再次获取同一把锁，演示 Redisson 的可重入锁机制。
        RLock lock = redissonClient.getLock(RedisKeyConstants.mealLockKey(request.getMealId()));
        lock.lock();
        try {
            String orderNo = "ODR-" + UUID.randomUUID().toString().replace("-", "");

            // 这两个 key 使用了相同的 Hash Tag，保证落到同一槽位。
            String stockKey = RedisKeyConstants.stockKey(request.getMealId());
            String orderKey = RedisKeyConstants.orderMarkerKey(request.getMealId(), request.getUserId());

            Long result = stringRedisTemplate.execute(
                    stockOrderRedisScript,
                    List.of(stockKey, orderKey)
            );

            if (result == null) {
                throw new IllegalStateException("Lua 脚本执行失败，返回值为空");
            }
            if (result == 0L) {
                throw new IllegalStateException("库存已抢完");
            }
            if (result == 2L) {
                throw new IllegalStateException("你已经抢过这份秒杀商品了");
            }
            if (result == 3L) {
                throw new IllegalStateException("Redis 中不存在库存 key，请先做缓存预热");
            }

            // Redis 侧扣减库存成功后，再落 MySQL 订单。
            Meal meal = mealMapper.selectMealDetailById(request.getMealId());
            if (meal == null) {
                throw new IllegalArgumentException("菜品不存在");
            }

            OrderInfo orderInfo = orderService.createOrder(
                    request.getUserId(),
                    request.getMealId(),
                    orderNo,
                    meal.getPrice()
            );

            return orderInfo.getOrderNo();
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

这段代码的作用：

- 这是“立即抢单”的核心代码
- 外层锁和内层锁是同一把锁，正好用来演示 Redisson 的可重入机制
- Lua 脚本负责 Redis 侧的原子扣减和重复抢单标记
- Watch Dog 会在方法执行时间变长时自动为锁续期

## 二十二、创建定时任务 `schedule/MealCacheWarmScheduler.java`

```java
package com.example.takeaway.schedule;

import com.example.takeaway.service.CacheService;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;

@Slf4j
@Component
@RequiredArgsConstructor
public class MealCacheWarmScheduler {

    private final CacheService cacheService;

    // 每天凌晨 3 点执行一次。
    @Scheduled(cron = "0 0 3 * * ?")
    public void warmHotMeals() {
        log.info("开始执行热门菜单缓存预热任务");
        // 传 null 表示预热全部热门菜单。
        cacheService.warmHotMeals(null);
        log.info("热门菜单缓存预热任务执行完成");
    }
}
```

这段代码的作用：

- 模拟每天凌晨对热门菜单做缓存预热
- 这是“防缓存雪崩”场景的定时任务入口

## 二十五、5 个核心场景怎么测试

### 场景 1：商家修改菜单与库存，触发异步删缓存

先查一次菜品详情，让 Redis 里出现缓存：

```bash
curl.exe http://localhost:8080/meals/1001
```

再改菜品信息：

```bash
curl.exe -X PUT "http://localhost:8080/merchant/meals/1001" -H "Content-Type: application/json" -d "{\"mealName\":\"1分钱霸王餐-升级版\",\"mealDesc\":\"库存同步测试\",\"price\":0.01,\"hotFlag\":1,\"status\":1}"
```

然后模拟 Canal 发消息：

```bash
curl.exe -X POST "http://localhost:8080/canal/mock/cache-invalid/1001?reason=UPDATE_MEAL"
```

最后再查一次：

```bash
curl.exe http://localhost:8080/meals/1001
```

你应该观察到：

- 第一次查询会先落缓存
- 改 MySQL 后，不直接删缓存
- 通过 MQ 删除缓存后，再次查询会加载最新数据

### 场景 2：热门菜单预热，防缓存雪崩

手动触发热门菜单预热：

```bash
curl.exe -X POST "http://localhost:8080/cache/warm/meals" -H "Content-Type: application/json" -d "{\"merchantId\":1}"
```

然后看 Redis TTL：

```bash
docker exec -it redis-7001 redis-cli -c -p 7001 TTL takeaway:meal:1001
docker exec -it redis-7001 redis-cli -c -p 7001 TTL takeaway:meal:1002
```

你应该观察到：

- 两个 key 都有 TTL
- TTL 不完全一样，因为我们做了随机过期时间

### 场景 3：布隆过滤器防缓存穿透

查一个明显不存在的菜品：

```bash
curl.exe http://localhost:8080/meals/-999
```

你应该观察到：

- 请求会很快失败
- 日志中能看到是布隆过滤器提前拦截，而不是一直去查 MySQL

### 场景 4：缓存击穿保护

先删除某个菜品缓存：

```bash
docker exec -it redis-7001 redis-cli -c -p 7001 DEL takeaway:meal:1001
```

然后快速连续请求多次：

```bash
curl.exe http://localhost:8080/meals/1001
curl.exe http://localhost:8080/meals/1001
curl.exe http://localhost:8080/meals/1001
```

你应该观察到：

- 只有拿到锁的线程去查数据库并重建缓存
- 其他线程短暂等待后直接命中新缓存

### 场景 5：立即抢单，防止超卖

先做缓存预热，让库存 key 进入 Redis：

```bash
curl.exe -X POST "http://localhost:8080/cache/warm/meals" -H "Content-Type: application/json" -d "{\"merchantId\":1}"
```

然后发起秒杀下单：

```bash
curl.exe -X POST "http://localhost:8080/seckill/orders" -H "Content-Type: application/json" -d "{\"userId\":10001,\"mealId\":1001}"
```

查订单结果：

```bash
curl.exe http://localhost:8080/orders/上一步返回的订单号
```

再让同一个用户重复抢：

```bash
curl.exe -X POST "http://localhost:8080/seckill/orders" -H "Content-Type: application/json" -d "{\"userId\":10001,\"mealId\":1001}"
```

你应该观察到：

- 第一次成功
- 第二次会被 Lua 脚本判定为重复抢单
- Redis 中的库存不会变成负数

## 二十七、常见报错排查

### 1. 报 `MOVED` 或 `CROSSSLOT`

原因：

- 说明你的 key 不在同一个槽位

解决：

- 检查是不是用了相同的 Hash Tag
- 用下面命令确认槽位是否一致

```bash
docker exec -it redis-7001 redis-cli -p 7001 CLUSTER KEYSLOT "{meal:1001}:stock"
docker exec -it redis-7001 redis-cli -p 7001 CLUSTER KEYSLOT "{meal:1001}:order:10001"
```

### 2. 报 `库存 key 不存在`

原因：

- 你没有先做缓存预热
- 或者删缓存后没有重新加载库存

解决：

- 先调用 `/cache/warm/meals`

### 3. 报 Redis Cluster 重定向到了容器内部地址

原因：

- 你在宿主机直接启动 Spring Boot
- 但 Redis Cluster 节点对外通告的是容器内部地址

解决：

- 本地第一次练习时，优先先看文档流程和代码逻辑
- 如果你真的要在宿主机直连 Cluster，记得把 Redis 节点的 announce 地址改成宿主机可访问的 IP
- 或者把 Spring Boot 应用也放进同一个 Docker 网络里运行

### 4. RabbitMQ 队列里有消息但没消费

原因：

- Spring Boot 没启动
- 队列名写错
- 消费者报异常后不断重试

解决：

- 打开 RabbitMQ 管理后台查看
- 地址：`http://localhost:15672`
- 账号密码：`guest / guest`

### 5. 布隆过滤器误判“可能存在”

原因：

- 布隆过滤器的特点就是“可以误判存在，但不会误判不存在”

解决：

- 这是正常现象
- 后面 Redis 和 MySQL 还会继续兜底校验

### 6. Redisson 锁释放报错

原因：

- 当前线程没有持有锁却执行了 `unlock()`

解决：

- 一定先判断 `lock.isHeldByCurrentThread()`

## 二十八、你在这个项目里真正学到了什么

1. 不是所有缓存问题都靠“多加缓存”解决，关键是失效策略和并发保护
2. MySQL 和 Redis 最稳妥的协作方式之一，不是代码里双写，而是“写数据库 + 异步删缓存”
3. 布隆过滤器的价值不在于绝对准确，而在于挡住大部分明显无效请求
4. Redisson 的可重入锁和看门狗，能让复杂业务比你手写 Redis 锁更稳
5. Redis Cluster 下如果要让 Lua 操作多个 key，Hash Tag 是绕不过去的重点
