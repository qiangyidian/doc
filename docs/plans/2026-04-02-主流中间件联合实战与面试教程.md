# 主流中间件联合实战与面试教程

**目标：** 把当前这个仓库里已经涉及到的主流中间件和核心组件，放到同一个真实后端业务场景里系统讲清楚。你看完之后，不只是知道每个中间件“单独怎么用”，还会知道它们在真实项目里怎么联合工作、各自负责什么、为什么这样设计、面试时常问什么。

**覆盖范围：**

- `MySQL`
- `MyBatis`
- `Redis`
- `RabbitMQ`
- `Kafka`
- `Elasticsearch`
- `Sa-Token`
- `Docker Compose`

**主线业务：** 我统一用一个“商城中台系统”来讲。它包含：

1. 用户登录和权限校验
2. 商品详情和缓存
3. 下单与库存扣减
4. 订单支付后的异步通知
5. 订单事件流和日志分发
6. 商品搜索和搜索索引同步

---

## 一、为什么一定要学“中间件联合使用”

很多人学中间件时，习惯一个一个单独学：

- Redis 学缓存
- RabbitMQ 学发送消息
- Kafka 学 topic
- Elasticsearch 学搜索

这样学有价值，但很容易出现一个问题：

**你知道每个组件的 API，却不知道它们在真实项目里怎么协作。**

真实后端项目里更常见的情况是：

- 数据最终落 MySQL
- 热点数据走 Redis
- 异步通知走 RabbitMQ
- 高吞吐事件流走 Kafka
- 搜索走 Elasticsearch
- 登录态和权限用 Sa-Token

也就是说，你以后在工作里真正面对的，不是单个中间件，而是一整条链路。

## 三、每个中间件在真实项目里到底负责什么

### 3.1 MySQL

作用：

- 保存最终真实数据
- 保存订单、商品、用户、支付结果

一句话理解：

**MySQL 是最终真相。**

### 3.2 MyBatis

作用：

- 帮你把 Java 代码和 SQL 串起来
- 负责参数传递、结果映射、动态 SQL

一句话理解：

**MyBatis 是 Java 和 MySQL 之间的桥。**

### 3.3 Redis

作用：

- 商品缓存
- 登录 Token / Session
- 分布式锁
- 热点数据保护

一句话理解：

**Redis 是高性能加速层。**

### 3.4 RabbitMQ

作用：

- 下单后异步发通知
- 支付成功后异步加积分
- 业务补偿消息

一句话理解：

**RabbitMQ 更偏业务异步解耦。**

### 3.5 Kafka

作用：

- 订单事件流
- 用户行为日志分发
- 高吞吐埋点数据流转

一句话理解：

**Kafka 更偏高吞吐事件流平台。**

### 3.6 Elasticsearch

作用：

- 商品搜索
- 日志搜索
- 多条件检索

一句话理解：

**Elasticsearch 更偏搜索和分析。**

### 3.7 Sa-Token

作用：

- 登录
- 权限
- Token 校验
- 会话管理

一句话理解：

**Sa-Token 是登录态和权限控制层。**

## 四、统一业务场景：商城中台系统

为了让所有中间件串起来，我统一定义下面 6 个真实场景。

### 场景 1：用户登录后台

使用：

- `Sa-Token + Redis`

目的：

- 登录成功后发 Token
- 后续请求通过 Token 识别用户
- Redis 存储登录态

### 场景 2：查询商品详情

使用：

- `MySQL + MyBatis + Redis`

目的：

- 商品详情先查 Redis
- Redis 没有再查 MySQL
- 再回写 Redis

### 场景 3：用户下单

使用：

- `MySQL + Redis`

目的：

- Redis 分布式锁保护库存
- MySQL 事务保证订单和库存一致

### 场景 4：支付成功后通知其他系统

使用：

- `RabbitMQ + MySQL`

目的：

- 支付主流程先成功
- 后续短信、积分、站内信异步处理

### 场景 5：订单事件流转

使用：

- `Kafka`

目的：

- 订单创建、支付、取消等事件发到 Kafka
- 给运营、风控、日志系统消费

### 场景 6：商品搜索

使用：

- `MySQL + Kafka/消息同步 + Elasticsearch`

目的：

- 商品主数据在 MySQL
- 搜索数据在 ES
- 商品变更后同步到 ES

## 七、场景 1：登录与权限，为什么要用 Sa-Token + Redis

登录场景最常见的链路是：

1. 用户输入账号密码
2. 后端查 MySQL 验证账号
3. 登录成功后生成 Token
4. Token 和 Session 信息存 Redis
5. 后续请求携带 Token 访问受保护接口

### 7.1 登录接口示例

```java
@RestController
@RequestMapping("/auth")
public class AuthController {

    @PostMapping("/login")
    public String login(@RequestParam String username, @RequestParam String password) {
        // 真实项目里这里应该查 MySQL 验证账号密码。
        if (!"admin".equals(username) || !"123456".equals(password)) {
            throw new RuntimeException("账号或密码错误");
        }

        // 登录成功后，Sa-Token 会为当前账号生成登录态。
        StpUtil.login(1001L);

        // 返回 Token 给前端，前端后续请求时要带上它。
        return StpUtil.getTokenValue();
    }
}
```

这段代码的作用：

- 演示登录接口的最小闭环
- `StpUtil.login()` 会建立登录态，后续请求就能识别当前用户

### 7.2 权限校验接口示例

```java
@RestController
@RequestMapping("/admin")
public class AdminController {

    @SaCheckLogin
    @GetMapping("/profile")
    public String profile() {
        // 只有已登录用户才能访问到这里。
        return "当前登录用户ID：" + StpUtil.getLoginIdAsLong();
    }
}
```

这段代码的作用：

- 演示登录校验注解怎么保护接口
- 这就是后端项目里“未登录不能访问后台接口”的基础实现方式

## 八、场景 2：商品详情为什么一定要用 MySQL + MyBatis + Redis

商品详情接口通常是高频读请求。

如果每次都直接打 MySQL，会有 3 个问题：

1. 数据库压力大
2. 延迟高
3. 热点商品会把 DB 打爆

所以标准写法通常是：

**先查 Redis，Redis 没有再查 MySQL，再回写 Redis。**

### 8.1 MyBatis 查询商品

```java
@Mapper
public interface ProductMapper {

    @Select("""
            select id, product_name, price, stock, status
            from t_product
            where id = #{id}
            """)
    ProductInfo findById(Long id);
}
```

这段代码的作用：

- 用 MyBatis 从 MySQL 查询商品
- 这是“真实数据源”层，Redis 没命中时最终还是要回到这里

### 8.2 Redis 缓存商品详情

```java
@Service
public class ProductService {

    @Resource
    private ProductMapper productMapper;

    @Resource
    private StringRedisTemplate stringRedisTemplate;

    @Resource
    private ObjectMapper objectMapper;

    public ProductInfo queryProduct(Long productId) throws Exception {
        String key = "mall:product:" + productId;

        // 第一步：先查 Redis。
        String cachedJson = stringRedisTemplate.opsForValue().get(key);
        if (cachedJson != null && !cachedJson.isBlank()) {
            return objectMapper.readValue(cachedJson, ProductInfo.class);
        }

        // 第二步：Redis 没有，再查 MySQL。
        ProductInfo product = productMapper.findById(productId);
        if (product == null) {
            throw new RuntimeException("商品不存在");
        }

        // 第三步：把最新数据回写到 Redis。
        stringRedisTemplate.opsForValue().set(
                key,
                objectMapper.writeValueAsString(product),
                Duration.ofMinutes(30)
        );

        return product;
    }
}
```

这段代码的作用：

- 演示最经典的 Cache-Aside 缓存模式
- 这是后端项目里 Redis 最常见、最重要的落地用法之一

## 九、场景 3：下单为什么要用 Redis 锁 + MySQL 事务

下单场景最怕两件事：

1. 超卖
2. 数据不一致

所以真实项目里非常常见的链路是：

1. 先用 Redis 分布式锁保护库存热点竞争
2. 再在 MySQL 事务里完成扣库存和创建订单

### 9.1 Redis 分布式锁保护库存

```java
@Service
public class OrderService {

    @Resource
    private RedissonClient redissonClient;

    public void createOrder(Long productId) {
        // 针对同一个商品加锁，避免并发下单时同时修改库存。
        RLock lock = redissonClient.getLock("lock:product:" + productId);

        lock.lock();
        try {
            // 真正下单逻辑交给事务方法处理。
            doCreateOrder(productId);
        } finally {
            // 一定要在 finally 里释放锁，避免死锁。
            lock.unlock();
        }
    }

    private void doCreateOrder(Long productId) {
        System.out.println("执行下单逻辑，productId=" + productId);
    }
}
```

这段代码的作用：

- 演示下单场景里 Redis 分布式锁的典型用法
- 它的目标不是替代数据库事务，而是先挡住并发冲击

### 9.2 MySQL 事务保证订单一致性

```java
@Service
public class OrderTxService {

    @Resource
    private ProductMapper productMapper;

    @Resource
    private OrderMapper orderMapper;

    @Transactional(rollbackFor = Exception.class)
    public void doCreateOrder(Long productId, Long userId) {
        // 第一步：扣减库存。
        int affectRows = productMapper.deductStock(productId);
        if (affectRows == 0) {
            throw new RuntimeException("库存不足");
        }

        // 第二步：创建订单。
        orderMapper.insertOrder(userId, productId);

        // 如果中途任何一步抛异常，事务会整体回滚。
    }
}
```

这段代码的作用：

- 演示 Spring 事务怎么保护“扣库存 + 创建订单”这组操作
- 这是电商、秒杀、抢购系统里特别经典的组合

## 十、场景 4：支付成功后为什么用 RabbitMQ

支付成功后，常常还会有很多“后续动作”：

1. 发短信
2. 发站内信
3. 增加积分
4. 通知发货系统

这些动作如果全都同步做，会拖慢主流程。  
所以支付主流程成功后，通常会发一条 MQ 消息，把后续动作异步化。

### 10.1 发送支付成功消息

```java
@Service
public class PaymentProducer {

    @Resource
    private RabbitTemplate rabbitTemplate;

    public void sendPaySuccessMessage(String orderNo) {
        // 把支付成功事件发到交换机。
        rabbitTemplate.convertAndSend(
                "payment.success.exchange",
                "payment.success.routing",
                orderNo
        );
    }
}
```

这段代码的作用：

- 演示 RabbitMQ 生产者发送消息
- 支付主流程只负责把事件发出去，不必等待所有后续动作执行完

### 10.2 消费支付成功消息

```java
@Component
public class PaymentSuccessConsumer {

    @RabbitListener(queues = "payment.success.queue")
    public void handle(String orderNo) {
        // 消费者收到支付成功消息后，可以异步做后续处理。
        System.out.println("收到支付成功消息，orderNo=" + orderNo);

        // 这里可以继续做：加积分、发短信、通知履约系统等。
    }
}
```

这段代码的作用：

- 演示 RabbitMQ 消费者处理异步消息
- 这就是业务异步解耦的核心价值

### 10.3 RabbitMQ 更适合什么场景

我建议你记住一句话：

**RabbitMQ 更适合“业务动作型异步消息”。**

典型例子：

- 下单后发通知
- 支付成功后加积分
- 商品更新后删缓存

## 十一、场景 5：订单事件流为什么用 Kafka

RabbitMQ 和 Kafka 都能发消息，但它们的定位不完全一样。

如果你的场景是：

- 订单创建事件
- 用户行为埋点
- 日志流
- 高吞吐数据分发

那 Kafka 往往更合适。

### 11.1 发送订单事件到 Kafka

```java
@Service
public class OrderEventProducer {

    @Resource
    private KafkaTemplate<String, String> kafkaTemplate;

    public void sendOrderCreatedEvent(String orderNo) {
        // 向 Kafka 的 topic 发送订单创建事件。
        kafkaTemplate.send("order-events", orderNo);
    }
}
```

这段代码的作用：

- 演示 Kafka 生产者发送事件
- 这类事件通常会被多个系统消费，例如风控、日志、运营分析系统

### 11.2 消费订单事件

```java
@Component
public class OrderEventConsumer {

    @KafkaListener(topics = "order-events", groupId = "order-analysis-group")
    public void consume(String orderNo) {
        // 这里可以做订单分析、用户行为分析、经营数据统计等。
        System.out.println("Kafka 收到订单事件：" + orderNo);
    }
}
```

这段代码的作用：

- 演示 Kafka 消费者的基本写法
- Kafka 更适合高吞吐、可扩展的事件流消费

### 11.3 RabbitMQ 和 Kafka 怎么选

你可以先这样记：

- `RabbitMQ`：更偏业务解耦、路由灵活、单条消息驱动
- `Kafka`：更偏高吞吐事件流、日志流、埋点流、大规模分发

不是谁替代谁，而是它们解决的问题重点不同。

## 十二、场景 6：搜索为什么一定要 MySQL + Elasticsearch 联合使用

很多新手会误以为：

- “既然 ES 搜索这么强，我把商品全放 ES 就行了”

这不是主流企业做法。

更常见的设计是：

- MySQL 保存商品主数据
- Elasticsearch 保存搜索索引数据

也就是说：

- **MySQL 是主库**
- **ES 是搜索副本**

### 12.1 商品数据写入 MySQL

```sql
-- 商品主数据先写入 MySQL。
INSERT INTO t_product (product_name, price, stock, status)
VALUES ('蓝牙音箱', 259.00, 60, 1);
```

这段代码的作用：

- 演示商品主数据先写 MySQL
- 这是主流设计，因为 MySQL 更适合作为最终真实数据源

### 12.2 把商品同步到 Elasticsearch

```java
@Service
public class ProductSearchSyncService {

    @Resource
    private ElasticsearchClient elasticsearchClient;

    public void syncProduct(ProductDoc productDoc) throws IOException {
        // 把商品写入 ES 索引。
        elasticsearchClient.index(i -> i
                .index("product_index")
                .id(String.valueOf(productDoc.getId()))
                .document(productDoc)
        );
    }
}
```

这段代码的作用：

- 演示商品数据同步到 ES
- 真实项目里通常会通过 MQ 或 Kafka 触发这一步，而不是每次都同步阻塞主流程

### 12.3 搜索商品

```java
@Service
public class ProductSearchService {

    @Resource
    private ElasticsearchClient elasticsearchClient;

    public SearchResponse<ProductDoc> search(String keyword) throws IOException {
        // 按商品名称做全文检索。
        return elasticsearchClient.search(s -> s
                        .index("product_index")
                        .query(q -> q
                                .match(m -> m
                                        .field("productName")
                                        .query(keyword)
                                )
                        ),
                ProductDoc.class
        );
    }
}
```

这段代码的作用：

- 演示 ES 最常见的全文检索能力
- 这正是数据库 `like '%关键词%'` 很难高效完成的事情

## 十三、这些中间件联合起来，真实链路长什么样

我建议你把整个系统记成下面这 6 步：

1. 用户登录  
   `Sa-Token + Redis`

2. 用户查商品详情  
   `Redis -> MySQL(MyBatis)`

3. 用户下单  
   `Redis 分布式锁 + MySQL 事务`

4. 支付成功  
   `RabbitMQ 异步通知后续业务`

5. 订单事件流转  
   `Kafka 发给多个消费系统`

6. 搜索商品  
   `Elasticsearch 查询，MySQL 做主数据`

这 6 步几乎就是现代 Java 后端项目里最主流的一条中间件组合链路。

## 十四、主流面试题总结：MySQL / MyBatis

### 14.1 面试题：MySQL 和 Redis 在项目里分别负责什么

推荐回答：

- MySQL 是最终真实数据源，保存订单、商品、用户等主数据
- Redis 是缓存层和高性能辅助层，常用于缓存、分布式锁、登录态、热点保护

### 14.2 面试题：为什么不推荐 `select *`

推荐回答：

- 会多查无用字段
- 会增加网络开销
- 可读性差
- 表结构变化时风险更大

### 14.3 面试题：联合索引最左匹配原则是什么

推荐回答：

如果你建了联合索引：

- `(user_id, order_status, create_time)`

那么查询条件要尽量从左往右使用这些字段，才能更好命中索引。

### 14.4 面试题：MyBatis 有什么优点

推荐回答：

- SQL 可控
- 动态 SQL 灵活
- 适合复杂查询
- 更适合对 SQL 性能敏感的项目

---

## 十五、主流面试题总结：Redis

### 15.1 面试题：Redis 常见使用场景有哪些

推荐回答：

1. 缓存热点数据
2. 登录态和 Session
3. 分布式锁
4. 计数器
5. 排行榜

### 15.2 面试题：缓存穿透、击穿、雪崩分别是什么

推荐回答：

- **缓存穿透**：查一个根本不存在的数据，请求穿过缓存直接打到数据库
- **缓存击穿**：某个热点 key 刚好失效，大量请求同时打到数据库
- **缓存雪崩**：大量 key 在同一时间集中失效，数据库瞬时压力过大

### 15.3 面试题：Redis 分布式锁为什么要加过期时间

推荐回答：

因为如果服务宕机但锁没释放，没有过期时间就可能造成死锁。

### 15.4 面试题：Redis 为什么适合做商品缓存

推荐回答：

- 内存访问快
- 读多写少场景很适合
- 热点商品可显著减轻 MySQL 压力

## 十六、主流面试题总结：RabbitMQ

### 16.1 面试题：RabbitMQ 里 exchange、queue、routingKey 分别是什么

推荐回答：

- `exchange`：交换机，负责路由消息
- `queue`：队列，真正存消息
- `routingKey`：路由键，用来匹配投递规则

### 16.2 面试题：RabbitMQ 为什么适合支付成功后的异步处理

推荐回答：

因为支付是主流程，积分、短信、通知是从流程。  
把从流程异步化后，主流程可以更快返回，也能降低耦合。

### 16.3 面试题：消息为什么要手动 ACK

推荐回答：

手动 ACK 能让你在业务真正处理成功后再确认消息，避免消息丢失。

### 16.4 面试题：RabbitMQ 和 Kafka 的区别

推荐回答：

- RabbitMQ 更偏业务异步和灵活路由
- Kafka 更偏高吞吐事件流和日志流

## 十七、主流面试题总结：Kafka

### 17.1 面试题：Kafka 的 Topic 和 Partition 是什么

推荐回答：

- `Topic`：消息主题
- `Partition`：分区，Kafka 通过分区实现扩展和并行消费

### 17.2 面试题：为什么 Kafka 吞吐量高

推荐回答：

- 顺序写磁盘
- 分区机制
- 批量发送
- 批量拉取

### 17.3 面试题：Kafka 如何保证同一业务键有序

推荐回答：

把同一个业务键，例如 `orderNo`，路由到同一个 partition，这样同一个分区内的消息顺序就能保证。

### 17.4 面试题：Kafka 适合什么场景

推荐回答：

- 日志流
- 埋点流
- 订单事件流
- 大数据采集和分析链路

## 十八、主流面试题总结：Elasticsearch

### 18.1 面试题：为什么数据库不适合直接做全文搜索

推荐回答：

因为数据库更擅长结构化数据的精确查询，而 ES 基于倒排索引，全文检索能力更强。

### 18.2 面试题：Elasticsearch 为什么快

推荐回答：

核心原因是倒排索引。  
它不是按行扫数据，而是先根据词项快速定位文档。

### 18.3 面试题：为什么 MySQL 和 ES 要同时存在

推荐回答：

- MySQL 负责主数据
- ES 负责搜索副本
- 两者分工不同，不能简单互相替代

### 18.4 面试题：ES 适合什么场景

推荐回答：

- 商品搜索
- 文章搜索
- 日志搜索
- 多条件检索和聚合分析

## 十九、主流面试题总结：Sa-Token

### 19.1 面试题：Sa-Token 和 JWT 有什么区别

推荐回答：

- JWT 更偏一种 Token 格式
- Sa-Token 是一整套认证授权框架，封装了登录、权限、会话、踢人下线等能力

### 19.2 面试题：为什么登录态常常配 Redis

推荐回答：

因为分布式系统里，如果登录态只存在单机内存中，多个节点之间无法共享。  
放到 Redis 后，所有服务节点都能共享会话数据。

### 19.3 面试题：权限校验一般放在哪一层

推荐回答：

- 登录校验通常放在拦截器或注解层
- 更细粒度的业务权限控制通常还要放在 Service 层补充校验

## 二十、如果你以后自己做一个中型项目，我建议的中间件组合

如果你是做一个典型 Java 后端项目，我建议的组合是：

   负责业务异步解耦

这套组合非常主流，也非常贴近真实企业项目。

## 二十二、最后总结

你可以把这份教程压缩成一句话：

**真正的后端开发，不是会一个中间件，而是会让多个中间件在正确的位置上协作。**

你以后做项目时，最重要的不是：

- Redis 会不会 set/get
- RabbitMQ 会不会发消息
- Kafka 会不会建 topic

而是：

- 什么数据应该放 MySQL
- 什么请求应该走 Redis
- 什么动作应该异步化
- 什么事件应该进 Kafka
- 什么查询应该交给 Elasticsearch

这才是“中间件联合使用”的核心能力。
