## 一、为什么后端开发必须学会“正确使用异常”

很多初学者第一次接触异常时，会把异常当成“报错信息”。  
但在真实后端开发里，异常远不只是报错，它本质上是一种 **错误传播机制**。

后端开发里最常见的 4 个错误做法是：

1. 到处写 `try-catch`，把业务代码包得很臃肿
2. 捕获异常后什么都不做，直接吞掉
3. 为了省事，所有地方都 `throw new RuntimeException("失败")`
4. 把真正的底层异常丢了，最后只剩一句“系统异常”

正确使用异常的目标是：

- **让错误能被准确发现**
- **让错误能被正确传播**
- **让错误能被统一处理**
- **让业务代码保持干净**

## 二、先建立异常体系的全局认知

在 Java 中，所有异常和错误的共同父类都是 `Throwable`。

它下面有两条主线：

1. `Error`
2. `Exception`

你可以先把它理解成：

- `Error`：JVM 或系统级严重错误，通常不是业务代码该处理的
- `Exception`：程序运行中可能遇到的问题，开发者通常需要面对

### 2.1 异常体系结构图

```text
Throwable
├── Error
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   └── NoClassDefFoundError
└── Exception
    ├── IOException
    ├── SQLException
    ├── ClassNotFoundException
    └── RuntimeException
        ├── NullPointerException
        ├── IllegalArgumentException
        ├── NumberFormatException
        ├── IndexOutOfBoundsException
        └── ClassCastException
```

这段代码的作用：

- 帮你建立 Java 异常体系的全局结构
- 以后看到一个异常名时，你能先判断它属于 `Error`、受检异常，还是运行时异常

### 2.2 `Error` 和 `Exception` 的区别

最核心的区别不是“名字不一样”，而是处理责任不同：

- `Error` 通常表示 JVM 层面的问题，例如内存溢出、类加载失败
- `Exception` 通常表示程序运行中遇到的问题，例如文件找不到、参数错误、空指针

在真实开发里，一般不要主动去捕获 `Error`。  
因为这种问题通常不是你在业务层 `catch` 一下就能修好的。

## 三、受检异常和非受检异常到底怎么理解

这是 Java 异常里最重要的一条分界线。

### 3.1 受检异常 `Checked Exception`

受检异常的特点是：

- 编译器会强制你处理
- 如果你不 `catch`，就必须 `throws`

典型例子：

- `IOException`
- `SQLException`
- `ClassNotFoundException`

### 3.2 非受检异常 `Unchecked Exception`

非受检异常通常指：

- `RuntimeException`
- 它的所有子类

特点是：

- 编译器不会强制你处理
- 即使你不写 `catch` 或 `throws`，代码也能编译通过

典型例子：

- `NullPointerException`
- `IllegalArgumentException`
- `NumberFormatException`

### 3.3 我在项目里更推荐哪一种

在现代 Java 后端开发里，**绝大多数业务异常和程序错误更常用 `RuntimeException` 体系**。

原因很简单：

1. 受检异常会把 `throws` 层层往上传，容易让代码很啰嗦
2. 很多异常其实调用方并没有能力恢复
3. Spring 生态里，大量异常处理方式天然更偏向运行时异常

但这不意味着 `Checked Exception` 没用。  
像文件 IO、反射加载、数据库驱动等场景，受检异常仍然很常见。

## 四、`throw` 和 `throws` 不要再混了

很多初学者会把这两个关键字混掉。你必须直接区分开：

- `throw`：在方法内部，主动抛出一个异常对象
- `throws`：写在方法签名上，声明“这个方法可能抛出什么异常”

### 4.1 `throw` 示例

```java
public class ThrowDemo {

    public void checkAge(int age) {
        // 当参数不符合业务要求时，主动抛出异常对象。
        if (age < 0) {
            // throw 是“立刻把异常扔出去”。
            throw new IllegalArgumentException("年龄不能小于 0");
        }

        System.out.println("年龄校验通过：" + age);
    }
}
```

这段代码的作用：

- 演示 `throw` 是怎么在方法内部主动抛异常的
- 这是参数校验场景里最常见的用法

### 4.2 `throws` 示例

```java
import java.io.IOException;
import java.nio.file.Files;
import java.nio.file.Path;

public class ThrowsDemo {

    public String readFile(String filePath) throws IOException {
        // Files.readString 可能抛出 IOException。
        // 这里没有在方法内部处理，而是通过 throws 往上声明。
        return Files.readString(Path.of(filePath));
    }
}
```

这段代码的作用：

- 演示 `throws` 是在方法签名上声明可能抛出的异常
- 这通常用于“当前方法不想处理，让调用方决定怎么办”的场景

## 五、`try-catch-finally` 的正确使用方式

这是 Java 异常里最基础，但也是最容易写错的语法。

### 5.1 标准写法

```java
public class TryCatchFinallyDemo {

    public int divide(int a, int b) {
        try {
            // try 块里写可能出问题的代码。
            System.out.println("开始执行除法计算");
            return a / b;
        } catch (ArithmeticException e) {
            // catch 块里处理特定异常。
            System.out.println("发生了算术异常：" + e.getMessage());
            return 0;
        } finally {
            // finally 块通常用来做收尾动作。
            // 无论 try/catch 是否正常执行，这里通常都会执行。
            System.out.println("finally 收尾逻辑执行");
        }
    }
}
```

这段代码的作用：

- 演示 `try-catch-finally` 的最标准结构
- `try` 里是风险代码，`catch` 里是异常处理，`finally` 里是收尾逻辑

### 5.2 `finally` 一定会执行吗

大多数情况下会，但不是绝对 100%。  
例如：

- JVM 进程被强制终止
- `System.exit()` 被调用
- 机器直接断电

所以更准确的说法是：

- **在正常的 Java 方法执行流程里，`finally` 通常会执行**

### 5.3 为什么不要在 `finally` 里写 `return`

这是经典坑点。

```java
public class FinallyReturnDemo {

    public int test() {
        try {
            // try 里先返回 1。
            return 1;
        } finally {
            // finally 里又返回 2，会把前面的返回值覆盖掉。
            return 2;
        }
    }
}
```

这段代码的作用：

- 演示为什么不应该在 `finally` 里写 `return`
- 它会覆盖原本的返回值，导致代码逻辑非常难排查

## 六、必须会的资源关闭写法：`try-with-resources`

在后端开发里，文件流、网络流、数据库连接这类资源都必须关闭。  
现代 Java 更推荐 `try-with-resources`，而不是手写 `finally close()`。

### 6.1 推荐写法

```java
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class TryWithResourcesDemo {

    public String readFirstLine(String filePath) {
        // try-with-resources 会在代码块结束后，自动调用资源的 close() 方法。
        try (BufferedReader reader = new BufferedReader(new FileReader(filePath))) {
            return reader.readLine();
        } catch (IOException e) {
            // 这里统一把底层 IO 异常包装成更明确的业务语义。
            throw new RuntimeException("读取文件失败：" + filePath, e);
        }
    }
}
```

这段代码的作用：

- 演示现代 Java 推荐的资源关闭方式
- 自动关闭资源，代码更短，也更不容易漏掉 `close()`

### 6.2 为什么它比 `try-finally` 更好

因为它能解决 3 个问题：

1. 代码更短
2. 不容易忘记关闭资源
3. 多资源关闭时异常信息更清晰

## 七、后端开发里最常见的异常处理场景

这部分我不按教科书讲，而是直接按真实项目场景来讲。

### 场景 1：参数不合法

例如：

- 用户传了负数金额
- 用户传了空手机号
- 分页参数小于 1

这种情况最常见的做法是抛出：

- `IllegalArgumentException`
- 或者自定义业务异常

```java
public class UserService {

    public void register(String phone, Integer age) {
        // 入参为空，是非常典型的参数错误。
        if (phone == null || phone.isBlank()) {
            throw new IllegalArgumentException("手机号不能为空");
        }

        // 业务规则不满足时，也应该明确抛异常。
        if (age == null || age < 18) {
            throw new IllegalArgumentException("注册用户年龄必须大于等于 18 岁");
        }

        System.out.println("注册成功");
    }
}
```

这段代码的作用：

- 演示参数校验失败时怎么主动抛异常
- 这比返回 `false` 或 `null` 更明确，也更容易统一处理

### 场景 2：调用第三方接口失败

真实项目里非常常见的场景是：

- 调短信接口
- 调支付接口
- 调地图接口
- 调其他微服务接口

这类异常不要原样直接往外抛，通常要做“包装”。

```java
public class SmsService {

    public void sendCode(String phone) {
        try {
            // 这里模拟第三方短信平台调用。
            callSmsVendor(phone);
        } catch (Exception e) {
            // 不要只抛一个没有上下文的 RuntimeException。
            // 这里要保留原始异常 e，形成异常链，方便后续排查。
            throw new RuntimeException("短信发送失败，手机号：" + phone, e);
        }
    }

    private void callSmsVendor(String phone) {
        throw new IllegalStateException("第三方短信服务超时");
    }
}
```

这段代码的作用：

- 演示“异常包装”和“异常链”怎么写
- 上层能看到更业务化的错误，下层真实异常也没有丢

### 场景 3：数据库查询结果不存在

这在后端项目里非常常见，例如：

- 根据用户 ID 查用户，结果没有
- 根据订单号查订单，结果没有

这种情况通常不是系统错误，而是**业务结果不满足预期**。

```java
public class OrderService {

    public String queryOrderStatus(String orderNo) {
        // 这里用 null 模拟数据库没有查到订单。
        String orderStatusFromDb = null;

        if (orderStatusFromDb == null) {
            // 没查到数据时，应该抛出业务异常，而不是继续往下执行。
            throw new BusinessException("订单不存在，orderNo=" + orderNo);
        }

        return orderStatusFromDb;
    }
}
```

这段代码的作用：

- 演示“没查到数据”这种场景应该抛业务异常
- 这类问题本质上不是空指针错误，而是业务层应该明确定义的状态

### 场景 4：批量任务里单条处理失败

比如导入 Excel、批量发券、批量同步数据时，经常会出现：

- 某一条数据失败
- 但整个批次不一定要立刻全停

```java
import java.util.List;

public class BatchImportService {

    public void importUsers(List<String> rows) {
        for (String row : rows) {
            try {
                // 每一行都单独处理，避免一条异常把整个批次直接打断。
                parseAndSave(row);
            } catch (Exception e) {
                // 这里记录失败行，但继续处理下一条。
                System.out.println("导入失败，原始数据：" + row + "，原因：" + e.getMessage());
            }
        }
    }

    private void parseAndSave(String row) {
        if (row == null || row.isBlank()) {
            throw new IllegalArgumentException("导入行不能为空");
        }
    }
}
```

这段代码的作用：

- 演示批量处理里“单条异常隔离”的常见写法
- 这类场景不是不抛异常，而是捕获后按业务策略继续执行

### 场景 5：线程池和异步任务里的异常

异步任务是异常最容易被忽略的地方。

```java
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class AsyncExceptionDemo {

    private final ExecutorService executorService = Executors.newFixedThreadPool(2);

    public void runTask() {
        CompletableFuture.supplyAsync(() -> {
            // 异步任务内部如果出错，不会自动像同步代码那样暴露在主线程里。
            throw new IllegalStateException("异步任务执行失败");
        }, executorService).exceptionally(ex -> {
            // exceptionally 用来兜底处理异步链路中的异常。
            System.out.println("捕获到异步异常：" + ex.getMessage());
            return null;
        });
    }
}
```

这段代码的作用：

- 演示异步场景下异常不会自然回到主线程
- 如果你不额外处理，很多异步异常就会被你“看不见”

### 场景 6：Controller 不要到处手写 `try-catch`

真实 Spring Boot 项目里，不建议每个 Controller 方法都这样写：

```java
public class WrongControllerStyle {

    public String getUser(Long userId) {
        try {
            if (userId == null) {
                throw new IllegalArgumentException("用户 ID 不能为空");
            }
            return "查询成功";
        } catch (Exception e) {
            // 这是不推荐的写法，因为所有接口都这样写会非常臃肿。
            return "失败：" + e.getMessage();
        }
    }
}
```

这段代码的作用：

- 演示一种不推荐的写法
- 问题不在于不能 catch，而在于每个接口都自己包一层，会让代码非常难维护

## 八、自定义异常：后端项目里最常见的做法

很多业务错误并不适合直接用 JDK 自带异常描述。  
例如：

- 余额不足
- 订单已取消
- 用户已被冻结
- 库存不足

这时就应该自定义业务异常。

### 8.1 最简单的自定义业务异常

```java
public class BusinessException extends RuntimeException {

    // 这里增加一个业务错误码字段，方便前后端约定错误码。
    private final Integer code;

    public BusinessException(String message) {
        // 这里调用父类构造器，把错误信息交给 RuntimeException 保存。
        super(message);
        this.code = 5000;
    }

    public BusinessException(Integer code, String message) {
        super(message);
        this.code = code;
    }

    public Integer getCode() {
        return code;
    }
}
```

这段代码的作用：

- 演示最常见的自定义业务异常写法
- 真实项目里一般会增加 `code` 字段，方便前端识别具体错误类型

### 8.2 使用自定义业务异常

```java
public class BalanceService {

    public void pay(Integer balance, Integer amount) {
        // 余额为空属于非法状态。
        if (balance == null) {
            throw new BusinessException(5001, "账户余额不能为空");
        }

        // 余额不足属于明确的业务异常。
        if (balance < amount) {
            throw new BusinessException(5002, "余额不足，无法完成支付");
        }

        System.out.println("支付成功");
    }
}
```

这段代码的作用：

- 演示业务层怎么使用自定义异常
- 这类异常通常会被全局异常处理器统一接住并转换成标准 JSON 返回

## 九、异常链：为什么包装异常时一定要保留原始异常

很多人会这样写：

```java
throw new RuntimeException("查询失败");
```

这种写法的问题是：**原始异常丢了**。  
排查问题时你只能看到“查询失败”，却不知道到底是 SQL 错了、网络断了，还是空指针。

正确写法应该是：

```java
public class ExceptionChainDemo {

    public void queryData() {
        try {
            // 模拟底层数据库调用异常。
            throw new IllegalStateException("数据库连接超时");
        } catch (Exception e) {
            // 第二个参数 e 就是“原始异常”，这样异常链就保留下来了。
            throw new RuntimeException("查询用户信息失败", e);
        }
    }
}
```

这段代码的作用：

- 演示异常链的正确写法
- 后续打日志时，你既能看到业务语义，也能看到真实根因

## 十、什么时候该捕获异常，什么时候该继续往外抛

这是异常使用里最容易混乱的问题。

### 10.1 适合捕获的情况

1. 你能真正处理它  
   例如：
   - 给默认值
   - 重试
   - 记录失败后继续处理下一条

2. 你要做资源释放或统一日志补充  

3. 你要把底层异常包装成更清晰的业务异常  

### 10.2 不适合捕获的情况

1. 你 catch 了也不知道怎么处理  
2. 你 catch 之后只打印一句日志然后什么都不做  
3. 你 catch 之后把异常吞掉，导致调用方误以为成功  

一句话总结：

- **能处理再 catch**
- **不能处理就往上抛**

## 十一、后端开发里非常重要的 5 个异常使用原则

### 原则 1：不要吞异常

错误示例：

```java
public class SwallowExceptionDemo {

    public void save() {
        try {
            throw new RuntimeException("保存失败");
        } catch (Exception e) {
            // 什么都不做，异常被直接吞掉了。
        }
    }
}
```

这段代码的作用：

- 演示“吞异常”这种最危险的写法
- 调用方会误以为方法成功执行了，实际内部已经失败

### 原则 2：不要只打印 `e.getMessage()`

错误示例：

```java
public class WrongLogDemo {

    public void test() {
        try {
            throw new RuntimeException("数据库连接失败");
        } catch (Exception e) {
            // 只打印 message，会丢失完整堆栈。
            System.out.println(e.getMessage());
        }
    }
}
```

这段代码的作用：

- 演示为什么只打印 `message` 不够
- 真正排错时，完整堆栈往往比一句文本更重要

### 原则 3：优先抛更具体的异常

例如：

- 参数错误优先用 `IllegalArgumentException`
- 字符串转数字失败优先用 `NumberFormatException`
- 类型转换失败优先用 `ClassCastException`

而不是一律：

- `throw new RuntimeException("失败")`

### 原则 4：不要把异常对象定义成静态变量复用

异常对象内部带有堆栈信息。  
如果你复用同一个异常对象，堆栈会混乱，排查问题会非常痛苦。

### 原则 5：异常信息一定要有业务含义

例如：

- `订单不存在，orderNo=ORD-1001`
- `短信发送失败，phone=13800000000`

这比单纯的：

- `失败`
- `error`

要强太多。

## 十二、Spring Boot 项目里最常见的异常落地方式

你如果在做后端接口开发，最终几乎一定会落到：

1. 业务层主动抛异常
2. 全局异常处理器统一接住
3. 返回标准 JSON 给前端

### 12.1 自定义统一返回对象

```java
import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

@Data
@NoArgsConstructor
@AllArgsConstructor
public class Result<T> {

    private Integer code;
    private String message;
    private T data;

    public static <T> Result<T> success(T data) {
        return new Result<>(200, "success", data);
    }

    public static <T> Result<T> fail(Integer code, String message) {
        return new Result<>(code, message, null);
    }
}
```

这段代码的作用：

- 定义接口统一返回格式
- 这样异常处理器就能把各种异常统一转换成标准 JSON

### 12.2 全局异常处理器

```java
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        // 业务异常通常是我们主动抛出的“可预期错误”。
        return Result.fail(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(IllegalArgumentException.class)
    public Result<Void> handleIllegalArgumentException(IllegalArgumentException e) {
        // 参数错误也可以单独拦截，返回更明确的提示。
        return Result.fail(400, e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        // 这里取出第一个参数校验错误信息返回给前端。
        String message = e.getBindingResult().getFieldError() != null
                ? e.getBindingResult().getFieldError().getDefaultMessage()
                : "参数校验失败";
        return Result.fail(400, message);
    }

    @ExceptionHandler(Exception.class)
    public Result<Void> handleException(Exception e) {
        // 最后一定要有一个兜底异常处理器，防止未知异常直接向前端泄露堆栈。
        return Result.fail(500, "系统繁忙，请稍后再试");
    }
}
```

这段代码的作用：

- 演示 Spring Boot 全局异常处理的核心写法
- 业务层只负责抛异常，最终由 `@RestControllerAdvice` 统一收口

### 12.3 Controller 和 Service 的推荐配合方式

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class OrderController {

    private final OrderQueryService orderQueryService = new OrderQueryService();

    @GetMapping("/orders/status")
    public Result<String> queryStatus(@RequestParam String orderNo) {
        // Controller 里不建议自己包 try-catch。
        // 直接调用业务逻辑，异常交给全局处理器统一接住。
        String status = orderQueryService.queryStatus(orderNo);
        return Result.success(status);
    }
}

class OrderQueryService {

    public String queryStatus(String orderNo) {
        if (orderNo == null || orderNo.isBlank()) {
            throw new IllegalArgumentException("订单号不能为空");
        }

        if ("ORD-404".equals(orderNo)) {
            throw new BusinessException(5004, "订单不存在");
        }

        return "PAID";
    }
}
```

这段代码的作用：

- 演示 Controller 不要到处手写 `try-catch`
- 真实业务异常在 Service 层抛出，最终交给全局异常处理器统一兜底

## 十三、一个完整的异常使用建议链路

如果你在真实项目里写一个接口，最推荐的异常处理链路通常是这样的：

1. **Controller**
   只接请求，不写一堆 `try-catch`

2. **Service**
   做参数校验、业务校验，必要时主动抛 `BusinessException`

3. **DAO / 第三方调用层**
   如果底层出错，可以向上抛，或者包装成更业务化的异常

4. **GlobalExceptionHandler**
   统一把异常转换成标准 JSON

5. **日志系统**
   记录必要的错误日志和堆栈，方便排查

这才是后端开发里最主流、最清晰的做法。

## 十四、异常使用中最容易踩的坑

### 坑 1：把业务异常和系统异常混在一起

例如：

- 用户余额不足，本来是业务异常
- 结果你直接返回“系统异常”

这会让前端和用户都误判问题性质。

### 坑 2：为了图省事，全都 `catch Exception`

如果你在很多地方一把抓：

```java
catch (Exception e)
```

那你就很容易丢掉更细粒度的错误处理机会。

### 坑 3：日志重复打印

如果底层已经完整打印过异常堆栈了，上层再原样打印一遍，日志会很乱。  
所以日志也要有层次，不能每一层都疯狂打堆栈。

### 坑 4：把 `null` 当成异常处理机制

例如：

- 查不到用户返回 `null`
- 调用方没判断直接空指针

这种写法比直接抛业务异常更危险。

### 坑 5：异步任务的异常没人处理

多线程、线程池、`CompletableFuture`、定时任务里的异常，如果不显式兜底，很容易悄悄失败。

## 十五、你在项目里可以怎么记住异常的使用原则

我建议你把异常使用压缩成下面这 7 句话：

1. `Error` 一般不处理，`Exception` 才是你主要面对的对象
2. 参数错误优先用具体异常，不要动不动就 `RuntimeException`
3. 能处理再 `catch`，不能处理就往上抛
4. 业务错误要用自定义异常表达清楚
5. 包装异常时要保留原始异常，别断异常链
6. 能自动关资源就用 `try-with-resources`
7. Spring Boot 项目里，最终要落到全局异常处理

## 十八、最后总结

你可以把 Java 异常理解成一句话：

**异常不是为了让程序报错，而是为了让错误被明确地表达、传播和处理。**

真正的后端开发里，异常使用得好，代码会很清晰：

- 参数问题直接拦下
- 业务问题明确抛出
- 底层问题向上包装
- 接口层统一处理

这样代码既干净，又容易排错，也更符合真实企业项目的写法。
