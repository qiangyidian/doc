# RAG 精排与压缩本地适配设计

**目标：** 在现有第二阶段 `Ollama + PGVector` 项目基础上，继续实现第三阶段的精排、上下文压缩和流水线调试能力，同时保持当前本地运行配置不变。

**当前基础：**
- 已完成第二阶段 `查询改写 -> 混合检索 -> RRF 融合 -> 问答/评估`。
- 当前模型提供方不是文档中的 OpenAI，而是本地 `Ollama`。
- 当前向量表是 `vector_store_ollama`，维度是 `768`，默认分类是 `知识`，默认目录是 `knowledge-base`。

**设计结论：**
- 类结构、接口层次、DTO 形状、流水线阶段严格遵循 `2026-04-14-RAG精排与压缩实现文档.md`。
- 仅在配置层做适配：继续使用 `spring.ai.ollama`、`vector_store_ollama`、`768` 维和现有分类/目录设置。
- 由于本地小模型在结构化输出上稳定性弱于云端大模型，`RerankService` 和 `ContextCompressionService` 除了文档中的 LLM 主路径外，必须补充“解析失败时的规则降级路径”，保证接口可用。

**第三阶段流水线：**
1. `QueryRewriteService` 输出检索查询列表。
2. `HybridRetrievalService` 输出候选 `RetrievedChunk`，并按 `candidateTopK` 截断为精排候选集。
3. `RerankService` 对候选做相关性精排，产出 `rerankScore` 和保留集合。
4. `ContextCompressionService` 先做去重，再对保留片段做压缩或规则裁剪，产出 `compressedContent`、最终上下文长度统计和 `selectedForAnswer`。
5. `RagChatServiceImpl` 基于压缩后的上下文组装 Prompt。
6. `PipelineDebugController` 输出候选、精排后、压缩后三个阶段快照。

**适配约束：**
- 不切换到 OpenAI。
- 不新增数据库表，不重建 PG 数据卷。
- 不破坏已有第二阶段接口。
- 新增字段必须向后兼容当前 JSON 风格。

**验证标准：**
- 第三阶段配置参数已出现在 `application.yml` 和 `RagProperties`。
- `/api/pipeline/debug` 能输出 `retrievedChunks / rerankedChunks / compressedChunks` 和压缩前后长度。
- `/api/rag/ask` 能输出 `rerankScore`、`compressedContent` 和压缩长度统计。
- 在本地运行条件下，即使 LLM 结构化输出失败，也能通过降级逻辑完成精排与压缩。
