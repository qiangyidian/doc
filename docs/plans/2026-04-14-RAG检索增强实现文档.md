# Spring Boot + Spring AI + PGVector 检索增强版 RAG 实现文档

**目标：** 基于上一份 `2026-04-14-最小RAG实现文档.md` 继续升级项目，完整演示“目录批量同步 -> 增量更新 -> 查询改写 -> 向量召回 + 关键词召回 -> 融合排序 -> 召回评估 -> 大模型回答”这条第二阶段检索增强链路。

**架构说明：** 这份文档严格承接上一阶段的最小可用 RAG 项目，不推翻原有结构，而是在原项目上逐步加入 `知识目录同步`、`增量更新`、`pg_trgm 关键词检索`、`混合检索融合`、`查询改写` 和 `Hit@K / MRR 召回评估`。这一阶段仍然不引入 `Reranker`、`上下文压缩` 和 `多租户治理`，重点先把“召回质量可观察、可增强、可验证”这件事做好。

**技术栈：** Java 17、Spring Boot 3.5.x、Spring AI 1.1.x、Spring Web、MyBatis、PostgreSQL 16、PGVector、`pg_trgm`、Docker Compose

---

## 一、这一阶段到底解决什么问题

上一份最小版文档，解决的是“先把 RAG 跑起来”。

但只跑起来，还远远不够。

当你开始真正用它回答问题时，很快就会遇到下面这些问题：

1. 只支持单文件导入，知识维护效率太低
2. 文件改过以后，旧向量还留在库里，知识会“脏”
3. 纯向量检索对专有名词、缩写、接口名、表名、代码词汇不够稳定
4. 用户问题很口语化时，直接拿原问题去检索，召回效果不理想
5. 你根本不知道当前检索效果到底是“偶尔碰巧命中”，还是“稳定可用”

所以，第二阶段的核心目标不是再做一个“更花哨”的接口，而是解决 4 件很实际的事：

1. 让知识库具备最基础的**目录级维护能力**
2. 让检索从“单路向量召回”升级成“**双路召回 + 融合排序**”
3. 让提问在检索前先经过一次**查询改写**
4. 让检索质量开始具备**最小评估闭环**

## 二、先讲清楚这一阶段的边界

为了和教程顺序保持一致，这份文档明确只做下面这些事情：

### 1、这份文档一定会做的能力

1. 目录批量同步
2. 基于 `file_path + file_hash` 的增量更新
3. 基于 `pg_trgm` 的轻量关键词检索
4. 向量检索 + 关键词检索的融合排序
5. 基于 `ChatClient` 的查询改写
6. 基于 `Hit@K`、`MRR` 的最小召回评估

### 2、这份文档暂时不做的能力

1. `Reranker` 精排
2. 上下文压缩与裁剪
3. 多租户隔离
4. 文档权限体系
5. 真正大规模的离线评测平台

### 3、为什么关键词检索这里选 `pg_trgm`，不是直接上 PostgreSQL 全文检索

这个点要说明白。

如果你的场景是英文文本，PostgreSQL 原生全文检索已经很好用了。

但当前我们是一个**中文教程演示项目**，第二阶段又要求尽量继续沿用 `PostgreSQL + PGVector` 这套最小基础设施，所以这里我会采用一个更务实的方案：

1. 用 `pg_trgm` 给 `vector_store.content` 建索引
2. 用 `ILIKE` + `similarity()` 做轻量关键词召回

这样做的好处是：

1. 不需要额外引入 ElasticSearch / OpenSearch
2. 对中文短词、接口名、字段名、缩写匹配更直接
3. 和当前最小版项目的演进路径更平滑

但也要诚实地说，它并不等价于完整的 BM25 搜索引擎。

如果你后面要做真正大规模、复杂查询语义、精细分词和高阶检索，就该进入下一层基础设施升级。

## 三、这一阶段的整体架构

你可以把第二阶段理解成：在第一阶段原有两条管道上，分别补“维护能力”和“召回增强能力”。

### 1、离线同步管道

```text
扫描知识目录
    ->
读取文件内容
    ->
计算 file_hash
    ->
判断是否为新增 / 未变化 / 已变化
    ->
已变化则先删除旧分块
    ->
重新分块
    ->
Embedding
    ->
写入 vector_store
    ->
更新 kb_document 台账
```

### 2、在线问答管道

```text
用户问题
    ->
查询改写
    ->
多条检索查询
    ->
向量召回
    +
关键词召回
    ->
融合排序
    ->
取 TopN 片段
    ->
拼 Prompt
    ->
大模型回答
```

### 3、评估管道

```text
准备标准问题集
    ->
执行检索
    ->
判断是否命中目标来源文档
    ->
统计 Hit@K
    ->
统计 MRR
    ->
输出每题结果和总体指标
```

## 五、修改 `init/postgres/init.sql`

这一阶段最重要的数据库升级有 3 个：

1. 开启 `pg_trgm` 扩展
2. 给 `kb_document.file_path` 建唯一索引，支持按路径增量同步
3. 给 `vector_store.content` 建 `pg_trgm` 索引，支撑关键词检索

请把上一份文档中的 `init.sql` 升级为下面这个版本：

```sql
-- 启用 PGVector 扩展，支持向量类型。
CREATE EXTENSION IF NOT EXISTS vector;

-- 启用 uuid 相关函数，便于后续扩展。
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 启用 pg_trgm 扩展。
-- 第二阶段关键词检索会基于它来做 ILIKE 和 similarity 检索加速。
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 文档台账表。
CREATE TABLE IF NOT EXISTS kb_document (
    id BIGSERIAL PRIMARY KEY,
    file_name VARCHAR(255) NOT NULL,
    file_path VARCHAR(500) NOT NULL,
    file_hash VARCHAR(64) NOT NULL,
    category VARCHAR(100) NOT NULL,
    status VARCHAR(32) NOT NULL,
    remark VARCHAR(500),
    last_sync_at TIMESTAMP NOT NULL DEFAULT NOW(),
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);

-- 基于文件路径做唯一约束。
-- 这样我们后面就可以天然支持“相同路径文件的增量更新”。
CREATE UNIQUE INDEX IF NOT EXISTS uk_kb_document_file_path
    ON kb_document(file_path);

CREATE INDEX IF NOT EXISTS idx_kb_document_category
    ON kb_document(category);

CREATE INDEX IF NOT EXISTS idx_kb_document_status
    ON kb_document(status);

-- 向量存储表。
-- 这一版仍然沿用最小版的数据结构，保持升级路径平滑。
CREATE TABLE IF NOT EXISTS vector_store (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    content TEXT NOT NULL,
    metadata JSONB,
    embedding VECTOR(1536)
);

-- HNSW 向量索引，用于向量相似度召回。
CREATE INDEX IF NOT EXISTS idx_vector_store_embedding_hnsw
    ON vector_store
    USING hnsw (embedding vector_cosine_ops);

-- 关键词检索索引。
-- pg_trgm 的 GIN 索引可以加速 LIKE / ILIKE / similarity 等文本检索。
CREATE INDEX IF NOT EXISTS idx_vector_store_content_trgm
    ON vector_store
    USING gin (content gin_trgm_ops);

-- JSONB 元数据索引，方便按 category/sourcePath 等字段过滤。
CREATE INDEX IF NOT EXISTS idx_vector_store_metadata_gin
    ON vector_store
    USING gin (metadata);
```

这段代码的作用：

- 让第二阶段项目具备“目录增量同步”的数据库基础
- 让 `vector_store.content` 开始支持轻量关键词召回
- 保持和第一阶段表结构兼容，不强迫你重做整个库

## 六、创建第二阶段的测试知识文档

为了验证“目录同步、混合检索、查询改写、召回评估”这几项能力，我们多补两篇测试文档。

### 6.1 创建 `knowledge-base/tutorial/03-hybrid-retrieval.md`

```md
# 混合检索为什么比纯向量检索更稳

在真实 RAG 项目中，纯向量检索并不是永远可靠的。

当用户问题里出现这些内容时，纯向量召回经常会波动：

- 接口名
- 数据库字段名
- 英文缩写
- 错别字
- 类名、方法名、配置项名称

所以工程里常见的做法，是把检索拆成两路：

1. 向量召回：负责找语义相关
2. 关键词召回：负责找术语精确命中

然后再把两路结果做融合排序。

这种方案的价值在于：

- 既保留语义召回能力
- 又提升专有名词的命中率
- 对用户措辞变化更稳
```

这段代码的作用：

- 提供“混合检索”主题知识，方便后面做关键词命中测试
- 这篇文档里包含了“接口名、字段名、缩写”等纯向量检索容易波动的表达

### 6.2 创建 `knowledge-base/tutorial/04-query-rewrite.md`

```md
# 查询改写为什么能提升召回率

用户提出的问题，未必天然适合检索。

例如用户可能会这样问：

- 这个东西为什么老是搜不出来
- 我明明文档里写了，为什么还是答不准
- 这种问法是不是太口语化了

对检索系统来说，更有效的表达通常会更明确：

- RAG 检索不到内容的常见原因
- 检索召回率低的排查方法
- 口语化问题如何改写为检索友好表达

所以查询改写的本质，不是替用户改变问题意图，
而是把用户原问题翻译成更适合知识库召回的检索表达。

在工程实践里，查询改写通常有两种做法：

1. 单查询改写
2. 多查询改写

第二阶段我们先做一个轻量版本：

- 保留原始问题
- 再生成 2 到 3 条更适合检索的改写问题
```

这段代码的作用：

- 提供“查询改写”主题知识
- 同时为后面多路检索测试准备样本文本

## 九、修改 `src/main/java/com/qiange/ragdemo/config/RagProperties.java`

这一阶段最典型的变化，就是配置类会明显变大。

原因不是代码变乱了，而是因为我们真正开始把“检索工程的可调参数”显式化了。

```java
package com.qiange.ragdemo.config;

import lombok.Data;
import org.springframework.boot.context.properties.ConfigurationProperties;

/**
 * 第二阶段 RAG 配置绑定类。
 *
 * 和第一阶段相比，这里多出了三类配置：
 * 1. 知识目录同步配置
 * 2. 检索增强配置
 * 3. 召回评估配置
 *
 * 这也是教程里非常重要的一个认知：
 * RAG 不是“写死一套参数”就结束了，
 * 而是一个需要持续调参与观察的系统。
 */
@Data
@ConfigurationProperties(prefix = "rag")
public class RagProperties {

    private KnowledgeBase knowledgeBase = new KnowledgeBase();

    private Retrieval retrieval = new Retrieval();

    private Evaluation evaluation = new Evaluation();

    @Data
    public static class KnowledgeBase {

        private String defaultCategory = "tutorial";

        private Integer chunkSize = 500;

        private Integer chunkOverlap = 80;

        /**
         * 默认同步目录。
         */
        private String defaultDirectory;

        /**
         * 是否递归扫描子目录。
         */
        private Boolean recursiveSync = true;
    }

    @Data
```

### 12.2 创建 `src/main/java/com/qiange/ragdemo/repository/KeywordSearchRepository.java`

```java
package com.qiange.ragdemo.repository;

import com.fasterxml.jackson.core.type.TypeReference;
import com.fasterxml.jackson.databind.ObjectMapper;
import lombok.RequiredArgsConstructor;
import org.springframework.ai.document.Document;
import org.springframework.jdbc.core.namedparam.MapSqlParameterSource;
import org.springframework.jdbc.core.namedparam.NamedParameterJdbcTemplate;
import org.springframework.stereotype.Repository;
import org.springframework.util.StringUtils;

import java.util.HashMap;
import java.util.List;
import java.util.Map;

/**
 * 关键词检索仓库。
 *
 * 这一版采用的是一个非常务实的轻量方案：
 * - 通过 pg_trgm 为 content 建索引
 * - 通过 ILIKE + similarity 实现关键词召回
 *
 * 这样做特别适合当前这个“中文教程演示 + PostgreSQL 单库”场景。
 */
@Repository
@RequiredArgsConstructor
public class KeywordSearchRepository {

    private final NamedParameterJdbcTemplate namedParameterJdbcTemplate;

    private final ObjectMapper objectMapper;

    public List<Document> search(String query, String category, int topK, double similarityThreshold) {
        String sql = """
                select content,
                       metadata::text as metadata_json,
                       similarity(content, :query) as keyword_score
                from vector_store
                where (
                    content ilike '%' || :query || '%'
                    or similarity(content, :query) >= :similarityThreshold
                )
                  and (
                    :category is null
                    or metadata ->> 'category' = :category
                  )
                order by
                    case when content ilike '%' || :query || '%' then 1 else 0 end desc,
                    similarity(content, :query) desc
                limit :topK
                """;

        MapSqlParameterSource parameterSource = new MapSqlParameterSource()
                .addValue("query", query)
                .addValue("category", StringUtils.hasText(category) ? category : null)
                .addValue("similarityThreshold", similarityThreshold)
                .addValue("topK", topK);

        return namedParameterJdbcTemplate.query(sql, parameterSource, (rs, rowNum) -> {
            String content = rs.getString("content");
            String metadataJson = rs.getString("metadata_json");
            Double keywordScore = rs.getDouble("keyword_score");

            Map<String, Object> metadata = new HashMap<>();
            if (StringUtils.hasText(metadataJson)) {
                metadata.putAll(objectMapper.readValue(metadataJson, new TypeReference<>() {
                }));
            }

            metadata.put("keywordScore", keywordScore);
            return new Document(content, metadata);
        });
    }
}
```

这段代码的作用：

- 提供第二阶段关键词检索能力
- 让我们在不引入额外搜索中间件的情况下，先把“关键词通道”补起来

```java
/**
 * 知识目录同步服务实现。
 *
 * 这是第二阶段离线索引管道的核心编排类。
 */
@Service
@RequiredArgsConstructor
public class KnowledgeDirectorySyncServiceImpl implements KnowledgeDirectorySyncService {

    private final RagProperties ragProperties;

    private final KnowledgeDocumentService knowledgeDocumentService;

    private final LocalKnowledgeLoader localKnowledgeLoader;

    private final KnowledgeChunker knowledgeChunker;

    private final VectorStore vectorStore;

    private final VectorStoreCleanupRepository vectorStoreCleanupRepository;

    @Override
    public SyncKnowledgeDirectoryResponse syncDirectory(SyncKnowledgeDirectoryRequest request) {
        String directoryPath = StringUtils.hasText(request.getDirectoryPath())
                ? request.getDirectoryPath()
                : ragProperties.getKnowledgeBase().getDefaultDirectory();

        String category = StringUtils.hasText(request.getCategory())
                ? request.getCategory()
                : ragProperties.getKnowledgeBase().getDefaultCategory();

        boolean recursive = request.getRecursive() != null
                ? request.getRecursive()
                : Boolean.TRUE.equals(ragProperties.getKnowledgeBase().getRecursiveSync());

        Path directory = Path.of(directoryPath).toAbsolutePath();
        if (!Files.exists(directory) || !Files.isDirectory(directory)) {
            throw new IllegalArgumentException("知识目录不存在：" + directory);
        }

        List<Path> files = listKnowledgeFiles(directory, recursive);

        int importedFiles = 0;
        int updatedFiles = 0;
        int skippedFiles = 0;
        int failedFiles = 0;

        for (Path file : files) {
            try {
                SyncAction action = syncSingleFile(file, category);
                switch (action) {
                    case IMPORTED -> importedFiles++;
                    case UPDATED -> updatedFiles++;
                    case SKIPPED -> skippedFiles++;
                }
            } catch (Exception e) {
                failedFiles++;
            }
        }

        return SyncKnowledgeDirectoryResponse.builder()
                .scannedFiles(files.size())
                .importedFiles(importedFiles)
                .updatedFiles(updatedFiles)
                .skippedFiles(skippedFiles)
                .failedFiles(failedFiles)
                .build();
    }

    private SyncAction syncSingleFile(Path file, String category) {
        Document originalDocument = localKnowledgeLoader.load(file.toString(), category, null);
        String normalizedPath = file.toAbsolutePath().toString();
        String fileHash = DigestUtils.md5DigestAsHex(originalDocument.getText().getBytes(StandardCharsets.UTF_8));

        KnowledgeDocumentEntity existing = knowledgeDocumentService.getByFilePath(normalizedPath);

        if (existing == null) {
            KnowledgeDocumentEntity created = knowledgeDocumentService.createPendingDocument(
                    normalizedPath,
                    fileHash,
                    category
            );

            try {
                List<Document> chunks = knowledgeChunker.split(originalDocument);
                vectorStore.add(chunks);
                knowledgeDocumentService.markImported(created.getId(), "首次同步成功，片段数量：" + chunks.size());
                return SyncAction.IMPORTED;
            } catch (Exception e) {
                knowledgeDocumentService.markImportFailed(created.getId(), e.getMessage());
                throw e;
            }
        }

        if (fileHash.equals(existing.getFileHash())) {
            return SyncAction.SKIPPED;
        }

        knowledgeDocumentService.markBeforeResync(
                existing.getId(),
                fileHash,
                "检测到文件内容变更，准备重新同步"
        );

        try {
            vectorStoreCleanupRepository.deleteBySourcePath(normalizedPath);
            List<Document> chunks = knowledgeChunker.split(originalDocument);
            vectorStore.add(chunks);
            knowledgeDocumentService.markImported(existing.getId(), "增量同步成功，片段数量：" + chunks.size());
            return SyncAction.UPDATED;
        } catch (Exception e) {
            knowledgeDocumentService.markImportFailed(existing.getId(), e.getMessage());
            throw e;
        }
    }

    private List<Path> listKnowledgeFiles(Path directory, boolean recursive) {
        try (Stream<Path> stream = recursive ? Files.walk(directory) : Files.list(directory)) {
            return stream.filter(Files::isRegularFile)
                    .filter(this::isKnowledgeFile)
                    .sorted()
                    .toList();
        } catch (IOException e) {
            throw new RuntimeException("扫描知识目录失败：" + directory, e);
        }
    }

    private boolean isKnowledgeFile(Path path) {
        String fileName = path.getFileName().toString().toLowerCase();
        return fileName.endsWith(".md") || fileName.endsWith(".txt");
    }

    private enum SyncAction {
        IMPORTED,
        UPDATED,
        SKIPPED
    }
}
```

这段代码的作用：

- 把第二阶段离线索引管道真正升级成“目录级同步 + 增量更新”
- 保证同一路径文件变化后，会先清理旧片段，再写入新片段

```java
 * 2. 额外生成少量检索友好问题

@Service

public class QueryRewriteServiceImpl implements QueryRewriteService {

    private final RagProperties ragProperties;

    @Override
    public List<String> rewriteQueries(String question) {

            return List.of(question);
        }

        String prompt = """
                你是一个 RAG 检索改写助手。
                请基于用户问题，生成 %s 条更适合知识检索的查询表达。

                约束：
                1. 不要改变原问题意图
                2. 尽量补全术语，而不是发散主题
                3. 每行返回一条，不要编号
                4. 不要输出解释

                用户问题：
                %s
                """.formatted(ragProperties.getRetrieval().getRewriteQueryCount(), question);

        LinkedHashSet<String> querySet = new LinkedHashSet<>();
        querySet.add(question);

        try {
            String response = chatClient.prompt()
                    .user(prompt)
                    .call()
                    .content();

            if (StringUtils.hasText(response)) {
                for (String line : response.split("\\R")) {
                    String trimmed = line.trim();
                    if (StringUtils.hasText(trimmed)) {
                        querySet.add(trimmed);
                    }
                }
            }
        } catch (Exception e) {
            // 查询改写失败时，不让主链路崩掉，直接回退到原始问题。
            return List.of(question);
        }

        return new ArrayList<>(querySet);
    }
}
```

```java
package com.qiange.ragdemo.service.impl;

import com.qiange.ragdemo.common.RagConstants;
import com.qiange.ragdemo.config.RagProperties;
import com.qiange.ragdemo.repository.KeywordSearchRepository;
import com.qiange.ragdemo.service.HybridRetrievalService;
import com.qiange.ragdemo.service.model.RetrievedChunk;
import lombok.RequiredArgsConstructor;
import org.springframework.ai.document.Document;
import org.springframework.ai.vectorstore.SearchRequest;
import org.springframework.ai.vectorstore.VectorStore;
import org.springframework.stereotype.Service;
import org.springframework.util.DigestUtils;
import org.springframework.util.StringUtils;

import java.nio.charset.StandardCharsets;
import java.util.ArrayList;
import java.util.Comparator;
import java.util.LinkedHashMap;
import java.util.List;
import java.util.Map;

/**
 * 混合检索服务实现。
 *
 * 第二阶段的核心策略如下：
 * 1. 对每条查询分别执行向量召回
 * 2. 对每条查询分别执行关键词召回
 * 3. 把多路结果统一做 RRF 融合
 * 4. 返回最终 TopN 片段
 */
@Service
@RequiredArgsConstructor
public class HybridRetrievalServiceImpl implements HybridRetrievalService {

    private final VectorStore vectorStore;

    private final KeywordSearchRepository keywordSearchRepository;

    private final RagProperties ragProperties;

    @Override
    public List<RetrievedChunk> retrieve(List<String> queries, String category) {
        Map<String, RetrievedChunk> mergedChunkMap = new LinkedHashMap<>();

        for (String query : queries) {
            List<Document> vectorDocuments = vectorSearch(query, category);
            mergeByRrf(mergedChunkMap, vectorDocuments, RagConstants.RETRIEVAL_SOURCE_VECTOR);

            List<Document> keywordDocuments = keywordSearchRepository.search(
                    query,
                    category,
                    ragProperties.getRetrieval().getKeywordTopK(),
                    ragProperties.getRetrieval().getTrgmSimilarityThreshold()
            );
            mergeByRrf(mergedChunkMap, keywordDocuments, RagConstants.RETRIEVAL_SOURCE_KEYWORD);
        }

        return mergedChunkMap.values().stream()
                .sorted(Comparator.comparing(RetrievedChunk::getFusionScore).reversed())
                .limit(ragProperties.getRetrieval().getFusionTopK())
                .toList();
    }

    private List<Document> vectorSearch(String query, String category) {
        SearchRequest.Builder builder = SearchRequest.builder()
                .query(query)
                .topK(ragProperties.getRetrieval().getVectorTopK())
                .similarityThreshold(ragProperties.getRetrieval().getSimilarityThreshold());

        if (StringUtils.hasText(category)) {
            builder.filterExpression("category == '" + escapeFilterValue(category) + "'");
        }

        List<Document> documents = vectorStore.similaritySearch(builder.build());
        return documents == null ? new ArrayList<>() : documents;
    }

    private void mergeByRrf(Map<String, RetrievedChunk> mergedChunkMap,
                            List<Document> documents,
                            String source) {
        for (int i = 0; i < documents.size(); i++) {
            Document document = documents.get(i);
            String uniqueKey = buildUniqueKey(document);

            // Reciprocal Rank Fusion 核心公式：
            // score += 1 / (k + rank)
            double deltaScore = 1.0 / (ragProperties.getRetrieval().getRrfK() + i + 1);

            RetrievedChunk existingChunk = mergedChunkMap.get(uniqueKey);
            if (existingChunk == null) {
                mergedChunkMap.put(uniqueKey, RetrievedChunk.builder()
                        .uniqueKey(uniqueKey)
                        .document(document)
                        .retrievalSource(source)
                        .fusionScore(deltaScore)
                        .build());
                continue;
            }

            existingChunk.setFusionScore(existingChunk.getFusionScore() + deltaScore);
            existingChunk.setRetrievalSource(mergeRetrievalSource(
                    existingChunk.getRetrievalSource(),
                    source
            ));
        }
    }

    private String buildUniqueKey(Document document) {
        Object sourcePath = document.getMetadata().get(RagConstants.METADATA_SOURCE_PATH);
        Object chunkIndex = document.getMetadata().get(RagConstants.METADATA_CHUNK_INDEX);

        if (sourcePath != null && chunkIndex != null) {
            return sourcePath + "#" + chunkIndex;
        }

        // 如果元数据不完整，就退化为对内容做摘要键。
        return DigestUtils.md5DigestAsHex(document.getText().getBytes(StandardCharsets.UTF_8));
    }

    private String mergeRetrievalSource(String current, String incoming) {
        if (current == null || current.equals(incoming)) {
            return incoming;
        }
        return RagConstants.RETRIEVAL_SOURCE_HYBRID;
    }

    private String escapeFilterValue(String value) {
        return value.replace("'", "\\'");
    }
}
```

这段代码的作用：

- 实现第二阶段最核心的“向量召回 + 关键词召回 + 融合排序”
- 用 `RRF` 把来自不同查询、不同通道的结果整合成统一排序结果

```java
                    .question(evalCase.getQuestion())
                    .expectedSourceFileName(evalCase.getExpectedSourceFileName())
                    .hit(hit)
                    .rank(rank)
                    .build());
        }

        int totalCases = evalCases.size();
        double hitRate = totalCases == 0 ? 0D : (double) hitCount / totalCases;
        double mrr = totalCases == 0 ? 0D : reciprocalRankSum / totalCases;

        return RetrievalEvaluationResponse.builder()
                .totalCases(totalCases)
                .hitRate(hitRate)
                .mrr(mrr)
                .caseResults(caseResults)
                .build();
    }

    private List<EvalCase> loadCases() {
        String caseFile = ragProperties.getEvaluation().getCaseFile();
        if (!StringUtils.hasText(caseFile)) {
            throw new IllegalArgumentException("未配置 rag.evaluation.case-file");
        }

        Path path = Path.of(caseFile);
        if (!Files.exists(path)) {
            throw new IllegalArgumentException("评估样本文件不存在：" + path.toAbsolutePath());
        }

        try {
            String json = Files.readString(path);
            return objectMapper.readValue(json, new TypeReference<>() {
            });
        } catch (IOException e) {
            throw new RuntimeException("读取评估样本失败：" + caseFile, e);
        }
    }

    private Integer findFirstRank(List<RetrievedChunk> chunks, String expectedSourceFileName) {
        for (int i = 0; i < chunks.size(); i++) {
            Object sourceFileName = chunks.get(i).getDocument()
                    .getMetadata()
                    .get(RagConstants.METADATA_SOURCE_FILE_NAME);

            if (expectedSourceFileName.equals(String.valueOf(sourceFileName))) {
                return i + 1;
            }
        }
        return null;
    }

    @Data
    private static class EvalCase {

        private String question;

        private String category;

        private String expectedSourceFileName;

        private String description;
    }
}
```

这段代码的作用：

- 正式把第二阶段“召回评估”落成代码
- 让你后面优化混合检索和查询改写时，第一次拥有基础指标反馈

```java
import com.qiange.ragdemo.common.RagConstants;
import com.qiange.ragdemo.common.Result;
import com.qiange.ragdemo.dto.ReferenceChunkResponse;
import com.qiange.ragdemo.dto.RetrievalDebugRequest;
import com.qiange.ragdemo.dto.RetrievalDebugResponse;
import com.qiange.ragdemo.service.HybridRetrievalService;
import com.qiange.ragdemo.service.QueryRewriteService;
import com.qiange.ragdemo.service.model.RetrievedChunk;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.ai.document.Document;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

import java.util.List;
import java.util.Map;

/**
 * 检索调试接口。
 *
 * 这个接口的价值非常大：
 * 它能让你在不调用模型生成答案的前提下，
 * 先单独检查“检索这一步到底做得怎么样”。
 */
@RestController
@RequestMapping("/api/retrieval")
@RequiredArgsConstructor
public class RetrievalDebugController {

    private final QueryRewriteService queryRewriteService;

    private final HybridRetrievalService hybridRetrievalService;

    @PostMapping("/debug-search")
    public Result<RetrievalDebugResponse> debugSearch(@Valid @RequestBody RetrievalDebugRequest request) {
        List<String> queries = queryRewriteService.rewriteQueries(request.getQuestion());
        List<RetrievedChunk> chunks = hybridRetrievalService.retrieve(queries, request.getCategory());

        RetrievalDebugResponse response = RetrievalDebugResponse.builder()
                .queries(queries)
                .chunks(chunks.stream().map(this::toReferenceChunkResponse).toList())
                .build();

        return Result.ok(response);
    }

    private ReferenceChunkResponse toReferenceChunkResponse(RetrievedChunk chunk) {
        Document document = chunk.getDocument();
        Map<String, Object> metadata = document.getMetadata();

        return ReferenceChunkResponse.builder()
                .sourceFileName(stringValue(metadata.get(RagConstants.METADATA_SOURCE_FILE_NAME)))
                .sourcePath(stringValue(metadata.get(RagConstants.METADATA_SOURCE_PATH)))
                .chunkIndex(intValue(metadata.get(RagConstants.METADATA_CHUNK_INDEX)))
                .content(document.getText())
                .retrievalSource(chunk.getRetrievalSource())
                .fusionScore(chunk.getFusionScore())
                .build();
    }

    private String stringValue(Object value) {
        return value == null ? "" : String.valueOf(value);
    }

    private Integer intValue(Object value) {
        if (value == null) {
            return null;
        }
        return Integer.parseInt(String.valueOf(value));
    }
}
```

这段代码的作用：

- 提供“只看检索、不看答案”的调试入口
- 是第二阶段最推荐优先使用的接口之一

## 十六、第二阶段应该按什么顺序启动

这一节请你务必认真看。

因为第二阶段最容易出错的，不是 Java 代码本身，而是**数据库初始化方式**。

### 第 1 步：如果你已经跑过第一阶段，优先重建 PostgreSQL 数据卷

原因是：

- `init.sql` 只会在数据库首次初始化时执行
- 你现在已经给 `init.sql` 新增了 `pg_trgm` 扩展和索引
- 如果你只是改文件但不重建库，这些变更并不会自动生效

教程阶段最省心的做法是：

```powershell
docker compose down -v
docker compose up -d
```

这会清掉之前的 PostgreSQL 数据卷，然后按第二阶段新脚本重新初始化。

如果你不想清库，也可以手动执行第二阶段新增的 SQL。

### 第 2 步：重新设置 OpenAI 相关环境变量

```powershell
$env:OPENAI_API_KEY="替换成你自己的 OpenAI Key"
```

如果是兼容 OpenAI 协议的服务，再额外配置：

```powershell
$env:OPENAI_BASE_URL="你的兼容接口地址"
```

### 第 3 步：启动 Spring Boot 应用

```powershell
mvn spring-boot:run
```

### 第 4 步：优先先做“目录同步”

不要一上来就直接问答。

第二阶段推荐顺序是：

1. 先同步目录
2. 再看检索调试
3. 再做问答
4. 最后跑评估

因为第二阶段我们真正优化的是“检索”，不是“回答文案”。

## 十九、第二阶段常见报错排查

### 1、报错：`function similarity(text, unknown) does not exist`

原因：

- `pg_trgm` 扩展没有安装成功
- 或者数据库不是按第二阶段 `init.sql` 初始化的

排查方式：

1. 确认是否执行过 `docker compose down -v`
2. 确认数据库里已执行 `create extension if not exists pg_trgm;`

### 2、问题：改了 `init.sql`，但关键词检索还是不生效

这是第二阶段最常见的坑。

原因是：

- `init.sql` 不是热更新脚本
- 你之前的数据库卷还在，容器重启并不会重新执行初始化 SQL

解决方式：

1. 教学阶段最直接的方法：`docker compose down -v` 后重新 `up -d`
2. 或者手动登录数据库执行新增 SQL

### 3、问题：目录同步一直显示 `skippedFiles`

原因通常有两个：

1. 文件内容真的没变
2. 你改的是文件格式、空格或换行，但最终读入内容没有产生有效差异

第二阶段是用 `file_hash` 判断是否需要重新同步，所以只要文本内容摘要没变，就会被判定为跳过。

### 4、问题：更新文件后，问答里仍然出现旧内容

重点检查两件事：

1. 是否真的执行了 `VectorStoreCleanupRepository.deleteBySourcePath`
2. `sourcePath` 元数据在新老文档里是否一致

如果 `sourcePath` 不一致，旧片段就删不掉。

### 5、问题：查询改写后反而检索更差

这是很正常的。

第二阶段的查询改写只是“轻量版”。

你可以优先从这几个方向调：

1. 把改写条数从 `2` 改成 `1`
2. 优化 `QueryRewriteServiceImpl` 的提示词，让它少发散
3. 临时关闭 `query-rewrite-enabled`，对比前后效果

### 6、问题：评估接口提示样本文件不存在

检查：

1. `rag.evaluation.case-file` 路径是否正确
2. 你启动应用时的当前工作目录是否是项目根目录

第二阶段这里用的是相对路径文件，所以启动目录不对，也会导致读不到。

## 二十、第三阶段应该做什么

如果你已经把第二阶段完整跑通，下一步就该进入教程的最后两篇：

1. `107`：Reranker 精排
2. `108`：上下文压缩与裁剪

第三阶段推荐的推进顺序是：

### 升级 1：加入 Reranker

目标：

- 解决“召回到了，但排位不对”的问题

意义：

- 第二阶段的混合检索已经能把相关片段拉回来
- 第三阶段先做 Reranker，才能让最相关的片段排在更前面

### 升级 2：加入上下文去重和压缩

目标：

- 减少重复片段
- 降低 Prompt 噪声
- 控制 Token 成本

意义：

- 第二阶段经常会因为多路召回把相近片段都带回来
- 如果不做压缩，模型上下文质量反而会下降

### 升级 3：把评估从“召回评估”扩展到“答案评估”

目标：

- 不只看命没命中
- 还要看最终答案是不是更准确、更稳定

这一步，就是从“检索工程”继续走向“完整 RAG 系统工程”。

## 二十一、你在第二阶段真正学到了什么

如果你把这份文档完整敲完并跑通，你真正学到的是下面这些很重要的工程事实：

1. 文档导入不是“一次性动作”，而是一个需要持续同步的流程
2. 纯向量检索不是万能的，关键词通道在很多业务词场景里非常重要
3. 查询改写并不是为了让模型“更聪明”，而是为了让检索“更对路”
4. 不做评估，所有检索优化都很容易停留在“感觉更好了”
5. RAG 的质量提升，很多时候不是靠更强模型，而是靠更稳的检索链路

这也是为什么教程后半段会把重点放在：

1. 召回评估
2. 双路召回
3. 查询改写
4. 精排
5. 上下文压缩

因为这些东西，才真正决定一个 RAG 系统在真实环境里好不好用。
