# 后端多线程系统学习实战文档

**目标：** 从零创建一个 Spring Boot 多线程示例项目，系统学习后端开发里最常见的多线程场景，包括线程池、`CompletableFuture` 并行聚合、`@Async` 异步通知、`CountDownLatch` 批量任务、`ReentrantLock` 共享资源保护、`Semaphore` 并发限流、`@Scheduled` 定时任务。

**架构说明：** 这份文档不引入 MySQL、Redis、MQ 等额外中间件，而是把重点全部集中在“多线程本身”。项目用一个普通 Spring Boot Web 应用来模拟真实后端开发场景，通过接口请求来触发线程池任务、异步通知、批量导出、库存扣减、限流保护、缓存预热，让你在一个项目里把后端最常见的并发写法串起来学。

**技术栈：** Java 17、Spring Boot 3.2.5、Spring Web、Spring Validation、Lombok、`ThreadPoolTaskExecutor`、`CompletableFuture`、`CountDownLatch`、`ReentrantLock`、`Semaphore`、`@Async`、`@Scheduled`

---

## 一、为什么后端开发必须学多线程

后端开发里，多线程不是“高阶技巧”，而是几乎天天都会遇到的东西。

最常见的场景就是：

1. 一个接口要同时查用户、价格、库存、优惠券，想并行提速
2. 主流程成功后，要异步发短信、发邮件、发站内信
3. 批量导出报表时，要把大任务拆成多个子任务并发执行
4. 同一个 JVM 里有共享变量，必须加锁防止并发覆盖
5. 某个第三方接口很脆弱，要限制同一时刻并发数
6. 凌晨要定时预热缓存、清理数据、生成日报

所以你学多线程，核心不是记 API，而是要知道：

- 什么时候要并发
- 什么时候要异步
- 什么时候要加锁
- 什么时候要限流
- 什么时候不能乱开线程

## 二、这份文档会教你哪些真实场景

这次我直接按真实后端开发里常用的 6 个场景来写：

### 场景 1：接口聚合查询提速

你会学到：

- `CompletableFuture`
- 自定义业务线程池
- 多个独立远程调用并行执行

典型业务：

- 查询订单详情页时，同时查用户信息、价格信息、库存信息

### 场景 2：主流程成功后异步通知

你会学到：

- `@Async`
- 为什么异步任务不要阻塞主线程

典型业务：

- 下单成功后异步发送短信和站内信

### 场景 3：批量导出报表

你会学到：

- `CountDownLatch`
- 子任务拆分
- 主线程等待所有任务完成后汇总结果

典型业务：

- 导出大量用户数据、订单数据、日报数据

### 场景 4：共享资源保护

你会学到：

- `ReentrantLock`
- 为什么同一个 JVM 里的共享变量不能裸写

典型业务：

- 单机库存扣减、单机计数器更新

### 场景 5：限流保护

你会学到：

- `Semaphore`
- 为什么有些任务不适合无限并发

典型业务：

- 限制同时调用第三方支付或第三方 OCR 的任务数量

### 场景 6：定时任务

你会学到：

- `@Scheduled`
- 为什么定时任务也要配合线程池思维看待

典型业务：

- 凌晨缓存预热、定时清理临时数据

## 八、创建线程池配置类 `config/AsyncThreadPoolConfig.java`

```java
package com.example.multithread.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.scheduling.concurrent.ThreadPoolTaskExecutor;

import java.util.concurrent.Executor;
import java.util.concurrent.ThreadPoolExecutor;

@Configuration
public class AsyncThreadPoolConfig {

    @Bean("businessExecutor")
    public Executor businessExecutor() {
        // 这个线程池用于普通业务异步任务，例如：
        // 接口聚合查询、异步通知、定时预热。
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        // 核心线程数：平时一直保留的线程数。
        executor.setCorePoolSize(4);
        // 最大线程数：高峰期最多能扩到多少线程。
        executor.setMaxPoolSize(8);
        // 队列容量：任务先进入队列，队列满了再扩线程。
        executor.setQueueCapacity(100);
        // 空闲线程存活时间，单位秒。
        executor.setKeepAliveSeconds(60);
        // 线程名前缀，方便你在日志里区分到底是谁在执行任务。
        executor.setThreadNamePrefix("biz-exec-");
        // 当队列满、线程也满时，让提交任务的调用线程自己执行。
        // 这种策略比直接丢任务更安全，适合学习阶段。
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }

    @Bean("exportExecutor")
    public Executor exportExecutor() {
        // 这个线程池专门给批量导出这种“拆分子任务”的场景使用。
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(6);
        executor.setMaxPoolSize(12);
        executor.setQueueCapacity(200);
        executor.setKeepAliveSeconds(60);
        executor.setThreadNamePrefix("export-exec-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}
```

这段代码的作用：

- 创建两个自定义线程池
- 这是后端多线程开发里非常重要的一步：不要到处手写 `new Thread()`
- 真实项目里，不同类型任务最好用不同线程池隔离

## 十一、创建并行聚合服务 `service/AggregateQueryService.java`

```java
package com.example.multithread.service;

import com.example.multithread.dto.OrderSummaryResponse;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.stereotype.Service;

import java.util.concurrent.CompletableFuture;
import java.util.concurrent.Executor;

@Service
public class AggregateQueryService {

    private final MockRemoteQueryService mockRemoteQueryService;

    // 这里注入我们自定义的业务线程池。
    private final Executor businessExecutor;

    public AggregateQueryService(MockRemoteQueryService mockRemoteQueryService,
                                 @Qualifier("businessExecutor") Executor businessExecutor) {
        this.mockRemoteQueryService = mockRemoteQueryService;
        this.businessExecutor = businessExecutor;
    }

    public OrderSummaryResponse queryOrderSummary(Long orderId) {
        long start = System.currentTimeMillis();

        // 任务 1：异步查询用户信息。
        CompletableFuture<String> userFuture = CompletableFuture.supplyAsync(
                () -> mockRemoteQueryService.queryUserName(orderId),
                businessExecutor
        );

        // 任务 2：异步查询价格信息。
        CompletableFuture<String> priceFuture = CompletableFuture.supplyAsync(
                () -> mockRemoteQueryService.queryPriceInfo(orderId),
                businessExecutor
        );

        // 任务 3：异步查询库存信息。
        CompletableFuture<String> stockFuture = CompletableFuture.supplyAsync(
                () -> mockRemoteQueryService.queryStockInfo(orderId),
                businessExecutor
        );

        // allOf 的作用是：等上面 3 个任务全部执行完成。
        CompletableFuture.allOf(userFuture, priceFuture, stockFuture).join();

        OrderSummaryResponse response = new OrderSummaryResponse();
        response.setOrderId(orderId);
        response.setUserName(userFuture.join());
        response.setPriceInfo(priceFuture.join());
        response.setStockInfo(stockFuture.join());
        response.setCostMillis(System.currentTimeMillis() - start);
        return response;
    }
}
```

这段代码的作用：

- 演示后端最常见的“接口聚合并行提速”
- 如果不用多线程，这里 3 个任务要串行执行
- 用 `CompletableFuture` 后，3 个任务可以并发跑，总耗时会明显下降

## 十二、创建异步通知服务 `service/AsyncNoticeService.java`

```java
package com.example.multithread.service;

import lombok.extern.slf4j.Slf4j;
import org.springframework.scheduling.annotation.Async;
import org.springframework.stereotype.Service;

@Slf4j
@Service
public class AsyncNoticeService {

    @Async("businessExecutor")
    public void sendOrderNotice(String orderNo, String phone) {
        // 这里的方法会在独立线程池里执行，不会阻塞主线程。
        log.info("线程={}，开始发送通知，orderNo={}", Thread.currentThread().getName(), orderNo);

        try {
            // 模拟调用短信服务、站内信服务。
            Thread.sleep(2000);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("异步通知线程被中断", e);
        }

        log.info("线程={}，通知发送完成，phone={}", Thread.currentThread().getName(), phone);
    }
}
```

这段代码的作用：

- 演示主流程成功后异步做“次要任务”
- 最典型的就是：下单成功后发短信、发邮件、写操作日志
- 这样主线程可以更快返回结果，不用等通知全部发完

## 十三、创建批量导出服务 `service/BatchExportService.java`

```java
package com.example.multithread.service;

import com.example.multithread.dto.BatchExportResponse;
import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.stereotype.Service;

import java.util.ArrayList;
import java.util.Collections;
import java.util.List;
import java.util.concurrent.CountDownLatch;
import java.util.concurrent.Executor;

@Slf4j
@Service
public class BatchExportService {

    private final Executor exportExecutor;

    public BatchExportService(@Qualifier("exportExecutor") Executor exportExecutor) {
        this.exportExecutor = exportExecutor;
    }

    public BatchExportResponse exportReport() {
        long start = System.currentTimeMillis();

        // 模拟一份 50 条数据的报表。
        List<Integer> dataList = new ArrayList<>();
        for (int i = 1; i <= 50; i++) {
            dataList.add(i);
        }

        // 每 10 条数据切成一片，总共 5 个分片。
        int chunkSize = 10;
        int taskCount = (int) Math.ceil(dataList.size() * 1.0 / chunkSize);

        // 线程安全的结果容器。
        List<String> resultList = Collections.synchronizedList(new ArrayList<>());
        CountDownLatch countDownLatch = new CountDownLatch(taskCount);

        for (int i = 0; i < taskCount; i++) {
            int startIndex = i * chunkSize;
            int endIndex = Math.min(startIndex + chunkSize, dataList.size());
            List<Integer> subList = dataList.subList(startIndex, endIndex);

            exportExecutor.execute(() -> {
                try {
                    // 模拟导出这个分片。
                    Thread.sleep(500);
                    String result = "线程=" + Thread.currentThread().getName()
                            + "，导出分片=" + subList;
                    resultList.add(result);
                    log.info(result);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    throw new RuntimeException("导出任务被中断", e);
                } finally {
                    // 不管成功还是失败，都要 countDown。
                    countDownLatch.countDown();
                }
            });
        }

        try {
            // 主线程在这里等待，直到所有分片导出完成。
            countDownLatch.await();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("主线程等待导出结果时被中断", e);
        }

        BatchExportResponse response = new BatchExportResponse();
        response.setTotalSize(dataList.size());
        response.setChunkResultList(resultList);
        response.setCostMillis(System.currentTimeMillis() - start);
        return response;
    }
}
```

这段代码的作用：

- 演示大任务拆小任务并发执行
- `CountDownLatch` 的核心价值是：子线程做完自己的事，主线程再统一收结果
- 这是报表导出、批处理、批量同步里非常常见的模型

## 十四、创建单机库存锁保护服务 `service/LocalStockService.java`

```java
package com.example.multithread.service;

import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

import java.util.concurrent.locks.ReentrantLock;

@Slf4j
@Service
public class LocalStockService {

    // 模拟单机库存。
    private int stock = 20;

    // 可重入锁，用来保护共享变量 stock。
    private final ReentrantLock lock = new ReentrantLock();

    public String deductStock(int count) {
        // 进入关键区前先加锁。
        lock.lock();
        try {
            if (stock < count) {
                return "库存不足，当前库存=" + stock;
            }

            // 这里故意 sleep 一下，让你更容易理解并发覆盖问题。
            Thread.sleep(300);

            stock = stock - count;
            return "扣减成功，剩余库存=" + stock + "，线程=" + Thread.currentThread().getName();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("库存扣减线程被中断", e);
        } finally {
            // finally 里释放锁，是最基本的加锁规范。
            lock.unlock();
        }
    }

    public int currentStock() {
        return stock;
    }
}
```

这段代码的作用：

- 演示单机场景下，为什么共享变量必须加锁
- 如果这里不加锁，多线程同时扣减库存时非常容易出现超扣
- `ReentrantLock` 是后端开发里非常常见的一种显式锁

## 十五、创建限流保护服务 `service/LimitedInvokeService.java`

```java
package com.example.multithread.service;

import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

import java.util.concurrent.Semaphore;

@Slf4j
@Service
public class LimitedInvokeService {

    // 这里只允许最多 3 个任务同时进入。
    private final Semaphore semaphore = new Semaphore(3);

    public String invokeHeavyTask() {
        boolean acquired = false;
        try {
            // 尝试拿许可。
            acquired = semaphore.tryAcquire();
            if (!acquired) {
                return "当前并发任务过多，请稍后再试";
            }

            log.info("线程={}，拿到许可，开始执行重量任务", Thread.currentThread().getName());
            Thread.sleep(2000);
            return "执行成功，线程=" + Thread.currentThread().getName();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("限流任务被中断", e);
        } finally {
            // 只有真正拿到许可的线程才归还许可。
            if (acquired) {
                semaphore.release();
            }
        }
    }
}
```

这段代码的作用：

- 演示如何限制同时执行某类任务的数量
- `Semaphore` 很适合保护第三方接口、OCR、报表生成这类昂贵操作
- 它不是锁，它更像“许可证明”

## 十六、创建定时预热服务 `service/CacheWarmService.java`

```java
package com.example.multithread.service;

import lombok.extern.slf4j.Slf4j;
import org.springframework.scheduling.annotation.Async;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Service;

@Slf4j
@Service
public class CacheWarmService {

    @Scheduled(cron = "0 0 3 * * ?")
    public void scheduledWarm() {
        // 真实业务里，这里可能是预热首页推荐、热门商品、配置缓存。
        log.info("线程={}，开始执行凌晨缓存预热任务", Thread.currentThread().getName());
        doWarm("scheduled");
    }

    @Async("businessExecutor")
    public void manualWarm() {
        // 手动触发时，也走线程池异步执行，避免接口卡顿。
        log.info("线程={}，收到手动缓存预热任务", Thread.currentThread().getName());
        doWarm("manual");
    }

    private void doWarm(String source) {
        try {
            Thread.sleep(1500);
            log.info("线程={}，缓存预热完成，来源={}", Thread.currentThread().getName(), source);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("缓存预热线程被中断", e);
        }
    }
}
```

这段代码的作用：

- 演示定时任务和异步线程池结合使用
- 很多后端定时任务本质上也是后台线程在跑
- 手动预热接口则演示了“接口触发 + 异步执行”的组合场景

## 十九、6 个场景怎么一步一步测试

### 场景 1：并行聚合查询

请求接口：

```bash
curl.exe "http://localhost:8080/thread/order-summary/1001"
```

你应该观察到：

- 日志里会出现多个不同的 `biz-exec-*` 线程名称
- 3 个模拟查询任务会并行执行
- 总耗时明显小于三段任务串行相加的总耗时

这个场景的重点：

- 学会 `CompletableFuture.supplyAsync()`
- 学会把多个独立任务并行化

### 场景 2：异步发送通知

请求接口：

```bash
curl.exe -X POST "http://localhost:8080/thread/async-notice" -H "Content-Type: application/json" -d "{\"orderNo\":\"ORD202603210001\",\"phone\":\"13800000000\"}"
```

你应该观察到：

- 接口会很快返回
- 但日志里还能继续看到后台线程在发送通知

这个场景的重点：

- 学会 `@Async`
- 学会“主流程和次要任务解耦”

### 场景 3：批量导出报表

请求接口：

```bash
curl.exe "http://localhost:8080/thread/export-report"
```

你应该观察到：

- 日志里会看到多个 `export-exec-*` 线程
- 不同分片被不同线程导出
- 主线程最终等待所有分片完成后再统一返回结果

这个场景的重点：

- 学会 `CountDownLatch`
- 学会“主线程等待多个子任务收口”的思路

### 场景 4：单机库存扣减

先看当前库存：

```bash
curl.exe "http://localhost:8080/thread/stock/current"
```

再扣减库存：

```bash
curl.exe -X POST "http://localhost:8080/thread/stock/deduct" -H "Content-Type: application/json" -d "{\"count\":1}"
```

你应该观察到：

- 每次扣减后库存都会减少
- 如果你并发调这个接口，加锁版本依然能保证不会超扣

这个场景的重点：

- 学会 `ReentrantLock`
- 学会共享变量为什么必须保护

### 场景 5：并发限流保护

请求接口：

```bash
curl.exe "http://localhost:8080/thread/limited-task"
```

你应该观察到：

- 一次调用时能正常返回
- 如果你短时间多次并发请求，其中一部分会得到“当前并发任务过多”

这个场景的重点：

- 学会 `Semaphore`
- 学会限制同一时刻允许多少任务进入

### 场景 6：手动触发缓存预热

请求接口：

```bash
curl.exe -X POST "http://localhost:8080/thread/cache-warm/manual"
```

你应该观察到：

- 接口会快速返回
- 后台线程继续完成预热任务

这个场景的重点：

- 学会 `@Scheduled` 和 `@Async` 在后台任务里的常见组合方式

## 二十、这份教程里每种技术到底解决什么问题

### 1. `ThreadPoolTaskExecutor`

解决问题：

- 统一管理线程
- 避免项目里到处 `new Thread()`
- 控制线程数量、队列容量、拒绝策略

### 2. `CompletableFuture`

解决问题：

- 适合多个“互不依赖”的查询任务并行执行
- 典型就是接口聚合提速

### 3. `@Async`

解决问题：

- 把次要任务从主线程中拆出去
- 典型就是异步通知、异步日志、异步回调

### 4. `CountDownLatch`

解决问题：

- 适合“主线程等多个子任务做完再汇总”
- 典型就是批量导出、批量同步

### 5. `ReentrantLock`

解决问题：

- 保护同一 JVM 内的共享资源
- 典型就是单机库存、单机计数器

### 6. `Semaphore`

解决问题：

- 控制同时允许多少个线程进入
- 典型就是第三方接口保护和重量任务限流

### 7. `@Scheduled`

解决问题：

- 固定时间自动执行后台任务
- 典型就是缓存预热、清理任务、日报生成

## 二十一、后端开发里写多线程最容易犯的错误

### 错误 1：到处手写 `new Thread()`

问题：

- 线程不可控
- 没法统一管理
- 很容易把系统线程资源耗光

正确做法：

- 优先使用线程池

### 错误 2：把所有任务都塞进同一个线程池

问题：

- 导出任务把线程池占满后，通知任务也会卡住

正确做法：

- 不同类型任务，按业务场景拆分线程池

### 错误 3：共享变量不加锁

问题：

- 多线程同时修改同一个变量，结果会错乱

正确做法：

- 用 `ReentrantLock`、`synchronized` 或无锁并发工具

### 错误 4：用了 `ThreadLocal` 但不清理

问题：

- 在线程池复用线程的情况下，可能把上一个请求的数据带到下一个请求

正确做法：

- 使用完一定 `remove()`

### 错误 5：以为多线程一定更快

问题：

- 并发有线程切换成本
- 任务之间如果本来就强依赖，硬并发反而更乱

正确做法：

- 只对真正独立、真正耗时的任务做并发

### 错误 6：在同一个类里自调用 `@Async` 方法

问题：

- 你在同一个类内部直接调用自己的 `@Async` 方法
- 这样不会经过 Spring 代理，异步不会生效

正确做法：

- 让 `@Async` 方法放在独立的 Service 里
- 通过 Spring 容器去调用它

## 二十二、你应该按什么顺序学习

我建议你按下面顺序学：

1. 先学线程池配置
2. 再学 `CompletableFuture`
3. 再学 `@Async`
4. 再学 `CountDownLatch`
5. 再学 `ReentrantLock`
6. 再学 `Semaphore`
7. 最后再学 `@Scheduled`

这样学的原因很简单：

- 线程池是基础
- 接口并行和异步通知是最常见业务
- 锁和限流是并发保护能力
- 定时任务是后台任务能力
