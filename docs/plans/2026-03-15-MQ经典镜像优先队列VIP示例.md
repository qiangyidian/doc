# MQ经典镜像优先队列VIP示例

**目标：** 从零创建一个独立的 Spring Boot 项目，使用 RabbitMQ 经典镜像队列和经典优先队列，实现 VIP 用户任务和普通用户任务的不同优先级处理。

**架构说明：** 这个项目使用 `Spring Boot + RabbitMQ 3.13.x + Docker Compose`。业务入口分成 VIP 和普通用户两个接口，但最终都进入同一个经典优先队列；VIP 消息携带更高优先级，普通消息携带更低优先级。RabbitMQ 使用 3 节点集群，并通过经典镜像策略把业务队列复制到集群节点。

**技术栈：** Java 17、Spring Boot 3.2.5、Spring Web、Spring AMQP、RabbitMQ 3.13 Management Image、Docker Compose、Maven、Lombok、Spring Boot Test

---

## 一、这个项目在生产里解决什么问题

很多业务里都会出现“不同用户等级需要不同处理速度”的场景，例如：

- VIP 用户下单后的任务要优先处理
- 付费用户的工单要优先处理
- 高等级会员的通知消息要优先发送

如果所有消息都无差别进入同一个普通队列，那么先来的普通用户消息会一直排在前面，VIP 用户虽然更重要，也只能按顺序等待。

这个项目就是专门练习下面这件事：

- 普通用户消息先进队列
- VIP 用户消息后进入队列
- 只要普通消息还在队列里等待，VIP 消息就能优先被取出消费

这就是 RabbitMQ 经典优先队列的核心价值。

## 二、这一份文档要特别说明的版本前提

这次你明确要求使用：

- `旧版镜像队列 + 经典优先队列`

所以这里必须把 RabbitMQ 版本固定为：

- `RabbitMQ 3.13.x`

原因很重要：

- RabbitMQ `4.0` 已经移除了 `classic mirrored queue`
- 你这次要练的是旧版经典镜像策略，所以不能使用 RabbitMQ 4.x

因此，这份文档只适用于：

- `rabbitmq:3.13-management`

不适用于：

- `rabbitmq:4.x`

      - "15672:15672"
    volumes:
      # 给节点挂数据卷，避免容器重启后数据全丢。
      - rabbit1-data:/var/lib/rabbitmq

  rabbit2:
    image: rabbitmq:3.13-management
    container_name: rabbit2
    hostname: rabbit2
    environment:
      RABBITMQ_ERLANG_COOKIE: rabbit_cluster_cookie_demo
      RABBITMQ_DEFAULT_USER: guest
      RABBITMQ_DEFAULT_PASS: guest
    ports:
      # 宿主机 5673 映射到容器内部 5672，表示第 2 个节点。
      - "5673:5672"
      - "15673:15672"
    volumes:
      - rabbit2-data:/var/lib/rabbitmq

  rabbit3:
    image: rabbitmq:3.13-management
    container_name: rabbit3
    hostname: rabbit3
    environment:
      RABBITMQ_ERLANG_COOKIE: rabbit_cluster_cookie_demo
      RABBITMQ_DEFAULT_USER: guest
      RABBITMQ_DEFAULT_PASS: guest
    ports:
      # 宿主机 5674 映射到容器内部 5672，表示第 3 个节点。
      - "5674:5672"
      - "15674:15672"

## 七、创建 `init-cluster.ps1`

```powershell
# 先等待容器里的 RabbitMQ 进程完全启动。
Start-Sleep -Seconds 15

function Wait-RabbitNode {
    param (
        [string]$ContainerName
    )

    while ($true) {
        docker exec $ContainerName rabbitmq-diagnostics -q ping 2>$null
        if ($LASTEXITCODE -eq 0) {
            Write-Host "$ContainerName 已经启动完成"
            break
        }

        Write-Host "等待 $ContainerName 启动..."
        Start-Sleep -Seconds 2
    }
}

Wait-RabbitNode -ContainerName rabbit1
Wait-RabbitNode -ContainerName rabbit2
Wait-RabbitNode -ContainerName rabbit3

# 让 rabbit2 加入 rabbit1 所在集群。
# stop_app：先停止 RabbitMQ 应用，但容器本身还在。
# reset：清空 rabbit2 自己原本的单机状态。
# join_cluster：把 rabbit2 挂到 rabbit1 的集群里。
# start_app：重新启动 RabbitMQ 应用。
docker exec rabbit2 rabbitmqctl stop_app
docker exec rabbit2 rabbitmqctl reset
docker exec rabbit2 rabbitmqctl join_cluster rabbit@rabbit1
docker exec rabbit2 rabbitmqctl start_app

# 让 rabbit3 加入 rabbit1 所在集群。
docker exec rabbit3 rabbitmqctl stop_app
docker exec rabbit3 rabbitmqctl reset
docker exec rabbit3 rabbitmqctl join_cluster rabbit@rabbit1
docker exec rabbit3 rabbitmqctl start_app

# 设置经典镜像策略。
# 这个策略的含义是：只要队列名是 user.priority.queue，就把它镜像到集群全部节点。
# ha-mode=all 表示镜像到所有节点。
# ha-sync-mode=automatic 表示镜像副本自动同步。
docker exec rabbit1 rabbitmqctl set_policy ha-priority-queue "^user\.priority\.queue$" "{\"ha-mode\":\"all\",\"ha-sync-mode\":\"automatic\"}" --apply-to queues

# 查看集群状态。
docker exec rabbit1 rabbitmqctl cluster_status

# 查看镜像策略是否生效。
docker exec rabbit1 rabbitmqctl list_policies
```

这段代码的作用：

- 等待 3 个节点都真正启动完成
- 让 `rabbit2`、`rabbit3` 加入 `rabbit1` 的集群
- 创建经典镜像策略 `ha-priority-queue`
- 这个策略会让业务队列复制到多个节点，满足你这次要求的“镜像集群”

## 十四、创建 RabbitMQ 配置类 `RabbitMqConfig.java`

```java
package com.example.mqvippriority.config;

import org.springframework.amqp.core.Binding;
import org.springframework.amqp.core.BindingBuilder;
import org.springframework.amqp.core.DirectExchange;
import org.springframework.amqp.core.Queue;
import org.springframework.amqp.core.QueueBuilder;
import org.springframework.amqp.support.converter.Jackson2JsonMessageConverter;
import org.springframework.amqp.support.converter.MessageConverter;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class RabbitMqConfig {

    // 统一把交换机、队列、路由键写成常量。
    // 这样后面如果名字改了，不需要全项目到处找字符串。
    public static final String PRIORITY_EXCHANGE = "user.priority.exchange";
    public static final String PRIORITY_QUEUE = "user.priority.queue";
    public static final String VIP_ROUTING_KEY = "user.vip";
    public static final String NORMAL_ROUTING_KEY = "user.normal";

    @Bean
    public DirectExchange priorityExchange() {
        // 使用 DirectExchange，是因为我们希望根据不同 routingKey
        // 把 VIP 和普通用户消息都投递到同一个目标队列。
        return new DirectExchange(PRIORITY_EXCHANGE, true, false);
    }

    @Bean
    public Queue priorityQueue() {
        Map<String, Object> arguments = new HashMap<>();
        // x-max-priority 表示这是一个经典优先队列，并且最大优先级是 10。
        // 注意：这个参数必须在队列声明时就带上，后面不能临时变出来。
        arguments.put("x-max-priority", 10);

        return QueueBuilder.durable(PRIORITY_QUEUE)
                .withArguments(arguments)
                .build();
    }

    @Bean
    public Binding vipBinding(Queue priorityQueue, DirectExchange priorityExchange) {
        // VIP routingKey 也绑定到同一个队列。
        return BindingBuilder.bind(priorityQueue)
                .to(priorityExchange)
                .with(VIP_ROUTING_KEY);
    }

    @Bean
    public Binding normalBinding(Queue priorityQueue, DirectExchange priorityExchange) {
        // 普通用户 routingKey 同样绑定到这个队列。
        return BindingBuilder.bind(priorityQueue)
                .to(priorityExchange)
                .with(NORMAL_ROUTING_KEY);
    }

    @Bean
    public MessageConverter messageConverter() {
        // 使用 JSON 转换器，避免默认的 Java 序列化格式不直观。
        // 这样你在管理后台里看到的消息内容也更容易理解。
        return new Jackson2JsonMessageConverter();
    }
}
```

这段代码的作用：

- 声明交换机、队列和两个绑定关系
- `x-max-priority=10` 是这次优先队列实验最核心的配置
- VIP 和普通用户都进入同一个物理队列，只是消息优先级不同

## 十五、创建生产者 `PriorityTaskProducer.java`

```java
package com.example.mqvippriority.mq;

import com.example.mqvippriority.config.RabbitMqConfig;
import org.springframework.amqp.rabbit.core.RabbitTemplate;
import org.springframework.stereotype.Component;

@Component
public class PriorityTaskProducer {

    // RabbitTemplate 是 Spring 封装好的 RabbitMQ 发送工具。
    private final RabbitTemplate rabbitTemplate;

    public PriorityTaskProducer(RabbitTemplate rabbitTemplate) {
        this.rabbitTemplate = rabbitTemplate;
    }

    public void send(String routingKey, PriorityTaskMessage message) {
        rabbitTemplate.convertAndSend(
                RabbitMqConfig.PRIORITY_EXCHANGE,
                routingKey,
                message,
                msg -> {
                    // 这里给消息设置优先级。
                    // RabbitMQ 会根据这个值决定同一个优先队列里的消息先后顺序。
                    // 数值越大，优先级越高。
                    msg.getMessageProperties().setPriority(message.getPriority());
                    return msg;
                }
        );
    }
}
```

这段代码的作用：

- 把消息发送到统一的交换机
- 通过 `setPriority` 把优先级真正写入 RabbitMQ 消息属性
- 没有这个设置，队列即使声明了优先级，也不会体现 VIP 和普通消息的差异

## 十六、创建业务服务 `PriorityTaskService.java`

```java
package com.example.mqvippriority.service;

import com.example.mqvippriority.config.RabbitMqConfig;
import com.example.mqvippriority.dto.SendTaskRequest;
import com.example.mqvippriority.dto.TaskResponse;
import com.example.mqvippriority.mq.PriorityTaskMessage;
import com.example.mqvippriority.mq.PriorityTaskProducer;
import org.springframework.stereotype.Service;

import java.time.LocalDateTime;
import java.util.ArrayList;
import java.util.List;

@Service
public class PriorityTaskService {

    // Service 层的作用不是直接和控制器混在一起写逻辑，
    // 而是把“组装消息 + 决定优先级 + 调用生产者”这部分逻辑收口在这里。
    private final PriorityTaskProducer priorityTaskProducer;

    public PriorityTaskService(PriorityTaskProducer priorityTaskProducer) {
        this.priorityTaskProducer = priorityTaskProducer;
    }

    public TaskResponse sendNormalTask(SendTaskRequest request) {
        return sendTask(request, 2, RabbitMqConfig.NORMAL_ROUTING_KEY, "普通用户任务发送成功");
    }

    public TaskResponse sendVipTask(SendTaskRequest request) {
        return sendTask(request, 8, RabbitMqConfig.VIP_ROUTING_KEY, "VIP 用户任务发送成功");
    }

    public List<TaskResponse> sendDemoTasks() {
        List<TaskResponse> result = new ArrayList<>();

        // 先连续发送 5 条普通用户任务。
        // 这样可以故意让队列里先堆积一批低优先级消息。
        for (int i = 1; i <= 5; i++) {
            SendTaskRequest request = new SendTaskRequest();
            request.setUserId(1000L + i);
            request.setUserType("NORMAL");
            request.setTaskName("普通任务-" + i);
            request.setContent("普通用户消息演示-" + i);
            result.add(sendNormalTask(request));
        }

        // 再发送 2 条 VIP 任务，便于观察后发先消费。
        // 这两条消息虽然是后发送的，但因为优先级更高，
        // 所以会优先从还在队列里等待的普通消息前面插队。
        for (int i = 1; i <= 2; i++) {
            SendTaskRequest request = new SendTaskRequest();
            request.setUserId(9000L + i);
            request.setUserType("VIP");
            request.setTaskName("VIP任务-" + i);
            request.setContent("VIP 用户消息演示-" + i);
            result.add(sendVipTask(request));
        }

        return result;
    }

    private TaskResponse sendTask(SendTaskRequest request, Integer priority, String routingKey, String remark) {
        PriorityTaskMessage message = new PriorityTaskMessage();
        // 下面这几行是在把接口收到的请求对象，转换成最终要发送的 MQ 消息对象。
        message.setUserId(request.getUserId());
        message.setUserType(request.getUserType());
        message.setTaskName(request.getTaskName());
        message.setContent(request.getContent());
        message.setPriority(priority);
        message.setCreateTime(LocalDateTime.now());

        priorityTaskProducer.send(routingKey, message);

        return new TaskResponse(
                request.getUserId(),
                request.getUserType(),
                request.getTaskName(),
                priority,
                routingKey,
                remark
        );
    }
}
```

这段代码的作用：

- 统一封装发送普通任务、发送 VIP 任务、批量演示任务的逻辑
- 普通任务固定优先级 `2`
- VIP 任务固定优先级 `8`
- `/tasks/demo` 接口会先发普通消息，再发 VIP 消息，用来做优先级实验

## 十七、创建消费者 `PriorityTaskConsumer.java`

```java
package com.example.mqvippriority.mq;

import com.example.mqvippriority.config.RabbitMqConfig;
import com.rabbitmq.client.Channel;
import org.springframework.amqp.core.Message;
import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.stereotype.Component;

import java.io.IOException;

@Component
public class PriorityTaskConsumer {

    @RabbitListener(queues = RabbitMqConfig.PRIORITY_QUEUE)
    public void consume(PriorityTaskMessage taskMessage, Message message, Channel channel) throws IOException {
        // deliveryTag 可以理解为 RabbitMQ 给当前这条消息分配的唯一投递编号。
        // 你后面调用 ack 或 nack 时，都要带上它。
        long deliveryTag = message.getMessageProperties().getDeliveryTag();

        try {
            System.out.println("收到消息 -> userType=" + taskMessage.getUserType()
                    + ", taskName=" + taskMessage.getTaskName()
                    + ", priority=" + taskMessage.getPriority()
                    + ", createTime=" + taskMessage.getCreateTime());

            System.out.println("开始处理消息 -> " + taskMessage.getTaskName());

            // 故意让每条消息处理慢一点。
            // 这样你在实验时更容易观察到 VIP 消息插队效果。
            Thread.sleep(3000);

            System.out.println("处理完成 -> " + taskMessage.getTaskName());

            // 手动确认消息处理成功。
            // false 表示只确认当前这一条，不批量确认之前的消息。
            channel.basicAck(deliveryTag, false);
        } catch (Exception ex) {
            System.out.println("处理失败 -> " + taskMessage.getTaskName() + ", error=" + ex.getMessage());
            // 这里用 basicNack 拒绝消息，并选择重新入队。
            // requeue=true 表示让 RabbitMQ 把消息重新放回队列。
            channel.basicNack(deliveryTag, false, true);
        }
    }
}
```

这段代码的作用：

- 消费者监听的是同一个优先队列 `user.priority.queue`
- `Thread.sleep(3000)` 是为了让实验效果更明显
- `basicAck` 和 `basicNack` 让你同时学到手动确认和异常处理

## 二十二、你应该看到什么效果

理想情况下，控制台日志会接近下面这种顺序：

```text
收到消息 -> userType=NORMAL, taskName=普通任务-1, priority=2
开始处理消息 -> 普通任务-1
处理完成 -> 普通任务-1

收到消息 -> userType=VIP, taskName=VIP任务-1, priority=8
开始处理消息 -> VIP任务-1
处理完成 -> VIP任务-1

收到消息 -> userType=VIP, taskName=VIP任务-2, priority=8
开始处理消息 -> VIP任务-2
处理完成 -> VIP任务-2

收到消息 -> userType=NORMAL, taskName=普通任务-2, priority=2
开始处理消息 -> 普通任务-2
处理完成 -> 普通任务-2
```

这段代码的作用：

- 它演示了“VIP 消息后进入队列，但先于仍在排队的普通消息被消费”
- 这就是优先队列生效的核心现象

### 你必须理解的一个原理

优先队列只能影响：

- 还在队列里等待投递的消息

不能影响：

- 已经被消费者取走、正在处理的消息

所以如果第一条普通消息已经被消费者拿走了，后来的 VIP 消息无法把它抢下来。这不是配置错误，而是优先队列机制本来就是这样。

## 二十四、常见报错排查

### 1. `join_cluster` 失败

常见原因：

- 3 个节点的 `RABBITMQ_ERLANG_COOKIE` 不一致
- 节点还没完全启动就开始执行集群脚本

解决方式：

- 确保 3 个节点 Cookie 完全相同
- 先等待节点启动完成再执行 `init-cluster.ps1`

### 2. 优先级看不出来

常见原因：

- 消费者 `prefetch` 太大
- 消费者处理太快
- 你观察的是已经被消费者取走的消息

解决方式：

- 确保 `prefetch = 1`
- 保留 `Thread.sleep(3000)`
- 先发普通消息，再发 VIP 消息

### 3. 队列没有优先级效果

常见原因：

- 队列没有配置 `x-max-priority`
- 发送消息时没有调用 `setPriority`

解决方式：

- 检查 `RabbitMqConfig.java`
- 检查 `PriorityTaskProducer.java`

### 4. 镜像策略没有生效

常见原因：

- 使用了 RabbitMQ 4.x
- policy 正则没有匹配到队列名

解决方式：

- 确保镜像版本是 `rabbitmq:3.13-management`
- 确保队列名是 `user.priority.queue`

## 二十五、你在这个项目中学到了什么

做完这个项目后，你应该真正理解下面几件事：

1. RabbitMQ 优先队列不是“两个队列谁先谁后”，而是“同一个队列里不同消息有不同优先级”
2. VIP 和普通用户可以用两个不同业务入口表达，但最终仍然进入同一个优先队列
3. 优先队列只能重排“还没被消费的消息”
4. 旧版经典镜像队列必须固定在 RabbitMQ 3.x，不能升级到 4.x
5. `prefetch=1` 和“慢消费”是观察优先级效果的关键实验条件
