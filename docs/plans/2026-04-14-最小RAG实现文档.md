# Spring Boot + Spring AI + PGVector 最小可用 RAG 实现文档

**目标：** 从零创建一个 `Spring Boot` 的最小可用 RAG 项目，完整演示“文档加载 -> 文本清洗 -> 分块 -> Embedding -> PGVector 入库 -> 相似度检索 -> Prompt 拼装 -> 大模型回答”这条最基础的闭环链路。

**架构说明：** 这份文档严格按你前面那套 RAG 教程的学习顺序来落地，但只实现 `Naive RAG` 的最小闭环，不直接上混合检索、Reranker、查询改写、上下文压缩和评估平台。整体采用 `Spring Boot + Spring AI + PostgreSQL + PGVector + MyBatis`，其中 `kb_document` 表负责记录文档台账，`vector_store` 表负责保存文档分块向量。

**技术栈：** Java 17、Spring Boot 3.5.x、Spring AI 1.1.x、Spring Web、MyBatis、PostgreSQL 16、PGVector、Docker Compose

---

## 一、这份文档解决什么问题

这不是一份“只把依赖列一下”的笔记，而是一份你后面可以直接照着创建项目的实现文档。

你最终会学到下面 6 个核心点：

1. 为什么最小可用 RAG 不应该一上来就堆很多高级组件
2. 如何在 Spring Boot 里把 `Chat Model`、`Embedding Model` 和 `PGVector` 串起来
3. 如何把本地 `md/txt` 文档读进来，并写入基础 `Metadata`
4. 如何做一个最基础、但方向正确的分块策略
5. 如何通过相似度搜索把相关片段取回，再拼成 Prompt 交给模型回答
6. 如何验证“知识库里有答案”和“知识库里没有答案”这两类最基本场景

## 二、先讲清楚这份文档的“教学化”处理

这份文档不是生产级大而全版本，而是为了让你先把最小闭环跑起来，所以我会明确做 4 个取舍：

1. **文档格式先只支持 `md` 和 `txt`**
   
   教程里讲了 PDF、Word、Excel、网页等多种格式，但最小闭环阶段先不处理扫描件 OCR、表格解析、复杂版式清洗。这样你能先把“读文档 -> 建向量 -> 检索回答”这条主链路跑通。

2. **先只做单知识库，不做多租户**
   
   教程里后面讲了 `tenant_id`、`doc_id`、状态管理、多租户隔离。最小闭环阶段只保留最关键的 `doc_id`、`file_name`、`category` 等字段，先把 Metadata 设计意识建立起来。

3. **先只做纯向量检索，不做混合检索**
   
   教程里第 103~108 篇讲了相似度局限、召回评估、双路召回、查询改写、Reranker、压缩。最小版先只做 `Vector Search`，因为这一步是后面所有高级能力的基础。

4. **先手动拼 Prompt，不直接用高级 Advisor 封装**
   
   Spring AI 里可以直接用更高级的封装，但最小学习版我会手动写“检索 -> 拼上下文 -> 调 ChatClient”的代码。这样你不会只会调框架，不知道 RAG 查询链路里每一步在干什么。

## 五、创建 `init/postgres/init.sql`

```sql
-- 先开启 pgvector 扩展。
-- 如果这一步没做，后面建向量列时会直接报错：type "vector" does not exist。
CREATE EXTENSION IF NOT EXISTS vector;

-- 这个扩展用于生成 UUID。
-- Spring AI 的默认向量表通常会用到 UUID 主键，这里直接提前准备好。
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 文档台账表。
-- 这张表不负责保存向量，只负责保存：
-- 文档编号、标题、分类、原始文件路径、导入状态、块数量等管理信息。
CREATE TABLE IF NOT EXISTS kb_document (
    id BIGSERIAL PRIMARY KEY,
    doc_id VARCHAR(64) NOT NULL UNIQUE,
    title VARCHAR(255) NOT NULL,
    file_name VARCHAR(255) NOT NULL,
    category VARCHAR(64) NOT NULL,
    source_path VARCHAR(512) NOT NULL,
    status VARCHAR(32) NOT NULL,
    chunk_count INT NOT NULL DEFAULT 0,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 按状态和分类建索引，方便后面做管理查询。
CREATE INDEX IF NOT EXISTS idx_kb_document_status ON kb_document(status);
CREATE INDEX IF NOT EXISTS idx_kb_document_category ON kb_document(category);

-- 向量存储表。
-- 这个表是最基础的 PGVector 表结构：
-- content 存文本块正文
-- metadata 存元数据 JSON
-- embedding 存向量数组
CREATE TABLE IF NOT EXISTS vector_store (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    content TEXT NOT NULL,
    metadata JSON,
    embedding VECTOR(1536)
);

-- 向量索引。
-- 这里使用 HNSW，因为它是教程里推荐的 PGVector 入门首选。
CREATE INDEX IF NOT EXISTS vector_store_embedding_hnsw_idx
ON vector_store
USING hnsw (embedding vector_cosine_ops);
```

这段代码的作用：

- 创建 `kb_document`，对应教程里的“文档台账”和 `doc_id` 设计
- 创建 `vector_store`，对应教程里的“向量存储”
- `VECTOR(1536)` 对应后面我们要用的 `text-embedding-3-small`
- 这一步也把教程里“向量维度一定要和 Embedding 模型一致”这个坑提前避免了

## 六、创建测试知识文档

### 6.1 创建 `knowledge-base/01-rag-intro.md`

```md
# 为什么需要 RAG

大模型并不真正理解最新业务文档，它只能根据训练时学过的公开数据进行生成。

如果企业内部有自己的产品手册、接口文档、制度文档和 FAQ，大模型本身并不知道这些内容。

RAG 的核心思路是：

1. 先把业务文档切分成小块
2. 再把每个小块转成向量写入向量数据库
3. 用户提问时，先从向量数据库里找相关片段
4. 再把相关片段和用户问题一起发给大模型生成答案

RAG 解决的是知识截止、私有数据和回答可追溯这三个问题。
```

这段代码的作用：

- 这是第一份最简单的测试知识文档
- 你后面导入它之后，就可以直接问“RAG 解决什么问题”
- 这份文档内容和前面教程的认知篇是一致的

### 6.2 创建 `knowledge-base/02-chunking.md`

```md
# 分块策略

分块是 RAG 系统效果差异最大的环节之一。

如果块太大，语义会混杂在一起，检索精度会下降。

如果块太小，单块上下文不完整，模型拿到片段后可能回答不完整。

通用建议是：

- 技术文档或产品手册，可以先从 300 到 600 Token 的块大小开始尝试
- 相邻块之间保留 10% 到 15% 的重叠区
- 先按段落边界切，再考虑 Token 长度，不要一上来就简单硬切
```

这段代码的作用：

- 这是第二份测试知识文档
- 你后面可以直接问“为什么分块不能太大也不能太小”
- 这份文档内容对应教程里“分块策略”那一篇

```yaml
server:
  # 应用端口。
  port: 8080

spring:
  application:
    # 应用名称，主要用于日志和识别。
    name: springboot-minimal-rag-demo

  datasource:
    # PostgreSQL 数据源。
    url: jdbc:postgresql://localhost:5432/rag_demo
    username: postgres
    password: 123456
    driver-class-name: org.postgresql.Driver

  ai:
    openai:
      # API Key 通过环境变量注入，避免直接写死到代码库。
      api-key: ${OPENAI_API_KEY:please-set-your-openai-api-key}

      # 如果你后面要改成 OpenAI 兼容接口，可以额外设置这个地址。
      # 比如走代理或者走兼容供应商时，通常只需要改这里和模型名。
      base-url: ${OPENAI_BASE_URL:https://api.openai.com}

      chat:
        options:
          # 这里先用一个成本和效果比较平衡的聊天模型。
          model: gpt-4o-mini
          # RAG 问答温度不宜太高，否则会增加“自由发挥”概率。
          temperature: 0.2

      embedding:
        options:
          # 这里固定使用 1536 维的嵌入模型。
          # 这样就能和 init.sql 里的 VECTOR(1536) 对上。
          model: text-embedding-3-small

    vectorstore:
      pgvector:
        # 因为我们已经手动建表，所以这里关闭自动建表。
        initialize-schema: false
        # 向量维度必须和 embedding 模型保持一致。
        dimensions: 1536
        # 指定向量表名，和 init.sql 里的表名保持一致。
        table-name: vector_store

mybatis:
  # Mapper XML 文件位置。
  mapper-locations: classpath:mapper/*.xml
  configuration:
    # 把下划线字段自动映射到驼峰属性。
    map-underscore-to-camel-case: true

rag:
  knowledge-base:
    # 默认知识库分类。
    default-category: tutorial
    # 这是最小版的基础分块大小。
    chunk-size: 500
    # 这是最小版的基础重叠大小。
    chunk-overlap: 80
  retrieval:
    # 纯向量检索先从 TopK=5 开始，和教程建议保持一致。
    top-k: 5
    # 相似度阈值先从 0.55 起步，偏保守但不至于过严。
    similarity-threshold: 0.55
```

这段代码的作用：

- 统一管理数据库、模型、向量维度和最基础的 RAG 参数
- 这里最重要的是 `embedding model` 和 `dimensions` 一定要一致
- `chunk-size`、`chunk-overlap`、`top-k`、`similarity-threshold` 都是在把教程里的选型和调参建议落成可执行配置

### 2、创建 `src/main/java/com/qiange/ragdemo/common/Result.java`

```java
package com.qiange.ragdemo.common;

import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

/**
 * 统一接口返回体。
 *
 * 这里不追求复杂，只做最小可用：
 * - success：标识是否成功
 * - message：给前端或调用方看的说明信息
 * - data：真正的返回数据
 *
 * 为什么教程型项目也建议统一返回？
 * 因为你后面调试导入接口、问答接口时，
 * 能更直观看出到底是业务失败还是系统异常。
 */
@Data
@NoArgsConstructor
@AllArgsConstructor
public class Result<T> {

    private boolean success;

    private String message;

    private T data;

    public static <T> Result<T> ok(T data) {
        return new Result<>(true, "success", data);
    }

    public static <T> Result<T> ok(String message, T data) {
        return new Result<>(true, message, data);
    }

    public static <T> Result<T> fail(String message) {
        return new Result<>(false, message, null);
    }
}
```

这段代码的作用：

- 统一接口响应格式，方便测试和排错
- 后面导入知识、RAG 问答都会直接复用它

### 1、创建 `src/main/java/com/qiange/ragdemo/loader/LocalKnowledgeLoader.java`

```java
package com.qiange.ragdemo.loader;

import com.qiange.ragdemo.common.RagConstants;
import org.springframework.ai.document.Document;
import org.springframework.stereotype.Component;
import org.springframework.util.StringUtils;

import java.io.IOException;
import java.nio.charset.StandardCharsets;
import java.nio.file.Files;
import java.nio.file.Path;
import java.time.LocalDateTime;
import java.util.HashMap;
import java.util.Map;

/**
 * 本地知识文件加载器。
 *
 * 这里我们先做最小能力：
 * - 支持 md
 * - 支持 txt
 * - 把文件内容读取成一个 Spring AI Document
 * - 给 Document 附带最基础的 Metadata
 *
 * 为什么这里不一开始就上 PDF / Word / Excel？
 * 因为教程强调过，文档处理是一个独立工程。
 * 最小项目先把纯文本链路打通，再逐步扩展格式支持，学习曲线会更稳。
 */
@Component
public class LocalKnowledgeLoader {

    public Document load(String filePath, String category, String title) {
        try {
            Path path = Path.of(filePath);

            if (!Files.exists(path)) {
                throw new IllegalArgumentException("文件不存在：" + filePath);
            }

            String fileName = path.getFileName().toString();
            String suffix = getSuffix(fileName);

            if (!"md".equalsIgnoreCase(suffix) && !"txt".equalsIgnoreCase(suffix)) {
                throw new IllegalArgumentException("当前最小版只支持 md/txt 文件，实际类型：" + suffix);
            }

            String content = Files.readString(path, StandardCharsets.UTF_8);

            if (!StringUtils.hasText(content)) {
                throw new IllegalArgumentException("文件内容为空，不允许导入空文档");
            }

            Map<String, Object> metadata = new HashMap<>();
            metadata.put(RagConstants.METADATA_SOURCE_FILE_NAME, fileName);
            metadata.put(RagConstants.METADATA_SOURCE_PATH, path.toAbsolutePath().toString());
            metadata.put(RagConstants.METADATA_CATEGORY, category);
            metadata.put(RagConstants.METADATA_TITLE, StringUtils.hasText(title) ? title : fileName);
            metadata.put("loadedAt", LocalDateTime.now().toString());

            return new Document(content, metadata);
        } catch (IOException e) {
            throw new RuntimeException("读取知识文件失败：" + filePath, e);
        }
    }

    private String getSuffix(String fileName) {
        int index = fileName.lastIndexOf('.');
        if (index < 0 || index == fileName.length() - 1) {
            return "";
        }
        return fileName.substring(index + 1);
    }
}
```

这段代码的作用：

- 把本地文件读取成 Spring AI `Document`
- 在最早阶段就给文档带上 `sourceFileName`、`sourcePath`、`category` 等 Metadata
- 对应教程里“文档加载”和“Metadata 设计”的落地动作

### 2、创建 `src/main/java/com/qiange/ragdemo/loader/KnowledgeChunker.java`

```java
package com.qiange.ragdemo.loader;

import com.qiange.ragdemo.common.RagConstants;
import com.qiange.ragdemo.config.RagProperties;
import lombok.RequiredArgsConstructor;
import org.springframework.ai.document.Document;
import org.springframework.stereotype.Component;

import java.util.ArrayList;
import java.util.HashMap;
import java.util.List;
import java.util.Map;

/**
 * 文本分块器。
 *
 * 这里我们没有直接使用黑盒分块器，而是手写一个“字符窗口 + overlap”的最小实现。
 * 原因很简单：
 * 1. 教学更透明，你能直观看懂 chunk-size 和 overlap 是怎么生效的
 * 2. 更容易和教程中的分块概念一一对应
 * 3. 后面你想替换成 TokenTextSplitter、递归分块器也更容易
 */
@Component
@RequiredArgsConstructor
public class KnowledgeChunker {

    private final RagProperties ragProperties;

    public List<Document> split(Document originalDocument) {
        String originalText = originalDocument.getText();
        String normalizedText = originalText.replace("\r\n", "\n").trim();

        Integer chunkSize = ragProperties.getKnowledgeBase().getChunkSize();
        Integer chunkOverlap = ragProperties.getKnowledgeBase().getChunkOverlap();

        int step = chunkSize - chunkOverlap;
        if (step <= 0) {
            throw new IllegalArgumentException("chunk-size 必须大于 chunk-overlap，否则会导致死循环");
        }

        List<Document> result = new ArrayList<>();
        int start = 0;
        int chunkIndex = 0;

        while (start < normalizedText.length()) {
            int end = Math.min(start + chunkSize, normalizedText.length());
            String chunkText = normalizedText.substring(start, end).trim();

            if (!chunkText.isEmpty()) {
                Map<String, Object> metadata = new HashMap<>(originalDocument.getMetadata());
                metadata.put(RagConstants.METADATA_CHUNK_INDEX, chunkIndex);
                metadata.put(RagConstants.METADATA_START_OFFSET, start);
                metadata.put(RagConstants.METADATA_END_OFFSET, end);

                result.add(new Document(chunkText, metadata));
                chunkIndex++;
            }

            if (end >= normalizedText.length()) {
                break;
            }

            start += step;
        }

        return result;
    }
}
```

这段代码的作用：

- 把一个完整文档切成多个可嵌入、可检索的小片段
- 通过 `chunk-overlap` 减少语义在切块边界处被截断
- 对应教程里“分块策略是 RAG 效果差异最大的环节”

### 4、创建 `src/main/java/com/qiange/ragdemo/service/impl/RagChatServiceImpl.java`

```java
package com.qiange.ragdemo.service.impl;

import com.qiange.ragdemo.common.RagConstants;
import com.qiange.ragdemo.config.RagProperties;
import com.qiange.ragdemo.dto.AskQuestionRequest;
import com.qiange.ragdemo.dto.AskQuestionResponse;
import com.qiange.ragdemo.dto.ReferenceChunkResponse;
import com.qiange.ragdemo.service.RagChatService;
import lombok.RequiredArgsConstructor;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.document.Document;
import org.springframework.ai.vectorstore.SearchRequest;
import org.springframework.ai.vectorstore.VectorStore;
import org.springframework.stereotype.Service;
import org.springframework.util.StringUtils;

import java.util.List;
import java.util.Map;
import java.util.stream.Collectors;

/**
 * RAG 问答服务实现。
 *
 * 整条主链路如下：
 * 1. 用用户问题去向量库检索相关片段
 * 2. 把片段整理成上下文
 * 3. 把“问题 + 上下文”一起交给大模型
 * 4. 返回答案，并把命中的片段一起带回去
 */
@Service
@RequiredArgsConstructor
public class RagChatServiceImpl implements RagChatService {

    private final VectorStore vectorStore;

    private final ChatClient chatClient;

    private final RagProperties ragProperties;

    @Override
    public AskQuestionResponse ask(AskQuestionRequest request) {
        SearchRequest.Builder searchBuilder = SearchRequest.builder()
                .query(request.getQuestion())
                .topK(ragProperties.getRetrieval().getTopK())
                .similarityThreshold(ragProperties.getRetrieval().getSimilarityThreshold());

        if (StringUtils.hasText(request.getCategory())) {
            searchBuilder.filterExpression("category == '" + escapeFilterValue(request.getCategory()) + "'");
        }

        List<Document> retrievedDocuments = vectorStore.similaritySearch(searchBuilder.build());

        if (retrievedDocuments == null || retrievedDocuments.isEmpty()) {
            return AskQuestionResponse.builder()
                    .answer("当前知识库中没有检索到足够相关的内容，请先补充知识文档后再提问。")
                    .references(List.of())
                    .build();
        }

        String context = buildContext(retrievedDocuments);

        String systemPrompt = """
                你是一个严格基于知识库回答问题的助手。
                你的回答必须遵守以下规则：
                1. 优先依据提供的知识片段回答
                2. 如果知识片段不足以回答，就明确说明“根据当前知识库无法确定”
                3. 不要编造知识库中不存在的事实
                4. 回答尽量结构化、简洁、可执行
                """;

        String userPrompt = """
                请基于以下知识片段回答问题。

                【知识片段开始】
                %s
                【知识片段结束】

                用户问题：
                %s
                """.formatted(context, request.getQuestion());

        String answer = chatClient.prompt()
                .system(systemPrompt)
                .user(userPrompt)
                .call()
                .content();

        List<ReferenceChunkResponse> references = retrievedDocuments.stream()
                .map(this::toReferenceChunkResponse)
                .collect(Collectors.toList());

        return AskQuestionResponse.builder()
                .answer(answer)
                .references(references)
                .build();
    }

    private String buildContext(List<Document> documents) {
        StringBuilder builder = new StringBuilder();

        for (int i = 0; i < documents.size(); i++) {
            Document document = documents.get(i);
            Map<String, Object> metadata = document.getMetadata();

            builder.append("片段").append(i + 1).append("：").append("\n");
            builder.append("来源文件：")
                    .append(metadata.getOrDefault(RagConstants.METADATA_SOURCE_FILE_NAME, "unknown"))
                    .append("\n");
            builder.append("分块序号：")
                    .append(metadata.getOrDefault(RagConstants.METADATA_CHUNK_INDEX, -1))
                    .append("\n");
            builder.append(document.getText()).append("\n\n");
        }

        return builder.toString();
    }

    private ReferenceChunkResponse toReferenceChunkResponse(Document document) {
        Map<String, Object> metadata = document.getMetadata();

        return ReferenceChunkResponse.builder()
                .sourceFileName(stringValue(metadata.get(RagConstants.METADATA_SOURCE_FILE_NAME)))
                .sourcePath(stringValue(metadata.get(RagConstants.METADATA_SOURCE_PATH)))
                .chunkIndex(intValue(metadata.get(RagConstants.METADATA_CHUNK_INDEX)))
                .content(document.getText())
                .build();
    }

    private String stringValue(Object value) {
        return value == null ? "" : String.valueOf(value);
    }

    private Integer intValue(Object value) {
        if (value == null) {
            return null;
        }
        return Integer.parseInt(String.valueOf(value));
    }

    private String escapeFilterValue(String value) {
        return value.replace("'", "\\'");
    }
}
```

这段代码的作用：

- 这是整个最小版 RAG 的“问答主链路”
- 把向量检索、上下文拼装、Prompt 约束、答案生成串起来
- 对应教程里的“相似度搜索局限”“检索结果要参与 Prompt 设计”的核心思想

### 5、创建 `src/main/java/com/qiange/ragdemo/service/KnowledgeImportFacade.java`

```java
package com.qiange.ragdemo.service;

import com.qiange.ragdemo.config.RagProperties;
import com.qiange.ragdemo.dto.ImportLocalFileRequest;
import com.qiange.ragdemo.entity.KnowledgeDocumentEntity;
import com.qiange.ragdemo.loader.KnowledgeChunker;
import com.qiange.ragdemo.loader.LocalKnowledgeLoader;
import lombok.RequiredArgsConstructor;
import org.springframework.ai.document.Document;
import org.springframework.ai.vectorstore.VectorStore;
import org.springframework.stereotype.Service;
import org.springframework.util.DigestUtils;
import org.springframework.util.StringUtils;

import java.nio.charset.StandardCharsets;
import java.util.List;

/**
 * 知识导入编排服务。
 *
 * 这个类不是简单 CRUD，它负责把完整导入流程串起来：
 * 1. 读文件
 * 2. 算哈希
 * 3. 建台账
 * 4. 文本分块
 * 5. 写入向量库
 * 6. 更新导入状态
 */
@Service
@RequiredArgsConstructor
public class KnowledgeImportFacade {

    private final KnowledgeDocumentService knowledgeDocumentService;

    private final LocalKnowledgeLoader localKnowledgeLoader;

    private final KnowledgeChunker knowledgeChunker;

    private final VectorStore vectorStore;

    private final RagProperties ragProperties;

    public Integer importLocalFile(ImportLocalFileRequest request) {
        String category = StringUtils.hasText(request.getCategory())
                ? request.getCategory()
                : ragProperties.getKnowledgeBase().getDefaultCategory();

        Document originalDocument = localKnowledgeLoader.load(
                request.getFilePath(),
                category,
                request.getTitle()
        );

        String fileHash = DigestUtils.md5DigestAsHex(originalDocument.getText().getBytes(StandardCharsets.UTF_8));

        KnowledgeDocumentEntity existing = knowledgeDocumentService.getByFileHash(fileHash);
        if (existing != null) {
            throw new IllegalArgumentException("相同内容的文件已经导入过，无需重复导入");
        }

        KnowledgeDocumentEntity record = knowledgeDocumentService.createPendingDocument(
                request.getFilePath(),
                fileHash,
                category
        );

        try {
            List<Document> chunkDocuments = knowledgeChunker.split(originalDocument);

            if (chunkDocuments.isEmpty()) {
                throw new IllegalArgumentException("切分后没有可入库的有效片段");
            }

            vectorStore.add(chunkDocuments);

            knowledgeDocumentService.markImported(
                    record.getId(),
                    "导入成功，片段数量：" + chunkDocuments.size()
            );

            return chunkDocuments.size();
        } catch (Exception e) {
            knowledgeDocumentService.markImportFailed(record.getId(), e.getMessage());
            throw e;
        }
    }
}
```

这段代码的作用：

- 这是整个最小版 RAG 的“导入主链路”
- 把加载、分块、Embedding 入库、台账记录串成一个完整动作
- 对应教程里的“两条管道”中的离线索引管道

## 十六、你需要按什么顺序启动和测试

这一节非常重要。

很多人不是代码不会写，而是“启动顺序错了”，最后误以为 RAG 很复杂。
这个最小项目请按下面顺序跑。

### 第 1 步：启动 PostgreSQL + PGVector

在项目根目录执行：

```powershell
docker compose up -d
```

如果你想确认容器是否正常启动，可以执行：

```powershell
docker ps
```

你应该能看到 PostgreSQL 相关容器处于运行状态。

### 第 2 步：配置 OpenAI API Key

当前 PowerShell 会话中执行：

```powershell
$env:OPENAI_API_KEY="替换成你自己的 OpenAI Key"
```

如果你走兼容 OpenAI 协议的模型服务，还可以额外设置：

```powershell
$env:OPENAI_BASE_URL="你的兼容接口地址"
```

### 第 3 步：启动 Spring Boot 应用

在项目根目录执行：

```powershell
mvn spring-boot:run
```

如果一切正常，应用会启动在：

```text
http://localhost:8080
```

### 第 4 步：先导入知识，再进行问答

这个顺序一定不能反。

原因是：

1. 没有先导入知识，向量库里就没有任何可检索片段
2. 没有可检索片段，RAG 问答自然只能返回“没有找到相关知识”

## 十九、常见报错排查

### 1、报错：`type "vector" does not exist`

原因：

- PGVector 扩展没有成功安装
- 或者初始化 SQL 没有执行成功

排查方式：

1. 确认容器是否正常启动
2. 确认 `init.sql` 是否被挂载并执行
3. 在数据库中确认 `create extension if not exists vector;` 是否已经执行

### 2、报错：向量维度不匹配

典型表现：

- 插入向量时报维度错误

原因：

- 你的 `embedding model` 维度和 `vector_store.embedding VECTOR(1536)` 不一致

当前文档默认是：

1. Embedding 模型：`text-embedding-3-small`
2. 向量维度：`1536`

这两者必须同时修改、同时保持一致。

### 3、报错：`OPENAI_API_KEY` 未配置

原因：

- 没有在当前终端会话中设置环境变量

处理方式：

```powershell
$env:OPENAI_API_KEY="你的 OpenAI Key"
```

然后重新启动应用。

### 4、问题：导入成功了，但问答总是“检索不到内容”

通常从这几个方向排查：

1. 问题和知识文本表述差异太大，相似度没过阈值
2. `similarity-threshold` 设得太高
3. `chunk-size` 过大，导致片段语义不聚焦
4. `chunk-size` 过小，导致重要语义被切碎
5. 你传了 `category`，但导入和查询时分类不一致

最小版建议先这样试：

1. 把 `similarity-threshold` 从 `0.55` 调低到 `0.45`
2. 保持 `top-k=5`
3. 检查问题措辞是否和知识文档主题接近

### 5、问题：重复导入时报错

这是当前最小版的预期行为，不是 Bug。

原因是：

- 我们用 `file_hash` 做了去重保护
- 相同内容的文件只允许导入一次

这样做的好处是：

- 避免重复知识被多次写入向量库
- 避免检索结果被重复片段污染

## 二十、这个项目接下来怎么升级

如果你已经把最小版跑通，下一步推荐按下面顺序升级：

### 升级 1：支持目录批量导入

把当前“单文件导入”扩展成“扫描某个目录下所有 md/txt 文件并批量导入”。

这一步解决的问题是：

- 不需要每次手动传一个文件路径
- 更接近真实知识库维护方式

### 升级 2：加入 BM25 / PostgreSQL 全文检索

把“纯向量召回”升级为“向量召回 + 关键词召回”。

这一步对应教程第 105 篇。

价值在于：

- 解决纯向量检索对精确术语、缩写、代码片段命中不稳定的问题

### 升级 3：加入查询改写

在真正检索之前，先让模型把用户问题改写成更适合检索的表达。

这一步对应教程第 106 篇。

价值在于：

- 用户问题口语化时，提升召回命中率

### 升级 4：加入 Reranker

把初步召回的若干片段再做一次相关性精排。

这一步对应教程第 107 篇。

价值在于：

- 降低“召回到了但排序不对”的问题

### 升级 5：加入上下文压缩

不要把所有召回片段原封不动喂给模型，而是做裁剪、去重、压缩。

这一步对应教程第 108 篇。

价值在于：

- 降低上下文噪声
- 降低 token 成本
- 提高最终答案稳定性

### 升级 6：加入离线评估体系

准备一批标准问题与标准答案，统计命中率、MRR、答案质量。

这一步对应教程第 104 篇。

价值在于：

- 让 RAG 优化不再靠“感觉”，而是靠指标

## 二十一、你在这个最小项目里真正学到了什么

如果你把这份文档完整敲完并跑通，你真正学到的不是“调用了一个接口”，而是下面这些工程事实：

1. RAG 不是只有一个问答接口，它至少包含“离线索引”和“在线检索生成”两条链路
2. 文档加载、Metadata、分块质量，会直接决定后面的召回效果
3. 向量数据库不是 RAG 的全部，真正决定体验的是“召回得对不对”
4. Prompt 在 RAG 中并没有消失，它从“直接回答”变成了“基于检索上下文回答”
5. 最小版项目能跑通，只代表闭环打通，不代表效果已经最优

这也是为什么教程后半段会把重心放在：

1. 混合检索
2. 查询改写
3. Reranker
4. 上下文压缩
5. 召回评估

因为这些东西，才是真正决定 RAG 质量上限的部分。
