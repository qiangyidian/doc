# Java 常见锁系统学习实战文档

**目标：** 从零创建一个 `Spring Boot 3` 示例项目，系统学习 Java 后端开发中最常见的锁的使用方式，既覆盖单机 JVM 内的普通锁，也覆盖分布式系统里的分布式锁。你看完之后，不只是知道 `synchronized`、`ReentrantLock`、`ReadWriteLock`、`StampedLock`、`Condition`、`AtomicInteger`、`Redisson` 是什么，还会知道它们分别适合什么场景、该怎么写、为什么这样写。

**架构说明：** 这份文档使用 `Spring Boot + Redisson + Redis + Docker Compose`。项目主线不是一个复杂业务系统，而是一个“锁实验场”，通过多个真实后端场景来学习锁，包括本地库存扣减、读多写少配置缓存、任务队列等待唤醒、乐观读、分布式下单锁、公平锁、分布式读写锁。这样你能在一个项目里把 Java 开发里最常见的锁串起来学。

**技术栈：** Java 17、Spring Boot 3.2.5、Spring Web、Spring Validation、Redisson 3.27.2、Redis 7、Docker Compose、Lombok、`synchronized`、`ReentrantLock`、`ReentrantReadWriteLock`、`Condition`、`StampedLock`、`AtomicInteger`

## 一、为什么 Java 后端开发必须系统学习锁

锁不是“面试题专用知识点”，而是你在真实后端开发里非常容易碰到的东西。

最常见的场景就是：

1. 同一个 JVM 内多个线程同时扣减库存
2. 读多写少的共享配置缓存
3. 生产者消费者队列的等待与唤醒
4. 统计计数器的高并发更新
5. 多个服务实例同时处理同一笔订单
6. 分布式系统里保证一个资源同一时间只能被一个节点操作

所以学锁的关键不是背 API，而是理解 3 个问题：

1. 这个场景是不是会发生并发冲突
2. 这个锁应该是单机锁还是分布式锁
3. 这个锁选型后，会不会带来性能和死锁问题

## 三、先建立锁的全局认知

很多初学者会问：

- `synchronized` 和 `ReentrantLock` 有什么区别？
- 什么时候该用读写锁？
- 分布式锁和本地锁到底差在哪？

你可以先记住下面这个总表：

| 锁类型 | 作用范围 | 典型场景 | 特点 |
|---|---|---|---|
| `synchronized` | 单 JVM | 简单共享资源保护 | 语法简单 |
| `ReentrantLock` | 单 JVM | 需要可中断、可超时、可公平的锁 | 更灵活 |
| `ReentrantReadWriteLock` | 单 JVM | 读多写少 | 提高读性能 |
| `Condition` | 单 JVM | 等待/唤醒队列 | 更细粒度控制 |
| `StampedLock` | 单 JVM | 高频读、低频写 | 支持乐观读 |
| `AtomicInteger` | 单 JVM | 简单计数器 | 无锁 CAS |
| `Redisson Lock` | 分布式 | 多实例下单、任务幂等 | 基于 Redis |

## 十二、创建 `synchronized` 场景服务 `SynchronizedStockService.java`

```java
package com.example.lockdemo.service;

import org.springframework.stereotype.Service;

@Service
public class SynchronizedStockService {

    // 用一个成员变量模拟当前库存。
    private int stock = 20;

    public synchronized int deduct(int count) {
        // synchronized 修饰实例方法，等价于锁住当前对象 this。
        // 同一时刻只能有一个线程进入这个方法。
        if (count <= 0) {
            throw new IllegalArgumentException("扣减数量必须大于 0");
        }

        if (stock < count) {
            throw new RuntimeException("库存不足，当前库存=" + stock);
        }

        stock -= count;
        return stock;
    }

    public synchronized int currentStock() {
        return stock;
    }
}
```

这段代码的作用：

- 演示最基础的 `synchronized` 用法
- 适用于单 JVM 内的简单共享资源保护，比如单机库存、单机计数器

## 十三、创建 `ReentrantLock` 场景服务 `ReentrantLockStockService.java`

```java
package com.example.lockdemo.service;

import org.springframework.stereotype.Service;

import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.ReentrantLock;

@Service
public class ReentrantLockStockService {

    private int stock = 20;

    // 这里创建的是可重入锁，默认是非公平锁。
    private final ReentrantLock lock = new ReentrantLock();

    public int deduct(int count) throws InterruptedException {
        if (count <= 0) {
            throw new IllegalArgumentException("扣减数量必须大于 0");
        }

        // tryLock 可以让我们在获取不到锁时快速失败，而不是一直死等。
        boolean success = lock.tryLock(3, TimeUnit.SECONDS);
        if (!success) {
            throw new RuntimeException("获取锁超时，请稍后重试");
        }

        try {
            if (stock < count) {
                throw new RuntimeException("库存不足，当前库存=" + stock);
            }

            stock -= count;
            return stock;
        } finally {
            // ReentrantLock 一定要在 finally 里手动释放。
            lock.unlock();
        }
    }

    public int currentStock() {
        return stock;
    }
}
```

这段代码的作用：

- 演示 `ReentrantLock` 的典型用法
- 相比 `synchronized`，它更灵活，支持 `tryLock`、可中断、公平锁等高级能力

## 十四、创建读写锁场景服务 `ReadWriteConfigService.java`

```java
package com.example.lockdemo.service;

import org.springframework.stereotype.Service;

import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.locks.ReentrantReadWriteLock;

@Service
public class ReadWriteConfigService {

    // 用 Map 模拟“配置缓存”。
    private final Map<String, String> configMap = new HashMap<>();

    // 读写锁：适合读多写少场景。
    private final ReentrantReadWriteLock readWriteLock = new ReentrantReadWriteLock();

    public String read(String key) {
        // 读操作加读锁，多个读线程可以并发进入。
        readWriteLock.readLock().lock();
        try {
            return configMap.getOrDefault(key, "未配置");
        } finally {
            readWriteLock.readLock().unlock();
        }
    }

    public void write(String key, String value) {
        // 写操作加写锁，写时不允许其他线程再读或写。
        readWriteLock.writeLock().lock();
        try {
            configMap.put(key, value);
        } finally {
            readWriteLock.writeLock().unlock();
        }
    }
}
```

这段代码的作用：

- 演示 `ReentrantReadWriteLock`
- 它最适合读多写少场景，例如配置中心缓存、本地字典缓存

## 十五、创建等待唤醒场景服务 `ConditionTaskQueueService.java`

```java
package com.example.lockdemo.service;

import org.springframework.stereotype.Service;

import java.util.ArrayDeque;
import java.util.Deque;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

@Service
public class ConditionTaskQueueService {

    private final Deque<String> taskQueue = new ArrayDeque<>();

    // 用一把显式锁保护队列。
    private final ReentrantLock lock = new ReentrantLock();

    // 队列为空时，消费者在这里等待。
    private final Condition notEmpty = lock.newCondition();

    // 队列已满时，生产者在这里等待。
    private final Condition notFull = lock.newCondition();

    private final int maxSize = 5;

    public void produce(String taskName) throws InterruptedException {
        lock.lock();
        try {
            while (taskQueue.size() >= maxSize) {
                // 队列满了，生产者进入等待。
                notFull.await();
            }

            taskQueue.addLast(taskName);

            // 新增任务后，唤醒一个等待中的消费者。
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    public String consume() throws InterruptedException {
        lock.lock();
        try {
            while (taskQueue.isEmpty()) {
                // 队列为空，消费者进入等待。
                notEmpty.await();
            }

            String task = taskQueue.removeFirst();

            // 消费掉一个任务后，唤醒一个等待中的生产者。
            notFull.signal();
            return task;
        } finally {
            lock.unlock();
        }
    }
}
```

这段代码的作用：

- 演示 `Condition` 在等待/唤醒场景中的用法
- 这是生产者消费者模型在显式锁体系下的经典写法

## 十六、创建 `StampedLock` 场景服务 `StampedLockPriceService.java`

```java
package com.example.lockdemo.service;

import org.springframework.stereotype.Service;

import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.locks.StampedLock;

@Service
public class StampedLockPriceService {

    private final Map<Long, Integer> priceMap = new HashMap<>();

    // StampedLock 适合“读非常多，写比较少”的场景。
    private final StampedLock stampedLock = new StampedLock();

    public int getPrice(Long productId) {
        // 先尝试乐观读，不会真正阻塞写线程。
        long stamp = stampedLock.tryOptimisticRead();
        Integer price = priceMap.get(productId);

        // 如果乐观读期间发生了写操作，validate 会失败。
        if (!stampedLock.validate(stamp)) {
            // 乐观读失败后，再退化成悲观读锁。
            stamp = stampedLock.readLock();
            try {
                price = priceMap.get(productId);
            } finally {
                stampedLock.unlockRead(stamp);
            }
        }

        return price == null ? 0 : price;
    }

    public void updatePrice(Long productId, Integer price) {
        long stamp = stampedLock.writeLock();
        try {
            priceMap.put(productId, price);
        } finally {
            stampedLock.unlockWrite(stamp);
        }
    }
}
```

这段代码的作用：

- 演示 `StampedLock` 的乐观读能力
- 它适合高频读、低频写场景，但也更复杂，使用时要更谨慎

```java
package com.example.lockdemo.service;

import org.springframework.stereotype.Service;

import java.util.concurrent.atomic.AtomicInteger;

@Service
public class AtomicCounterService {

    // AtomicInteger 不是传统意义上的锁，但它是高并发计数器的常见方案。
    private final AtomicInteger visitCount = new AtomicInteger(0);

    public int increment() {
        // 通过 CAS 原子更新计数值。
        return visitCount.incrementAndGet();

    public int current() {
        return visitCount.get();
    }
}
```

这段代码的作用：

- 它经常和锁一起被问到，因为它适合替代某些简单锁场景

## 十八、创建分布式可重入锁服务 `RedissonOrderService.java`

```java
package com.example.lockdemo.service;

import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.springframework.stereotype.Service;

@Service
public class RedissonOrderService {

    private final RedissonClient redissonClient;

    public RedissonOrderService(RedissonClient redissonClient) {
        this.redissonClient = redissonClient;
    }

    public String createOrder(Long userId, Long productId) {
        // 分布式锁 key 通常要和具体业务资源绑定。
        String lockKey = "lock:order:product:" + productId;

        RLock lock = redissonClient.getLock(lockKey);
        lock.lock();
        try {
            // 这里模拟真实的下单逻辑。
            // Redisson 的锁是可重入锁，同一个线程再次获取同一把锁不会死锁。
            return "下单成功，userId=" + userId + ", productId=" + productId;
        } finally {
            lock.unlock();
        }
    }
}
```

## 十九、创建分布式公平锁服务 `RedissonFairLockService.java`

```java
package com.example.lockdemo.service;

import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.springframework.stereotype.Service;

@Service
public class RedissonFairLockService {

    private final RedissonClient redissonClient;

    public RedissonFairLockService(RedissonClient redissonClient) {
        this.redissonClient = redissonClient;
    }

    public String grab(Long userId) {
        // 公平锁会尽量按照请求锁的先后顺序来获取锁。
        RLock fairLock = redissonClient.getFairLock("lock:grab:fair");
        fairLock.lock();
        try {
            return "公平锁抢单成功，userId=" + userId;
        } finally {
            fairLock.unlock();
        }
    }
}
```

这段代码的作用：

- 演示 Redisson 公平锁的基本用法
- 公平锁更强调“先来先得”，但性能通常会低于非公平锁

## 二十、创建分布式读写锁服务 `RedissonReadWriteService.java`

```java
package com.example.lockdemo.service;

import org.redisson.api.RReadWriteLock;
import org.redisson.api.RedissonClient;
import org.springframework.stereotype.Service;

@Service
public class RedissonReadWriteService {

    private final RedissonClient redissonClient;

    public RedissonReadWriteService(RedissonClient redissonClient) {
        this.redissonClient = redissonClient;
    }

    public String readConfig(String key) {
        RReadWriteLock rwLock = redissonClient.getReadWriteLock("lock:config:" + key);
        rwLock.readLock().lock();
        try {
            return "读取配置成功，key=" + key;
        } finally {
            rwLock.readLock().unlock();
        }
    }

    public String writeConfig(String key, String value) {
        RReadWriteLock rwLock = redissonClient.getReadWriteLock("lock:config:" + key);
        rwLock.writeLock().lock();
        try {
            return "写入配置成功，key=" + key + ", value=" + value;
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

这段代码的作用：

- 演示 Redisson 分布式读写锁
- 当你的配置、字典、规则在多节点部署下也需要读多写少保护时，这种锁会很有价值

## 二十二、整个项目的学习顺序建议

这份项目我建议你按下面顺序去学，不要一上来就直接看 Redisson。

### 第一步：先吃透单 JVM 内的锁

顺序建议：

1. `synchronized`
2. `ReentrantLock`
3. `ReentrantReadWriteLock`
4. `Condition`
5. `StampedLock`

原因：

- 这些锁都是单机 JVM 内的基础能力
- 不懂这些，再去看分布式锁，你会把很多问题混在一起

### 第二步：再看无锁原子类

学习：

- `AtomicInteger`

原因：

- 它经常和“锁”一起出现在面试和实际开发中
- 适合理解“不是所有并发问题都必须加传统锁”

### 第三步：最后再看分布式锁

顺序建议：

1. `Redisson 可重入锁`
2. `Redisson 公平锁`
3. `Redisson 读写锁`

原因：

- 分布式锁依赖 Redis
- 同时会多出网络、超时、续期、锁丢失这些问题

## 二十五、锁选型建议：真实开发里该怎么选

这个部分非常重要。  
你以后工作里最容易卡住的，不是不会写锁，而是不知道该选哪种锁。

### 25.1 场景很简单，只是单机共享变量保护

优先考虑：

- `synchronized`

原因：

- 语法简单
- 可读性高
- 足够应对大部分简单单机场景

### 25.2 需要超时获取锁、可中断、手动控制

优先考虑：

- `ReentrantLock`

原因：

- 更灵活
- 可以 `tryLock`
- 可中断
- 可选公平锁

### 25.3 读多写少

优先考虑：

- `ReentrantReadWriteLock`
- `StampedLock`

区别：

- `ReentrantReadWriteLock` 更稳、更常规
- `StampedLock` 更激进，适合高频读，但使用更复杂

### 25.4 简单计数器

优先考虑：

- `AtomicInteger`
- `LongAdder`

原因：

- 没必要为了简单计数上重锁

### 25.5 多实例部署，多个服务节点都可能同时操作同一个资源

优先考虑：

- `Redisson` 分布式锁

原因：

- JVM 锁只在单机生效
- 分布式系统里必须上分布式锁

## 二十六、常见面试题总结

### 26.1 `synchronized` 和 `ReentrantLock` 有什么区别

推荐回答：

1. `synchronized` 是 Java 关键字，`ReentrantLock` 是 JUC 提供的类
2. `synchronized` 使用简单，自动释放锁
3. `ReentrantLock` 更灵活，支持 `tryLock`、可中断、公平锁、多个 `Condition`

### 26.2 什么叫可重入锁

推荐回答：

同一个线程已经持有这把锁时，再次获取同一把锁不会被自己阻塞，这就叫可重入。

例如：

- `synchronized`
- `ReentrantLock`
- `Redisson Lock`

都属于可重入锁。

### 26.3 公平锁和非公平锁有什么区别

推荐回答：

- 公平锁更强调先来先得
- 非公平锁更强调吞吐量
- 公平锁通常更公平，但性能可能更差

### 26.4 `ReadWriteLock` 适合什么场景

推荐回答：

适合读多写少场景。  
例如：

- 本地配置缓存
- 本地字典缓存
- 本地规则缓存

### 26.5 `StampedLock` 为什么快

推荐回答：

因为它支持乐观读。  
在很多读操作下，它不一定真的加悲观锁，从而减少锁竞争。

### 26.6 `AtomicInteger` 为什么不算传统锁

推荐回答：

因为它基于 CAS 实现原子更新，不是靠互斥阻塞线程来保护共享资源。

### 26.7 JVM 锁和分布式锁的本质区别是什么

推荐回答：

- JVM 锁只在当前进程内生效
- 分布式锁要在多个服务实例之间都能互斥

### 26.8 Redis 分布式锁为什么不能只写 `setnx`

推荐回答：

因为光 `setnx` 不够，还必须考虑：

1. 过期时间
2. 锁释放安全
3. 锁续期
4. 可重入
5. 网络超时

这也是为什么实际项目里更常用 Redisson，而不是自己手写裸 Redis 锁。

### 26.9 Redisson 的看门狗是什么

推荐回答：

当业务执行时间比较长时，Redisson 会自动帮你给锁续期，避免锁提前过期导致别的线程误入。

### 26.10 为什么分布式锁也要放在 finally 里释放

推荐回答：

因为无论业务成功还是失败，都必须释放锁，否则很容易造成后续请求一直拿不到锁。

## 二十八、最后总结

你可以把锁的学习压缩成一句话：

**锁不是为了炫技，而是为了在并发场景下保护共享资源的一致性。**

真正的后端开发里，你最重要的能力不是“记住所有锁的 API”，而是：

- 判断这里需不需要锁
- 判断该用本地锁还是分布式锁
- 判断该选简单锁、显式锁、读写锁还是无锁原子类

你把这份文档吃透之后，再去看并发、分布式锁、秒杀系统、订单系统，理解会顺很多。
