# Java 全局异常处理实战文档

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 从零搭建一个真实企业级 Spring Boot 项目中的全局异常处理体系。让初学者理解如何告别到处写 `try-catch` 的习惯，通过统一拦截机制，返回标准规范的 JSON 错误信息。

**Architecture:** 项目主链路是 `Controller/Service 抛出异常 -> @ControllerAdvice 统一拦截 -> 匹配 @ExceptionHandler -> 转换为统一的 Result 格式返回给前端`。

**Tech Stack:** Java 17, Spring Boot 3.2.5, Spring Web, Spring Validation, Lombok

---

## 一、这个机制在真实公司里为什么是必需的

在初学 Java 时，我们处理错误通常是就地 `try-catch` 并打印堆栈。但在企业级前后端分离开发中，这种做法会导致灾难：

- **痛点1：前端无法解析** —— 后端报错时默认会返回 Tomcat 的 500 HTML 错误页，前端代码无法把 HTML 渲染成友好的提示框。
- **痛点2：错误码混乱** —— A 开发抛 `{"code": 500, "msg": "失败"}`，B 开发抛 `{"status": -1, "error": "失效"}`，前端不知所措。
- **痛点3：代码极其臃肿** —— 如果每个 Controller 方法都在做 `try-catch` 包装返回结果，业务代码会被错误处理逻辑完全淹没。
- **痛点4：敏感信息泄露** —— 比如数据库连接失败或 SQL 写错，默认报错会把完整的 SQL 和表结构抛给前端，极度不安全。

**全局异常处理机制** 则完美解决了这些问题：后端开发人员在业务层遇到不合规的数据只需“大胆往外抛异常”，由于有顶层的“保安（全局拦截器）”统一处理并格式化输出，业务代码变得非常干净。

## 三、开始前先记住 2 个核心约束

1. **业务逻辑受阻，直接抛出 `BusinessException`**。例如“余额不足”、“用户不存在”，不要在 Service 层返回 null 或者 Result 对象，直接抛异常是最优雅的。
2. **永远要写一个兜底的 `Exception` 拦截**。防止那些我们没有预料到的系统级异常（如空指针、数据库宕机）以不规范的格式泄露出去。

### Task 1: 创建状态码枚举与统一返回对象

企业开发要求前后端通过约定的 Code 码进行通信。

**Files:**
- Create: `src/main/java/com/example/exceptiondemo/common/ResultCode.java`
- Create: `src/main/java/com/example/exceptiondemo/common/Result.java`

**Step 1: 创建 `ResultCode.java`**

```java
package com.example.exceptiondemo.common;

import lombok.AllArgsConstructor;
import lombok.Getter;

@Getter
@AllArgsConstructor
public enum ResultCode {
    
    SUCCESS(200, "操作成功"),
    BAD_REQUEST(400, "参数错误"),
    UNAUTHORIZED(401, "请先登录"),
    FORBIDDEN(403, "没有操作权限"),
    SYSTEM_ERROR(500, "系统开小差了，请稍后再试"),
    
    // 以下为具体业务错误码段，按模块划分
    USER_NOT_FOUND(1001, "用户不存在"),
    USER_PASSWORD_ERROR(1002, "账号或密码错误"),
    USER_FROZEN(1003, "账号已被冻结");

    private final Integer code;
    private final String message;
}
```

这段代码的作用：

- 将项目中所有可能的错误码集中管理，绝对禁止硬编码写死类似 `400` 的数字。
- 将 HTTP 状态码的概念与业务状态码结合，前端只需识别这个枚举对应的值。

**Step 2: 创建 `Result.java`**

```java
package com.example.exceptiondemo.common;

import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

@Data
@NoArgsConstructor
@AllArgsConstructor
public class Result<T> {

    private Integer code;
    private String message;
    private T data;

    public static <T> Result<T> success(T data) {
        return new Result<>(ResultCode.SUCCESS.getCode(), ResultCode.SUCCESS.getMessage(), data);
    }

    public static <T> Result<T> success() {
        return success(null);
    }

    public static <T> Result<T> fail(ResultCode resultCode) {
        return new Result<>(resultCode.getCode(), resultCode.getMessage(), null);
    }

    public static <T> Result<T> fail(Integer code, String message) {
        return new Result<>(code, message, null);
    }
}
```

这段代码的作用：

- 定义所有接口必须遵守的 JSON 外壳结构：`code`, `message`, `data`。
- 提供 `success()` 和 `fail()` 静态工厂方法，方便快速构建返回结果。

### Task 2: 定义业务自定义异常

开发中经常会遇到“密码错误”、“找不到该商品”等业务受阻情况，这不是系统的 Bug，而是业务规则。我们需要用专属的异常来承载这一信息。

**Files:**
- Create: `src/main/java/com/example/exceptiondemo/exception/BusinessException.java`

**Step 1: 创建 `BusinessException.java`**

```java
package com.example.exceptiondemo.exception;

import com.example.exceptiondemo.common.ResultCode;
import lombok.Getter;

@Getter
public class BusinessException extends RuntimeException {

    private final Integer code;

    // 方式一：直接使用错误码枚举构造
    public BusinessException(ResultCode resultCode) {
        super(resultCode.getMessage());
        this.code = resultCode.getCode();
    }

    // 方式二：自定义错误信息构造（常用于灵活提示）
    public BusinessException(Integer code, String message) {
        super(message);
        this.code = code;
    }
}
```

这段代码的作用：

- 继承 `RuntimeException` (运行时异常)，如果是 `Exception` (受查异常) 则会导致调用方被迫到处写 `throws`。
- 内部持有 `code` 属性，以便将来被全局处理器捕获时能提取出指定的业务错误码返回给前端。

### Task 3: 核心机制 - 全局异常处理器

这是整个体系的灵魂，它像一个漏斗在 Controller 外面接住抛出来的各种 “雷”。

**Files:**
- Create: `src/main/java/com/example/exceptiondemo/exception/GlobalExceptionHandler.java`

**Step 1: 创建 `GlobalExceptionHandler.java`**

```java
package com.example.exceptiondemo.exception;

import com.example.exceptiondemo.common.Result;
import com.example.exceptiondemo.common.ResultCode;
import lombok.extern.slf4j.Slf4j;
import org.springframework.validation.BindingResult;
import org.springframework.validation.FieldError;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    /**
     * 第一层：精准捕获自定义的业务异常
     */
    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        // 业务异常通常是代码主动抛出的正常拦截，打印INFO即可，没必要打印堆栈制造恐慌
        log.info("业务发生异常，code: {}, msg: {}", e.getCode(), e.getMessage());
        return Result.fail(e.getCode(), e.getMessage());
    }

    /**
     * 第二层：捕获实体参数校验异常（配合 @Valid / @Validated 使用）
     */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        BindingResult bindingResult = e.getBindingResult();
        StringBuilder errorMsg = new StringBuilder();
        // 拼接所有没通过的校验信息
        for (FieldError fieldError : bindingResult.getFieldErrors()) {
            errorMsg.append(fieldError.getField()).append(": ")
                    .append(fieldError.getDefaultMessage()).append(", ");
        }
        log.warn("参数校验未通过: {}", errorMsg);
        return Result.fail(ResultCode.BAD_REQUEST.getCode(), errorMsg.toString());
    }
    
    /**
     * 第三层：兜底拦截 未知系统异常 (如 NullPointerException, SQLException)
     */
    @ExceptionHandler(Exception.class)
    public Result<Void> handleSystemException(Exception e) {
        // 未知系统异常属于真正的Bug，必须打印 Error 级别日志和完整堆栈配合排查
        log.error("系统发生未知异常", e);
        // 对外只展示模棱两可的"系统异常"，绝不暴露真实的数据库或空指针错误信息
        return Result.fail(ResultCode.SYSTEM_ERROR);
    }
}
```

这段代码的作用：

- `@RestControllerAdvice` 结合了 `@ControllerAdvice` 和 `@ResponseBody`，作用是拦截 Controller 并返回 JSON。
- `@ExceptionHandler` 指定每个方法专门处理哪种类型的异常。越具体的异常写在越前面。
- 这里分了三层拦截策略，将业务警报、参数不合规、系统崩溃彻底区分开来并妥善安放。

### Task 4: 触发场景演示 (Controller 层) 

最后我们写一个测试的 Controller 来看一下刚才写的体系是怎么生效的，你可以不用在方法里写 `try-catch` 了。

**Files:**
- Create: `src/main/java/com/example/exceptiondemo/dto/UserRegisterRequest.java`
- Create: `src/main/java/com/example/exceptiondemo/controller/UserController.java`

**Step 1: 创建 `UserRegisterRequest.java`**

```java
package com.example.exceptiondemo.dto;

import jakarta.validation.constraints.NotBlank;
import lombok.Data;

@Data
public class UserRegisterRequest {
    
    @NotBlank(message = "用户名不能为空")
    private String username;
    
    @NotBlank(message = "密码不能为空")
    private String password;
}
```

**Step 2: 创建 `UserController.java`**

```java
package com.example.exceptiondemo.controller;

import com.example.exceptiondemo.common.Result;
import com.example.exceptiondemo.common.ResultCode;
import com.example.exceptiondemo.dto.UserRegisterRequest;
import com.example.exceptiondemo.exception.BusinessException;
import org.springframework.validation.annotation.Validated;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/users")
public class UserController {

    /**
     * 场景一：模拟业务正常流程被中断 (比如发现用户不存在)
     * 测试方式：GET /users/1
     */
    @GetMapping("/{id}")
    public Result<String> getUser(@PathVariable Long id) {
        if (id == 1L) {
            // 直接抛异常，不需要组装错误 Result，异常处理器会代劳
            throw new BusinessException(ResultCode.USER_NOT_FOUND); 
        }
        return Result.success("获取到的合法用户数据: admin" );
    }

    /**
     * 场景二：模拟必填参数没有传
     * 测试方式：POST /users (传一个空的 JSON {})
     */
    @PostMapping
    public Result<String> register(@Validated @RequestBody UserRegisterRequest request) {
        // 如果对象不合规，@Validated 会自动抛出 MethodArgumentNotValidException 异常，这在上面已经被接管了
        return Result.success("注册成功：" + request.getUsername());
    }

    /**
     * 场景三：模拟研发人员没有注意到的系统 Bug (如空指针异常)
     * 测试方式：GET /users/bug
     */
    @GetMapping("/bug")
    public Result<String> triggerBug() {
        String str = null;
        // 这一行会导致 NullPointerException
        str.length(); 
        return Result.success("这条是代码永远执行不到的地方");
    }
}
```

这段代码的作用：

- 演示了 Controller 中的代码可以有多么“极简”。没有繁琐的 `if-else` 返回，也没有任何处理错误的冗余逻辑。
- 这三种场景最终呈现给前端的始终是统一包含 `code`, `message`, `data` 的 JSON 格式结构。

## 避坑提示 💡

1. **不要大包大揽抛出 Exception**：很多初学者在 Service 抛异常时直接 `throw new Exception("业务报错")`，这会被兜底拦截器当作系统级 Bug 处理，破坏了业务的区分度，务必使用自定义的 `BusinessException`。
2. **多模块项目下的拦截范围**：如果有多个子模块，要注意 `@RestControllerAdvice` 默认拦截所有包下的 Controller，必要时可以通过 `basePackages` 属性限定它只作用于指定的包防止冲突。
