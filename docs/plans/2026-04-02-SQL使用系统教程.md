# SQL 使用系统教程

**目标：** 用一份完整文档把 SQL 从“基础语法”讲到“后端实战场景”。你看完之后，不只是知道 `select`、`insert`、`update`、`delete` 怎么写，还会知道在真实后端开发里如何设计表、如何写多表查询、如何做分页、如何做事务、如何避免慢 SQL。

**写作思路：** 这份教程不走“只背语法”的路线，而是围绕一个最常见的电商后端场景来写。你会看到：建库建表、插入数据、查询订单、统计销量、分页列表、事务扣库存、索引优化，这些才是你工作里最常遇到的 SQL 使用方式。

**技术栈：** MySQL 8、SQL、事务、索引、聚合查询、多表关联、子查询、分页、`EXPLAIN`

---

## 一、为什么后端开发必须系统学 SQL

很多人学 Java、Spring Boot、Redis、MQ 的时候很积极，但一到 SQL 就只会：

1. `select * from 表名`
2. `where id = ?`
3. 复制粘贴一些复杂语句却不理解

这在真实后端开发里是不够的。

因为后端开发每天都在和 SQL 打交道，例如：

- 用户列表分页查询
- 订单详情多表关联查询
- 热门商品销量统计
- 修改库存和创建订单事务处理
- 通过索引优化慢查询

所以你学 SQL，目标不是死记语法，而是要真正理解：

- 数据怎么存
- 查询怎么写
- 更新怎么保证安全
- 多表怎么关联
- 慢 SQL 为什么会慢

## 二、这份教程使用什么业务场景来讲

我统一用一个很常见的“商城后端”场景来讲。

这套场景里有 4 张核心表：

1. `t_user` 用户表
2. `t_product` 商品表

你可以把它理解成：

- 一个用户可以下很多订单

这样设计的好处是：

- 能覆盖最常见的后端业务模型

## 三、先认识 SQL 的 4 大类操作

SQL 在后端开发里大致可以分成 4 大类：

1. `DDL`
   用来定义数据库结构，例如建库、建表、改表。

2. `DML`
   用来操作数据，例如新增、修改、删除。

3. `DQL`
   用来查询数据，例如 `select`。

4. `TCL`
   用来控制事务，例如 `begin`、`commit`、`rollback`。

后面这份文档会按真实使用频率来讲，重点放在：

- 建表
- 增删改查
- 多表查询
- 事务
- 索引

## 五、创建用户表 `t_user`

```sql
-- 用户表：保存用户的基础信息。
CREATE TABLE t_user (
    -- 主键 ID，使用自增方式生成。
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    -- 用户名，不能为空，并且要求唯一。
    username VARCHAR(50) NOT NULL UNIQUE,

    -- 手机号，不能为空，并且要求唯一。
    phone VARCHAR(20) NOT NULL UNIQUE,

    -- 用户状态：1 表示正常，0 表示禁用。
    status TINYINT NOT NULL DEFAULT 1,

    -- 创建时间，默认取当前时间。
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

    -- 更新时间，默认当前时间，后续每次更新自动刷新。
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

这段代码的作用：

- 创建用户表
- 这里你可以看到真实开发里最常见的字段：主键、唯一约束、状态字段、创建时间、更新时间

## 六、创建商品表 `t_product`

```sql
-- 商品表：保存商品的基础数据。
CREATE TABLE t_product (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    -- 商品名称不能为空。
    product_name VARCHAR(100) NOT NULL,

    -- 商品价格，保留两位小数。
    price DECIMAL(10, 2) NOT NULL,

    -- 商品库存。
    stock INT NOT NULL DEFAULT 0,

    -- 商品状态：1 上架，0 下架。
    status TINYINT NOT NULL DEFAULT 1,

    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

这段代码的作用：

- 创建商品表
- 这个表后面会被用于库存扣减、商品列表分页、销量统计等场景

---

## 七、创建订单表 `t_order`

```sql
-- 订单主表：保存订单的整体信息。
CREATE TABLE t_order (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    -- 订单号通常不直接暴露数据库自增主键，而是使用业务单号。
    order_no VARCHAR(64) NOT NULL UNIQUE,

    -- 下单用户 ID。
    user_id BIGINT NOT NULL,

    -- 订单总金额。
    total_amount DECIMAL(10, 2) NOT NULL,

    -- 订单状态：NEW-新建，PAID-已支付，CANCELLED-已取消。
    order_status VARCHAR(20) NOT NULL,

    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- 给 user_id 建索引，后面按用户查订单时会更快。
    INDEX idx_user_id (user_id)
);
```

这段代码的作用：

- 创建订单主表
- 这里用 `order_no` 保存业务单号，用 `user_id` 建索引，是后端项目中很典型的写法

## 八、创建订单明细表 `t_order_item`

```sql
-- 订单明细表：一张订单里可能有多个商品明细。
CREATE TABLE t_order_item (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    -- 所属订单 ID。
    order_id BIGINT NOT NULL,

    -- 商品 ID。
    product_id BIGINT NOT NULL,

    -- 购买数量。
    quantity INT NOT NULL,

    -- 成交单价。
    item_price DECIMAL(10, 2) NOT NULL,

    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,

    -- 给 order_id 建索引，后面查询订单明细时会非常常用。
    INDEX idx_order_id (order_id),

    -- 给 product_id 建索引，后面做销量统计和商品维度分析会用到。
    INDEX idx_product_id (product_id)
);
```

这段代码的作用：

- 创建订单明细表
- 订单明细是电商后端的经典设计，它能很好支撑一对多查询和销量统计

## 十、最常用的查询语句 `SELECT`

### 10.1 查询整张表

```sql
-- 查询所有用户。
SELECT id, username, phone, status, create_time
FROM t_user;
```

这段代码的作用：

- 演示最基本的单表查询
- 这里特意没有用 `select *`，因为真实开发里更推荐按需写字段

### 10.2 条件查询 `WHERE`

```sql
-- 查询状态为正常的用户。
SELECT id, username, phone
FROM t_user
WHERE status = 1;
```

这段代码的作用：

- 演示最基础的条件过滤
- `WHERE` 是后端开发里使用频率最高的 SQL 子句之一

### 10.3 多条件查询

```sql
-- 查询价格大于等于 100，并且库存大于 0 的商品。
SELECT id, product_name, price, stock
FROM t_product
WHERE price >= 100
  AND stock > 0;
```

这段代码的作用：

- 演示多条件查询
- 后端开发里非常常见，例如商品筛选、订单筛选、用户筛选

## 十一、排序和分页

### 11.1 排序 `ORDER BY`

```sql
-- 按商品价格从高到低排序。
SELECT id, product_name, price
FROM t_product
ORDER BY price DESC;
```

这段代码的作用：

- 演示排序查询
- 这是商品列表、订单列表、排行榜场景里的基本能力

### 11.2 分页 `LIMIT`

```sql
-- 查询第一页，每页 2 条。
-- offset = 0，size = 2。
SELECT id, product_name, price, stock
FROM t_product
ORDER BY id ASC
LIMIT 0, 2;
```

这段代码的作用：

- 演示最基础的 MySQL 分页写法
- 后端接口里最常见的分页参数通常就是 `pageNum` 和 `pageSize`

### 11.3 后端常用分页换算

如果前端传：

- `pageNum = 3`
- `pageSize = 10`

那么 SQL 里的偏移量是：

- `offset = (pageNum - 1) * pageSize = 20`

对应 SQL：

```sql
-- 查询第 3 页，每页 10 条。
SELECT id, order_no, total_amount, order_status
FROM t_order
ORDER BY create_time DESC
LIMIT 20, 10;
```

这段代码的作用：

- 演示真实后端分页参数是如何映射到 SQL 的
- 这是分页接口里必须熟悉的换算逻辑

## 十二、聚合函数：统计类查询必须会

### 12.1 统计总数 `COUNT`

```sql
-- 统计订单总数。
SELECT COUNT(*) AS total_order_count
FROM t_order;
```

这段代码的作用：

- 演示统计总条数
- 分页接口通常都会先查一遍总数，再查分页数据

### 12.2 求和 `SUM`

```sql
-- 统计所有已支付订单的总金额。
SELECT SUM(total_amount) AS total_paid_amount
FROM t_order
WHERE order_status = 'PAID';
```

这段代码的作用：

- 演示金额求和
- 这类 SQL 在财务统计、销售统计场景里很常见

### 12.3 平均值 `AVG`

```sql
-- 统计商品平均价格。
SELECT AVG(price) AS avg_price
FROM t_product;
```

这段代码的作用：

- 演示平均值统计
- 在运营分析类需求中非常常见

## 十三、分组统计 `GROUP BY`

### 13.1 按订单状态分组统计

```sql
-- 按订单状态分组，统计每种状态有多少单。
SELECT order_status, COUNT(*) AS order_count
FROM t_order
GROUP BY order_status;
```

这段代码的作用：

- 演示分组统计
- 这类语句在后台看板、数据报表、运营面板里非常常见

### 13.2 按商品统计销量

```sql
-- 按商品维度统计销量。
SELECT product_id, SUM(quantity) AS total_quantity
FROM t_order_item
GROUP BY product_id;
```

这段代码的作用：

- 演示订单明细表按商品聚合统计销量
- 这是商品销量榜、热卖商品排序的基础 SQL

### 13.3 使用 `HAVING`

```sql
-- 查询销量总数大于等于 1 的商品。
SELECT product_id, SUM(quantity) AS total_quantity
FROM t_order_item
GROUP BY product_id
HAVING SUM(quantity) >= 1;
```

这段代码的作用：

- 演示 `HAVING` 是对分组后的结果再做过滤
- 它和 `WHERE` 不一样，`WHERE` 是分组前过滤，`HAVING` 是分组后过滤

## 十四、多表关联查询：后端开发最重要的 SQL 能力之一

### 14.1 查询订单和用户信息

```sql
-- 查询订单号、下单用户、订单金额、订单状态。
SELECT
    o.order_no,
    u.username,
    o.total_amount,
    o.order_status
FROM t_order o
INNER JOIN t_user u ON o.user_id = u.id;
```

这段代码的作用：

- 演示最常见的内连接查询
- 后端项目里，订单和用户、订单和商品、用户和角色，几乎都离不开多表关联

### 14.2 查询订单详情页数据

```sql
-- 查询订单明细页：订单号、商品名、购买数量、成交单价。
SELECT
    o.order_no,
    p.product_name,
    oi.quantity,
    oi.item_price
FROM t_order o
INNER JOIN t_order_item oi ON o.id = oi.order_id
INNER JOIN t_product p ON oi.product_id = p.id
WHERE o.order_no = 'ORD202604020001';
```

这段代码的作用：

- 演示订单详情页常见的三表关联查询
- 这是电商、外卖、商城类项目里特别高频的 SQL

### 14.3 左连接 `LEFT JOIN`

```sql
-- 查询所有用户及其订单，如果某个用户还没有订单，也要查出来。
SELECT
    u.id,
    u.username,
    o.order_no,
    o.order_status
FROM t_user u
LEFT JOIN t_order o ON u.id = o.user_id;
```

这段代码的作用：

- 演示左连接的使用场景
- 左连接常用于“主表必须全部保留，从表有就查，没有就为空”的场景

## 十五、子查询：复杂查询里很常见

### 15.1 查询下过订单的用户

```sql
-- 查询下过订单的用户信息。
SELECT id, username, phone
FROM t_user
WHERE id IN (
    -- 子查询：先查出有订单的用户 ID。
    SELECT user_id
    FROM t_order
);
```

这段代码的作用：

- 演示 `IN + 子查询`
- 这类写法常用于“先查一批 ID，再到主表查详情”

### 15.2 使用 `EXISTS`

```sql
-- 查询至少有一笔订单的用户。
SELECT u.id, u.username
FROM t_user u
WHERE EXISTS (
    -- EXISTS 关注的是“子查询是否有结果”，而不是具体结果值是什么。
    SELECT 1
    FROM t_order o
    WHERE o.user_id = u.id
);
```

这段代码的作用：

- 演示 `EXISTS` 的使用
- 在某些关联判断场景里，`EXISTS` 比 `IN` 更适合

## 十九、真实后端场景：创建订单为什么要用事务

真实项目里，“创建订单”通常不是一条 SQL 能完成的。  
它至少会涉及两步：

1. 扣减商品库存
2. 插入订单主表和订单明细表

如果中间某一步失败，就必须整体回滚，否则数据就会脏。

### 19.1 手动事务示例

```sql
-- 开启事务。
START TRANSACTION;

-- 第一步：扣减商品库存。
UPDATE t_product
SET stock = stock - 1
WHERE id = 1
  AND stock > 0;

-- 第二步：插入订单主表。
INSERT INTO t_order (order_no, user_id, total_amount, order_status)
VALUES ('ORD202604020010', 1, 279.00, 'NEW');

-- 第三步：插入订单明细表。
INSERT INTO t_order_item (order_id, product_id, quantity, item_price)
VALUES (LAST_INSERT_ID(), 1, 1, 279.00);

-- 如果前面都成功，就提交事务。
COMMIT;
```

这段代码的作用：

- 演示事务的最基本使用
- 保证“扣库存 + 创建订单”这组操作要么都成功，要么都失败

### 19.2 回滚事务

```sql
-- 开启事务。
START TRANSACTION;

-- 模拟一步更新。
UPDATE t_product
SET stock = stock - 1
WHERE id = 2
  AND stock > 0;

-- 假设后续发现业务校验失败，此时需要回滚。
ROLLBACK;
```

这段代码的作用：

- 演示事务回滚
- 在真实项目里，一旦中间步骤失败，就必须回滚，避免脏数据

## 二十、后端开发里必须认识的索引

SQL 会不会慢，很多时候和索引直接相关。

### 20.1 创建索引

```sql
-- 给商品名称创建普通索引。
CREATE INDEX idx_product_name
ON t_product (product_name);
```

这段代码的作用：

- 演示创建普通索引
- 适合用于高频查询字段

### 20.2 联合索引

```sql
-- 给订单表创建联合索引。
CREATE INDEX idx_user_status_create_time
ON t_order (user_id, order_status, create_time);
```

这段代码的作用：

- 演示联合索引
- 这类索引很适合用于“按用户查订单，再按状态筛选，再按时间排序”的高频场景

### 20.3 查看执行计划 `EXPLAIN`

```sql
-- 查看订单查询 SQL 的执行计划。
EXPLAIN
SELECT id, order_no, total_amount, order_status
FROM t_order
WHERE user_id = 1
  AND order_status = 'PAID'
ORDER BY create_time DESC;
```

这段代码的作用：

- 演示如何查看 SQL 执行计划
- `EXPLAIN` 是排查慢 SQL 时最基础、最常用的工具

## 二十一、写 SQL 时最常见的 8 个坑

### 21.1 不要随手 `select *`

原因：

- 多查无用字段
- 网络传输更大
- 影响可读性
- 表结构一变，风险更高

### 21.2 `update` 和 `delete` 一定要先确认 `where`

这是数据库事故里最常见的坑之一。

### 21.3 不要在索引列上乱做函数运算

例如：

```sql
-- 不推荐：在索引列上做函数处理，可能导致索引失效。
SELECT *
FROM t_order
WHERE DATE(create_time) = '2026-04-02';
```

这段代码的作用：

- 演示一种会导致索引容易失效的写法
- 在高并发系统里，这类写法很容易把查询拖慢

### 21.4 分页太深会变慢

例如：

```sql
-- 深分页示例：当 offset 很大时，性能通常会越来越差。
SELECT id, order_no
FROM t_order
ORDER BY id
LIMIT 100000, 20;
```

这段代码的作用：

- 演示深分页问题
- 数据量大时，深分页通常需要结合“游标分页”或“基于主键翻页”优化

### 21.5 不要把业务判断全部堆到 SQL 里

SQL 很强，但不是所有业务逻辑都适合写进 SQL。  
复杂业务规则应该分层处理，别把一条 SQL 写成谁都看不懂的怪物。

### 21.6 对金额一定用 `DECIMAL`

不要用：

- `FLOAT`
- `DOUBLE`

来存金额，因为会有精度问题。

### 21.7 业务单号一般不要直接用自增主键代替

原因：

- 自增主键更适合内部关联
- 业务单号更适合对外展示

### 21.8 统计类查询要特别注意索引和数据量

因为聚合、分组、排序在数据量大时最容易慢。

## 二十二、后端项目里最常见的 SQL 使用链路

如果你以后在真实 Spring Boot 项目里写一个接口，最常见的 SQL 使用链路通常是：

1. `Controller`
   接收前端请求参数，例如页码、订单号、用户 ID。

2. `Service`
   做业务校验和参数整理，例如分页换算、状态判断。

3. `Mapper / Repository`
   编写 SQL 去查数据库。

4. `MySQL`
   返回结果给后端。

5. `Service`
   再把查询结果组装成 DTO 返回给前端。

所以你学 SQL，不是为了单独写 SQL，而是为了把它放进完整的后端链路里。

## 二十三、你可以把这份 SQL 教程怎么学

我建议你按这个顺序练：

1. 先自己把 4 张表建出来
2. 把测试数据插进去
3. 把文档里的每条 SQL 都亲手执行一遍
4. 尝试自己改条件，比如换排序、换分页、换状态
5. 自己再加 2 张表，例如优惠券表、支付表
6. 再去写更复杂的多表查询和统计查询

这样你会学得很扎实，而不是只“看懂”

## 二十四、最后总结

你可以把 SQL 理解成一句话：

**SQL 不是后端开发的附属技能，而是后端开发的核心基础能力。**

真正的工作里，SQL 写得好，会带来这些直接收益：

- 查询更准
- 接口更快
- 数据更安全
- 排错更容易

所以你这份教程吃透之后，下一步最值得补的就是：

1. SQL 索引优化专题
2. SQL 事务与锁专题
3. MyBatis / JPA 如何承接这些 SQL
