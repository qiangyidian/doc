# 二手图书网站基础架构设计

**日期：** 2026-04-10

## 一、这一步的目标

这一份文档的目标不是直接把全部业务一次性做完，而是先把“校园二手图书系统”的基础骨架定下来。

你后面每提出一个新需求，都要建立在这份基础架构之上继续往前推进，这样项目不会越做越乱。

这一步要解决的是下面 5 件事：

- 系统第一阶段到底采用什么技术架构
- 前端和后端如何拆分
- 后端模块应该怎么分层
- 前端目录应该怎么组织
- 后续文档应该按什么顺序继续往下写

---

## 二、项目背景和目标定位

这个系统不是一个纯练手的玩具项目，而是一个准备真实落地、真实运行的校园二手图书网站。

它至少要支撑下面这些真实业务场景：

- 学生发布自己想出售的二手图书
- 学生浏览、搜索、筛选图书
- 学生下单购买图书
- 卖家查看自己发布的书和订单
- 管理员审核、下架、处理违规信息

因为你还希望在开发过程中学习技术，所以这个项目必须满足两个要求：

1. 结构要足够真实，后面能继续接 MySQL、Redis、MQ、文件上传、后台管理
2. 复杂度要控制住，不能一开始就把项目拆成过重的微服务

所以第一阶段最合适的路线不是微服务，而是：

- `前后端分离`
- `后端单体应用`
- `前端单独工程`

## 三、为什么第一阶段选择这种架构

对于你的项目，当前有 3 条常见路线：

### 方案 A：前后端分离 + 单体后端 + 单独 Vue3 前端

后端使用一个 `Spring Boot` 工程，前端使用一个 `Vue3` 工程，双方通过 REST 接口通信。

优点：

- 最适合边开发边学习
- 部署和调试难度低
- 后面扩展 Redis、MQ、搜索、对象存储都比较顺
- 项目真实感足够强

缺点：

- 后期业务量特别大时，可能需要再做服务拆分

### 方案 B：前后端分离 + 单体后端 + 双前端工程

前端拆成用户端和管理端两个独立 Vue 工程。

优点：

- 职责更清晰
- 更接近中大型后台系统的组织方式

缺点：

- 对你当前阶段来说维护成本偏高
- 文档数量和调试量会明显增加

### 方案 C：微服务架构

一开始就拆用户服务、图书服务、订单服务、消息服务。

优点：

- 架构上更“企业化”

缺点：

- 当前阶段完全没有必要
- 学习成本会转移到注册中心、网关、配置中心、链路治理，而不是业务本身

### 本项目的结论

本项目第一阶段明确采用：

- `方案 A：前后端分离 + 单体后端 + 单独 Vue3 前端`

这是当前最稳妥、最适合持续迭代的路线。

## 五、系统总体架构图

```text
用户浏览器
   |
   | HTTP / HTTPS
   v
Vue3 前端网站
   |
   | REST API
   v
Spring Boot 后端
   |
   +------> MySQL
   |
   +------> Redis（后续阶段接入）
   |
   +------> RabbitMQ（后续阶段接入）
   |
   +------> MinIO/对象存储（后续阶段接入）
```

这套结构的意义：

- 浏览器只和 Vue3 前端交互
- Vue3 前端只负责页面、状态管理、接口调用
- Spring Boot 后端统一承载业务规则、权限、数据读写
- MySQL 先作为第一阶段核心数据源
- Redis、MQ、对象存储后面按需求逐步接入

## 六、系统角色划分

第一阶段建议先明确 3 类角色：

### 1. 普通用户

也就是校园学生用户，核心能力包括：

- 注册登录
- 浏览图书
- 发布图书
- 购买图书
- 查看自己的订单

### 2. 卖家

在系统里，卖家本质上仍然是普通用户，只是他发布了图书。

他需要额外具备：

- 管理自己发布的图书
- 修改图书状态
- 查看售卖记录

### 3. 管理员

管理员负责平台治理：

- 审核图书信息
- 下架违规图书
- 查看用户和订单情况

这里故意不把“卖家”单独做成一套完全独立账号体系，而是把它当成用户的一种业务身份，这样系统更简单，也更符合大多数二手交易平台的实现方式。

## 七、第一阶段核心业务范围

第一阶段不追求大而全，只先把最核心的业务主链路打通。

### 第一阶段要包含的内容

- 用户端基础页面框架
- 后端基础工程框架
- 用户模块基础模型
- 图书发布模块基础模型
- 图书列表和图书详情基础接口
- 订单基础模型
- 后台管理基础入口
- 统一返回对象、统一异常处理、基础参数校验

### 第一阶段先不做的内容

- 即时聊天
- 复杂推荐算法
- 优惠券系统
- 秒杀抢购
- 搜索引擎
- 消息中心
- 完整支付接入

这样做的目的很明确：

- 先把系统主骨架搭起来
- 先把“用户 -> 发布图书 -> 浏览图书 -> 下单”这条主链跑通
- 后续再逐步做性能优化和中间件增强

│   │   │       │   ├── service

│   │   │       │   ├── mapper
│   │   │       │   ├── entity

│   │       └── mapper
│   └── test
```

### 这一套分层的含义

#### `controller`

负责接收 HTTP 请求、参数校验入口、返回结果。

#### `service`

负责业务编排，也就是一个完整用例的执行流程。

#### `manager`

负责更细粒度的领域逻辑、跨表组合逻辑、通用业务拼装。

这个层不是强制必须，但对于真实项目很有用，因为它可以避免 `service` 越写越胖。

#### `mapper`

负责数据库访问。

#### `entity`

数据库实体对象。

#### `dto`

接口入参对象或服务内部传输对象。

#### `vo`

接口出参对象，给前端返回时更清晰。

│   ├── api

│   │   └── app.js

│   ├── utils
│   ├── styles
│   └── assets

```

### 前端这样拆的原因

- `api` 统一放接口调用
- `router` 管理路由
- `stores` 管理登录态、全局状态
- `layouts` 区分用户端页面框架和后台管理框架
- `views` 只关心页面本身
- `components` 放可复用组件

这样后面不管你加登录、发布图书、订单管理还是后台审核，都有明确位置。

## 十、数据库第一阶段核心表建议

第一阶段不需要一次把所有表都建全，但至少要围绕下面几张核心表设计。

### 1. 用户表 `user`

作用：

- 存储学生用户基本信息
- 后续可以扩展昵称、头像、手机号、学院、校区等字段

### 2. 图书表 `book`

作用：

- 存储图书主信息
- 比如书名、作者、价格、成色、简介、卖家、状态

### 3. 图书图片表 `book_image`

作用：

- 一个图书可能有多张图片
- 不建议把多张图直接拼在一列字符串里

### 4. 订单表 `book_order`

作用：

- 存储一次购买行为的主信息
- 包括买家、卖家、订单状态、订单金额

### 5. 订单操作记录表 `order_record`

作用：

- 记录订单状态变化
- 后续接入 MQ、售后、日志时会更方便

## 十二、接口风格建议

为了后面前后端联调更稳定，接口风格建议尽早统一。

### 接口前缀

建议统一使用：

- `/api`

例如：

- `/api/books`
- `/api/orders`
- `/api/users/me`
- `/api/admin/books`

### 返回结构

建议统一成下面这种格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

这样前端处理接口结果会更统一，后面做异常处理也方便。

### 状态值设计

不要一开始就在代码里到处写魔法值，建议状态统一收敛到枚举。

例如：

- 图书状态：`DRAFT`、`ON_SALE`、`LOCKED`、`SOLD`、`OFFLINE`
- 订单状态：`CREATED`、`WAIT_SELLER_CONFIRM`、`FINISHED`、`CANCELLED`
- 用户状态：`ENABLE`、`DISABLED`

## 十七、这一份文档的结论

校园二手图书系统第一阶段采用下面这套基础架构：

- `前后端分离`
- `Spring Boot 单体后端`
- `Vue3 单独前端工程`
- `MySQL 作为第一阶段核心数据源`
- `Redis、RabbitMQ、对象存储在后续阶段逐步接入`

后端以模块化单体为主，前端以单工程多页面模块为主。

这套方式既适合真实落地，也适合你一边开发一边学习。
